# Berth

Докинг-библиотека для [Avalonia](https://avaloniaui.net) по модели IntelliJ IDEA: шесть фиксированных слотов тул-панелей (3 стороны × 2 группы), пять режимов отображения (dock pinned / dock unpinned / undock / float / window), документная зона как дерево сплитов с таб-группами, сохранение и восстановление раскладки. Сознательно **не** free-form докинг: строгие правила → перечислимое пространство состояний → надёжность.

Целевые платформы: десктоп (Windows/macOS/Linux) и браузер (WASM; плавающие панели — оверлеями). Мобильные хосты собираются best effort, поведение докинга на таче не специфицируется.

**Статус: фазы 1 (ядро tool windows без UI, задачи 1.1–1.8) и 2 (статический UI + ходячий скелет, задачи 2.1–2.4) завершены.** В ядре: типы состояния и инварианты, операции с вытеснением слоёв, веса и ресайзы, дерево таб-групп док-зоны с нормализацией и инвариантами, операции док-зоны с активностью и фолбэками, персистентность (JSON-формат со схемой, `Apply` с отчётом об исправлениях, golden-тесты), жизненный цикл контента (политики, фабрики, заявочное владение вкладками, координатор `ContentLifecycle`) и деревья внутри панелей — `ContentTree`, canHost, вкладка-тело с мостом фабрик, перемещения панель ↔ док-зона теми же командами (`Berth.Core` + тесты). В UI-слое: `BerthWorkspace` материализует состояние — стрипы с «⋯», панели сторон со сплиттерами, декоратор панели, Undock-оверлей — обрабатывает ввод командами ядра (клик по иконке, меню, флайаут «⋯», drag сплиттеров с коммитом на отпускании) и связан с жизненным циклом контента: тело панели материализуется через `ContentLifecycle` (Control — напрямую, view model — через DataTemplates приложения), каждый жест докладывается `NotifyTransition` (`Berth.Avalonia` + headless-тесты). Демо-скелет живёт в `samples/Berth.Demo` (панели с разными политиками жизненного цикла, MVVM- и Control-контент) и собирается всегда, включая CI. Геометрия пары и Undock-оверлея выверена по трассировкам эталона после обкатки демо: доля пары выводится нормализацией предпочтений панелей, толщина оверлея — вес стороны. Фаза 3 (режимы и фокус) завершена: материализация `BerthWorkspace` стала инкрементальной (TW-9.13, задача 3.0) — постоянный каркас с коллапсом ячеек, кэш хостов панелей, diff-обновление на смену состояния; клавиатурный фокус и view-state контента не теряются как побочный эффект материализации, вид контента строится однократно и переживает перекладку и закрытие/переоткрытие. Поверх неё заработали автоскрытие и активация (TW-6.1/6.2/6.6, задача 3.1): DockUnpinned/Undock-панель закрывается, потеряв клавиатурный фокус (попапы, деактивация приложения и меню собственной иконки — исключения) или по клику вне её; активация командой переносит фокус в контент панели, получение фокуса контентом активирует панель, активная подсвечена. Фазу закрыли шорткаты активации (TW-5.5, задача 3.2): приложение привязывает свой кеймап к публичной команде `BerthWorkspace.ActivateToolWindow` с трёхпозиционной логикой («активна» = клавиатурный фокус внутри панели, как у эталона), тултип иконки стрипа показывает подсказку шортката от необязательного провайдера приложения, а перенос фокуса TW-6.6 расширен реатачем хоста активной панели — перенос в другой слот или слой восстанавливает фокус, не давая автоскрытию закрыть перемещаемую панель. Фаза 4 (документная зона UI) начата: задача 4.0 материализовала дерево док-зоны главного окна — таб-группы с полосами вкладок, сплиты со сплиттерами, контент активной вкладки каждой группы; хосты вкладок кэшируются по id и переприсоединяются только семантикой команды (DA-9.6 — зеркало TW-9.13: ключёванная реконсиляция, дети сплита размещаются Grid-индексами, поворот сплита не реатачит вовсе), контент тянется лениво планировщиком вне рендера (отказ фабрики закрывает вкладку штатно, спящие вкладки — заглушки с заголовком от публичного `TabTitleProvider`), ввод — клики и меню вкладок (Close, Split×4, Rotate, Move to панели-владельца) и сплиттеры → `SetSplitShares`; проводка активности замкнута: фокус в контенте вкладки активирует её и сбрасывает подсветку панелей (DA-6.4, TW-6.5), `Esc` из панели ведёт в текущую вкладку (TW-6.3), закрытие сфокусированной панели возвращает фокус документу (DA-E21). Фазу закрыла задача 4.1 — полоса вкладок и сплиты внутри панелей (UI для TW-9.5), после сверки с эталоном (`plugins/terminal`: движок вкладок терминала — общий `InternalDecoratorImpl`, меню-сплит эталона копирует контент, переносит только DnD): деревья панелей материализуются той же проекцией, что и док-зона (DA-9.6 расширен), кэш хостов вкладок един на раскладку — перенос панель ↔ док-зона переприсоединяет тот же хост с построенным видом; полоса корневой группы живёт в ряду заголовка декоратора (= эталон) и скрыта ровно для одиночного тела (DA-8.4), при корне-сплите полосы — по группам; тело панели идёт той же записью моста через ленивый планировщик, DisposeOnClose и `Unregister` роняют вид вместе с контентом; меню вкладки — Close, «Split and Move» ×4, Rotate, «Move to Document Area»/«Move to владельца» с активацией и фокусом вслед (закрытый приёмник открывается по DA-E39) и хвост полного меню («Remove from Sidebar», «Hide» — TW-5.16); принадлежность клика панелям для автоскрытия фиксируется на нажатии (TW-6.2). Окна документов — фаза 6. Фаза 5 (DnD внутри окна) начата: задача 5.0 принесла контроллер перетаскивания и жесты слотов (TW-5.17) — перетаскивание иконок стрипа и заголовков панелей поверх команд ядра (ADR-0004): порог отделяет клик от drag, жест до отпускания — чистая визуализация (ghost-плашка и маркер вставки в оверлей-слое, захват указателя на стабильном корне), дроп в зоны стрипов = ровно один `Move` с отображением позиции по TW-1.5, внешнее изменение состояния перестраивает цели, не прерывая жест, отмена (`Esc`, отпускание вне целей, потеря захвата) не оставляет следа; фокус-активация заголовка перенесена на отпускание, drag-релиз исключён из pointer-пути автоскрытия (TW-6.2) — решения выверены по свежей трассировке DnD-механизмов эталона. Фазу закрыла задача 5.1 — перетаскивание вкладок деревьев (DA-9.7): источники — заголовки вкладок всех материализованных деревьев, включая полосу в ряду заголовка декоратора; цели — полосы вкладок (`MoveTab` по позиции вставки), краевые клинья групп по диагоналям с долей 0.2 (свой — `SplitTab`, чужой — композиция `MoveTab` + `SplitTab`) и центры групп, приоритет полоса → клинья → центр (= эталон `processDropOver`/`dragToSplitRatio`); недопустимые по canHost деревья зон не дают (чужие панели, спящие вкладки, закрытые панели), тождественный дроп — без команды и активации, дроп исполняет последовательность команд по образцу пунктов меню 4.1 с активацией и фокусом вслед; press-фокус заголовка вкладки отложен на отпускание (расширение TW-6.6) — отменённый drag не оставляет следа; маркер клина — полупрозрачное превью половины группы. Фаза 6 (плавающие окна) начата: задача 6.0 материализовала плавающий слой на десктопе — открытые `Float`/`Window`-панели становятся окнами ОС (Float — owned поверх главного и вне таскбара, Window — независимое; TW-7.1/7.2), хостя те же кэшированные декораторы (белосписочный реатач слоя TW-9.13), а окна документов — независимыми окнами с деревом вкладок той же проекции над единым кэшем хостов (DA-7.1, DA-9.6; окна сопоставляются состоянию пересечением множеств вкладок — идентичности у них нет, DA-1.3); жесты окон сводятся к командам ядра: системное закрытие = `Close`/`CloseTab`×N (TW-7.3, DA-7.3), перемещение и ресайз = `SetFloatingBounds`/`SetDocumentWindowBounds` с гвардами равенства, закрытие главного окна — teardown без команд, раскладка цела для следующей сессии (TW-7.5, DA-7.6); проводка фокуса и автоскрытия охватывает все окна воркспейса (перенос фокуса между своими окнами — обычная потеря фокуса, цель в другом окне активирует его), «View Mode» предлагает все пять режимов с передачей экранных границ контента (TW-5.6), вкладочные меню получили «Move to New Window» и возврат из окна документов, а валидатор сохранённых границ по экранам — публичный хелпер `FloatingBoundsValidation` для `Apply` (TW-7.4). Задача 6.1 довела плавающий слой до браузера — оверлей-псевдоокна (ADR-0006): та же раскладка на платформе без окон ОС материализует открытые `Float`-панели и окна документов псевдоокнами в оверлей-слое workspace (TW-7.7, DA-7.5) — перемещение за заголовок, ресайз за края, свой z-порядок (клик/фокус поднимает), у документного псевдоокна title bar с «×» = `CloseTab`×N; хранимый режим `Window` деградирует только эффективно — в псевдо-Float (чистая функция ядра `GetEffectiveMode`, TW-7.6), возможности платформы публичны как read-only `CanFloat`/`CanUseWindowed` на `BerthWorkspace`, меню строятся по ним (в браузере скрыт только `Window`), «экран» браузера — рабочая область workspace с публичным валидатором границ `CreateOverlayValidator`. Обе поведенческие спеки приняты (tool-windows v0.20, document-area v0.13); дальше — межоконный drag.
