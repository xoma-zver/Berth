# Berth

Библиотека докинга для Avalonia по строгой модели IntelliJ IDEA (не free-form). Карта документации: `docs/README.md`; решения: `docs/adr/`; поведение: `docs/spec/`; задачи: `docs/BACKLOG.md`.

## Языки

Код, идентификаторы, XML-doc — английский. Документация (`docs/`), сообщения коммитов, общение — русский.

## Команды

```sh
dotnet build src/Berth.Core && dotnet build src/Berth.Avalonia
dotnet test tests/Berth.Core.Tests
dotnet test tests/Berth.Avalonia.Tests   # headless UI-тесты
```

Полное решение (`IdeaDocking.slnx`) включает мобильные хосты и требует workloads — для задач библиотеки собирать только `src/` и `tests/`.

## Архитектурные правила

- `src/Berth.Core` не ссылается на Avalonia (граница закреплена `ArchitectureTests`). Вся логика раскладки — в Core; UI-слой только материализует состояние и переводит ввод в команды ядра.
- Поведение определяется `docs/spec/*`. Изменение поведения = сначала правка/дополнение спеки, затем код и тест с id пункта (`TW-…`, `INV-…`) в имени.
- Каждый жест DnD завершается вызовом команды ядра, доступной также из меню/кода (ADR-0004).
- Пиксели не проникают в Core — только веса/доли (ADR-0002).
- Публичный API меняется осознанно: дифф `PublicAPI.Unshipped.txt` — часть ревью.

## Протокол сессии

1. Прочитать `docs/BACKLOG.md`; взять верхнюю незакрытую задачу текущей фазы, если владелец не указал иную.
2. Нетривиальную задачу начинать с плана и согласования.
3. Реализация вместе с тестами; ноль предупреждений (`TreatWarningsAsErrors` включён).
4. Обновить BACKLOG, при необходимости — спеку и OPEN-QUESTIONS.
5. Работать в ветке задачи; в `main` не коммитить — мердж делает владелец после ревью.
6. Сессию завершать чистым атомарным коммитом, не выжимая контекст до дна.

## Анти-слоп

- Никакого мёртвого кода, закомментированного кода, абстракций «на будущее».
- TODO в коде — только со ссылкой на задачу бэклога.
- Любая неуверенность — вопрос владельцу или запись в `docs/OPEN-QUESTIONS.md`, не молчаливое решение.
- Поведение не выдумывать: сверяться со спекой; пробел в спеке — сначала пункт в спеку (с флагом ⚠, если спорно).

## Definition of Done задачи

- [ ] Тесты на каждый затронутый пункт спеки; property-тесты при изменении инвариантов
- [ ] `dotnet build` без предупреждений; тесты зелёные (оба проекта)
- [ ] XML-doc на новый публичный API; `PublicAPI.Unshipped.txt` обновлён
- [ ] BACKLOG/спека/OPEN-QUESTIONS актуальны
- [ ] Один атомарный коммит в ветке задачи с внятным сообщением (что и почему)
