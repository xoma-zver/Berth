# Спецификация поведения: document area (документная зона)

**Статус: v0.18 — принята.**

История версий (v0.1 → v0.18: редакции по задачам фаз 1–7 и хозяйственным, каждая с
решениями владельца) — в git-истории этого файла; нормативный текст ниже самодостаточен.
Редакционное сжатие (уборка, 2026-07) поведения не меняло: развёрнутые обоснования
вердиктов и происхождение пунктов — в
[reference/idea-tracing-document-area.md](../reference/idea-tracing-document-area.md)
и git-истории.

Документ определяет *поведение* документной зоны (не API и не визуальный дизайн): дерево сплитов
с таб-группами (ADR-0005), отдельные окна документов и общую модель дерева, которую переиспользует
контент тул-панелей (спека tool-windows, раздел 9). Слоты, режимы, вытеснение и геометрия сторон
тул-панелей — спека tool-windows; настоящий документ покрывает внутренность деревьев и их хостов.

Соглашения — те же, что в tool-windows:

- Нормативные пункты: `DA-<раздел>.<номер>`; инварианты: `INV-D<номер>`; краевые случаи:
  `DA-E<номер>`. Идентификаторы не переиспользуются: удалённый пункт помечается «отменён»,
  новый получает следующий номер.
- Каждый пункт — тест (или группа тестов) с тем же идентификатором в имени.
- Метки: **[Core]** — проверяется в ядре без UI; **[UI]** — проверяется в слое Avalonia (headless).
- Эталонов два: **(= IDEA)** — соответствует IntelliJ IDEA; **(= VSC)** — соответствует VS Code;
  **(≈)** — наше упрощение/обобщение при том же наблюдаемом эффекте; **(≠)** — сознательное
  расхождение; **⚠** — решение под вопросом.

## 1. Термины и хосты деревьев

- **Вкладка (tab)** — единица контента со стабильным `Id`; анонимного контента не существует
  (ADR-0003). Контент вкладок создают фабрики; ядро оперирует только идентификаторами.
- **Владелец вкладки** — определяется реестром по `Id` (TW-9.7): либо **док-зона** (такая вкладка
  называется **документом**), либо конкретная **панель P** (панельная вкладка). Механизм
  объявления владельцев — заявки владения фабрик (TW-9.11); для этой спеки владелец — тотальная
  функция, известная ядру (единственность заявки проверяется при резолве — TW-9.11).
- **Таб-группа (группа)** — упорядоченный список вкладок плюс активная вкладка группы.
- **Сплит** — узел с ориентацией, упорядоченными детьми (≥ 2) и долями детей.
- **Дерево** — корневой узел (группа или сплит) со всеми потомками.
- **Хост дерева** — то, что владеет одним деревом и отображает его: док-зона главного окна,
  окно документов (раздел 7) или тул-панель (раздел 8). **Хосты док-зоны** = главное окно
  плюс окна документов.

**DA-1.1** [Core] Модель дерева едина для всех хостов: одна структура, одни операции, одна
нормализация (ADR-0005, TW-9.5). Хосты различаются только правилом допуска вкладок `canHost`
(INV-D5) и жизненным циклом: окно документов исчезает вместе с последней вкладкой (INV-D6),
панель живёт по правилам спеки tool-windows, док-зона главного окна существует всегда (TW-2.1).

**DA-1.2** [Core] Ориентация сплита: `Row` — дети раскладываются слева направо; `Column` — сверху
вниз. Направления сплит-операций: `Left`/`Right` порождают или пополняют `Row`, `Up`/`Down` —
`Column`.

**DA-1.3** [Core] Адресация в командах — без идентификаторов групп и окон:

- группа адресуется любой своей вкладкой («группа, содержащая X») либо ссылкой «корень хоста H»
  (панель — своим `Id`) — последняя нужна только для пустой корневой группы (DA-2.3);
- окно документов адресуется любой своей вкладкой (окна непусты, INV-D6);
- сплит адресуется хостом и путём — последовательностью индексов детей от корня.

Обоснование: группы и окна не имеют идентичности сверх содержимого; отсутствие id исключает класс
ошибок «висячая ссылка» по построению и убирает генерацию идентификаторов из чистого ядра. Плата:
UI пересчитывает пути после каждой операции. (= IDEA по узлам дерева: ни id, ни переопределённой
идентичности; ≠ VSC — там персистентные числовые id групп. Единственный счётчик-id эталона — на
самом окне документов, для повторного присоединения bounds; у нас `Bounds` — поле состояния окна,
id не нужен и для этого)

## 2. Модель дерева

**DA-2.1** [Core] Узлы дерева:

| Узел | Поля |
|---|---|
| Сплит | ориентация `Row`/`Column`; упорядоченные дети (≥ 2); доля каждого ребёнка ∈ (0..1), сумма 1 |
| Группа | упорядоченные `Id` вкладок; активная вкладка — одна из них (null только у пустой) |

После нормализации дети сплита — группы либо сплиты **перпендикулярной** ориентации (INV-D1).

**DA-2.2** [Core] Доли описывают размеры детей вдоль ориентации сплита; пиксели в ядро не
попадают (ADR-0002). Минимальные размеры контента — константы слоя UI; при нехватке места доли
клемпируются на этапе рендера, состояние не меняется (аналог TW-2.8, см. также INV-D3).
[клемп — UI]

**DA-2.3** [Core] Пустой может быть только корневая группа дерева: у док-зоны главного окна (нет
открытых документов) и у панели (все вкладки уехали или не восстановились, DA-8.4). Прочие пустые
группы устраняет нормализация (N1); окно документов с пустым деревом не существует (INV-D6).
(= IDEA: пустые сплиты схлопываются; ≠ VSC: там пустая группа — легальный житель сетки)

**DA-2.4** [Core] Каждая вкладка присутствует в раскладке не более одного раза — суммарно по всем
деревьям всех хостов (INV-D2). Инвариант — про уникальность **ключа** (`Id`), а не про запрет
второго представления документа: наш `Id` соответствует эталонному composite (каждый composite и
у IDEA встречается в дереве ровно один раз), не файлу. Следствия:

- сплит-операции **переносят** вкладку, а не копируют (DA-5.5);
- второе независимое представление документа легально и не требует поддержки ядра: приложение
  выдаёт вкладке собственный `Id`, чья фабрика создаёт второй вид над тем же контентом; общий
  буфер — забота контентного слоя (ADR-0003). Гранулярность ключа выбирает приложение: ключуя
  вкладки документами, оно получает «повторное открытие активирует» (DA-5.1); для второго вида —
  выдаёт новый `Id`;
- недублируемость конкретного контента (терминал) — тоже решение приложения: не выдавать
  клон-`Id`.

UI-жест «Split and Duplicate» в v1 не предоставляется (раздел 11). (= IDEA по модели дерева:
уникальность узла; различие — у IDEA ключ вкладки навязан (`VirtualFile`), у нас гранулярность
ключа свободна. Решение ADR-0005: перечислимость состояния и однозначность адресации DA-1.3)

**DA-2.5** [Core] Состояние уровня раскладки (дополняет TW-3.1):

| Поле | Смысл |
|---|---|
| дерево док-зоны главного окна | существует всегда; минимум — пустая корневая группа (TW-2.1) |
| текущая вкладка главного окна | память хоста (DA-6.1); null ⇔ дерево без вкладок |
| окна документов | список: `Bounds` + дерево + текущая вкладка окна (определена всегда, INV-D6); порядок списка — порядок создания (≈ IDEA: порядок — наша конвенция для детерминизма сериализации); z-порядок — [UI] |
| `ActiveDockHost` | активный хост док-зоны: главное окно либо окно документов; обновляется `ActivateTab` (DA-5.3), нормализуется при `Apply` (DA-9.2) |

Текущая вкладка — per-host (= IDEA); `ActiveDockHost` — явное поле вместо живого резолва из
фокуса у эталона, тот же приём, что `ActiveToolWindowId` в спеке tool-windows. Деревья панелей
живут в состоянии панелей (`ContentTree`, TW-3.1).

## 3. Нормализация

Нормализация приводит любое дерево к каноническому виду. Выполняется после каждой операции ядра и
при `Apply` (раздел 9); правила применяются до неподвижной точки (терминируемость: каждое
структурное правило уменьшает число узлов).

**DA-3.1** [Core] Правила:

- **N1** (пустые группы): группа без вкладок удаляется из родителя; корневая группа остаётся
  (DA-2.3). Спящие вкладки (DA-9.4) — вкладки: группа, содержащая только их, непуста.
- **N2** (вырожденные сплиты): сплит без детей удаляется из родителя (в корне — заменяется пустой
  группой); сплит с одним ребёнком заменяется этим ребёнком.
- **N3** (вливание): ребёнок-сплит той же ориентации, что родитель, вливается в родителя: его дети
  встают на его место в том же порядке, доля s влитого распределяется его детям умножением
  (gᵢ → s·gᵢ).
- **N4** (доли): невалидные доли (вне (0..1), NaN) заменяются равными по сплиту; затем вектор
  долей каждого сплита перенормируется к сумме 1.
- **N5** (активность): активная вкладка группы, не принадлежащая группе, заменяется первой
  вкладкой группы (null — только у пустой корневой); невалидная текущая вкладка хоста
  переназначается по правилу DA-6.3 в границах хоста — id, существующий в хосте, но не активный
  в своей группе, заменяется активной вкладкой своей группы, а для отсутствующего в хосте id
  опорной позиции нет, и DA-6.3 вырождается в «активная вкладка первой непустой группы в обходе
  в глубину» (у главного окна с пустым деревом — null); невалидный `ActiveDockHost` — главным
  окном.

Правила N1–N5 — внутридеревянные. Нормализация уровня док-зоны дополнительно удаляет окна
документов, чьё дерево не содержит вкладок (INV-D6, ср. DA-7.2); какой хост активен, от этого не
меняется — но если удалённым оказался сам активный хост, активным становится главное окно (в
представлении окон упорядоченным списком это ремап: уцелевший активный хост сохраняется под
новым индексом).

**DA-3.2** [Core] Каждая операция ядра возвращает нормализованное дерево. Нормализация
идемпотентна (property-тест: normalize ∘ normalize = normalize) и не меняет канонические деревья —
состояния, произведённые самим ядром, проходят её без единого исправления (round-trip-инвариант,
ср. TW-10.4). (≈ IDEA: там гарантия поддерживается инкрементально, выделенной нормализации нет)

## 4. Инварианты

Проверяются после каждой операции; нарушение — ошибка ядра (property-тесты).

**INV-D1** [Core] Деревья канонические: корень — группа или сплит; нет пустых групп вне корня;
нет сплитов с менее чем двумя детьми; ребёнок сплита не повторяет его ориентацию. (≠ IDEA:
у бинарного дерева эталона понятия канонической формы нет — одноориентационная вложенность там
легальна и перманентна; каноничность — цена и выигрыш n-арности, ADR-0005)

**INV-D2** [Core] Каждый `Id` вкладки встречается не более одного раза суммарно по всем деревьям
раскладки (док-зона главного окна, окна документов, панели).

**INV-D3** [Core] Доли детей каждого сплита ∈ (0..1) и в сумме дают 1 (числовой допуск — константа
ядра, фиксируется тестами). (≠ IDEA: там каждый бинарный узел клемпит долю к жёсткому [0.1, 0.9]
прямо в состоянии. Не заимствуем осознанно: для n-арного узла фиксированный пол несостоятелен —
k детей требуют k·пол ≤ 1, — и настоящей гарантии он не даёт даже у эталона: вложенность
перемножает доли. Минимальные размеры — забота рендера: DA-2.2, DA-E24)

**INV-D4** [Core] Активная вкладка каждой непустой группы принадлежит группе; null — только у
пустой корневой. Текущая вкладка каждого хоста док-зоны принадлежит этому хосту и является
активной вкладкой своей группы; у окна документов она определена всегда (INV-D6), у главного окна
null — ровно когда его дерево без вкладок. `ActiveDockHost` указывает на существующий хост.

**INV-D5** [Core] Допуск вкладок (`canHost`, TW-9.7, TW-9.11): деревья хостов док-зоны содержат
любые вкладки — документы, панельные вкладки любых панелей, спящие (DA-9.4); дерево панели P
содержит вкладки, чей резолв владельца даёт P либо никого — незаявленный id спит в дереве своего
предполагаемого владельца, вместе со спящей или живой P (TW-9.11). Невыразима вкладка с
подтверждённым **иным** владельцем; позднее подтверждение (заявка новой регистрации) чинится
переездом (DA-9.2). Перемещение вкладки в панель из другого хоста требует резолва владельца ровно
в эту панель: спящую вкладку перенести в панель нельзя — владелец не подтверждён (DA-8.2);
перемещения внутри одного хоста допустимы всегда — вкладка уже легально в нём.

**INV-D6** [Core] Окно документов существует ⇔ его дерево содержит хотя бы одну вкладку.
(= IDEA для окон без тул-панелей; полный гейт эталона слабее — окно, хостящее докированную
тул-панель, переживает пустое дерево вкладок, — но хостинг панелей в окнах документов вне рамок
v1, раздел 11)

## 5. Операции

Все операции — команды ядра; UI (клики, меню, жесты DnD) только вызывает их (ADR-0004). Команда
над `Id`, отсутствующим в раскладке, — ошибка вызывающей стороны (как в операциях тул-панелей).
Каждая команда завершается нормализацией (DA-3.2). Композитные закрытия («закрыть другие»,
«закрыть все») — композиции UI/приложения над `CloseTab`, ядро их не определяет.

**DA-5.1** [Core] `OpenDocument(id)`: открывает документ в текущей группе эффективного хоста
(DA-6.1) — вкладка вставляется **после активной** вкладки группы (в пустую — единственной) и
активируется (DA-5.3). Уже присутствующий где-либо в раскладке id — включая спящий в дереве
панели — только активируется (DA-5.3), переноса нет, в том числе открытый в другом хосте
(DA-E33): второй копии не бывает (INV-D2). `id` с подтверждённым владельцем-панелью — ошибка:
панельные вкладки открывает `OpenPanelTab` (TW-9.12), а не эта команда.
(= IDEA: вставка после активной — дефолт эталона; активация уже открытого = IDEA в пределах
окна, межоконное переоткрытие эталон решает клоном composite — у нас это клон-`Id` приложения,
DA-2.4; ≈ VSC: там настройка; позиция вставки — кандидат в настройку v2)

**DA-5.2** [Core] `CloseTab(id)`: вкладка удаляется из своей группы — в любом хосте, владелец не
участвует (панельная вкладка в док-зоне закрывается так же). Если вкладка была активной в группе,
активной становится **предыдущая** по порядку вкладка, а если закрыта первая — новая первая.
Опустевшая группа исчезает нормализацией; судьба текущей вкладки хоста-донора — DA-6.3
(у панели текущей вкладки хоста нет: в её дереве действует только правило активной вкладки
группы). Освобождение контента — контракт фабрик (ADR-0003, TW-9.2), не забота раскладки.
(= IDEA: дефолт «предыдущая соседка», MRU и «правая соседка» там опции; ≈ VSC: дефолт MRU —
история фокуса живёт в UI, см. DA-6.3)

**DA-5.3** [Core] `ActivateTab(id)`: вкладка становится активной в своей группе. Вкладка в хосте
док-зоны h дополнительно становится текущей вкладкой h и делает h активным хостом
(`ActiveDockHost = h`), а `ActiveToolWindowId` сбрасывается в null (TW-6.5). Вкладка в дереве
панели P активирует панель (`ActiveToolWindowId = P`, TW-6.5); состояние хостов док-зоны при этом
не меняется (DA-6.1). Активация вкладки в дереве **закрытой** панели дополнительно открывает
панель по правилу TW-5.1 (вытеснение слоя, R1) — активная панель всегда открыта (INV-5, DA-E39).
(≈ IDEA: там «выбор вкладки» и «фокус» — независимые оси; наша команда объединяет их — все пути,
которыми управляет пользователь, совпадают, а программный «выбор без фокуса» ядру v1 не нужен)

**DA-5.4** [Core] `MoveTab(id, target, index)`: перемещение вкладки в группу `target` (адресация
DA-1.3) на позицию `index` — позицию в списке вкладок приёмника после изъятия перемещаемой,
с клемпом в допустимый диапазон (как TW-5.7). Внутри группы это переупорядочивание; между
группами/хостами — перенос с валидацией `canHost` (INV-D5): проверка требуется при смене хоста
на панель — резолв владельца обязан подтвердить её (конфликт заявок — ошибка, TW-9.11);
перемещения внутри одного хоста и переносы в хосты док-зоны проверки не требуют. Недопустимое
перемещение — ошибка, состояние не меняется. Перемещённая вкладка становится активной вкладкой
группы-приёмника и текущей вкладкой хоста-приёмника (у приёмника-панели — только активной
вкладкой группы: текущей вкладки хоста у панели нет, DA-6.1) (= IDEA); в группе-доноре
активность переходит по правилу DA-5.2, текущая вкладка хоста-донора — по DA-6.3; фолбэк донора
применим только при переносе **между хостами** — внутри одного хоста перенесённая вкладка из
хоста не исчезает, и действует одно правило приёмника (в том числе при переупорядочивании).
`ActiveDockHost` и `ActiveToolWindowId` командой не меняются — активность раскладки назначает UI
следом (`ActivateTab`), если фокус последовал за вкладкой.

Перенос — **не** «закрытие + открытие»: контент вкладки при `MoveTab`/`SplitTab` (DA-5.5)/
`MoveTabToNewWindow` (DA-5.7) не освобождается и не пересоздаётся; политики удержания контента
(TW-9.2: `DisposeOnClose`) на перенос не реагируют. (= IDEA: живая сессия переезжает
панель ↔ док-зона без пересоздания)

**DA-5.5** [Core] `SplitTab(id, direction)` — «сплит переносом» (DA-2.4): рядом с группой вкладки
в направлении `direction` создаётся новая группа, и вкладка переносится в неё:

- направление вдоль ориентации родительского сплита — новая группа вставляется соседним ребёнком
  (перед группой при `Left`/`Up`, после — при `Right`/`Down`): n-арность вместо вложенности
  (ADR-0005);
- направление поперёк ориентации (или сплит группы-корня) — группа заменяется сплитом
  соответствующей ориентации из старой и новой групп; сторона та же, что и вдоль: новая группа —
  перед старой при `Left`/`Up`, после — при `Right`/`Down`;
- новая группа получает половину доли разделяемой (обе — по половине; = IDEA: proportion 0.5);
  если перенос опустошает разделяемую группу, новая группа наследует её долю **целиком** — доли
  соседей по сплиту не меняются (DA-E1);
- перенесённая вкладка активна в новой группе; прочее — как `MoveTab` (DA-5.4).

Сплит единственной вкладки группы схлопывается нормализацией: структура и доли — в точности
исходные (новая группа наследует место и долю донора), активность — по общим правилам переноса
(DA-E1; ≈ IDEA: там Move-действия дизейблены на группе с одной вкладкой, а интерактивный chooser
создаёт и тут же схлопывает — наша единая политика «разрешить, нормализация уладит» покрывает оба
несогласованных поведения эталона).

**DA-5.6** [Core] `SetSplitShares(host, path, shares)`: задаёт вектор долей детей сплита по адресу
(DA-1.3). Валидация: длина совпадает с числом детей, каждая доля ∈ (0..1), сумма 1 (допуск
INV-D3); иначе — ошибка. Хостом может быть и панель: её сплиты адресуются id панели и путём
(DA-1.3). UI сводит перетаскивание сплиттера к этой команде, меняя доли только
пары соседних детей; троттлинг допустим (ср. TW-5.9).

**DA-5.7** [Core] `MoveTabToNewWindow(id, bounds)`: создаёт окно документов с деревом из одной
группы с этой вкладкой и границами `bounds` (пиксели передаёт UI — ядро их не выдумывает,
ADR-0002); перенесённая вкладка — текущая вкладка нового окна. Допуск — как у док-зоны (INV-D5):
и документы, и панельные вкладки (TW-9.8). Прочее — как `MoveTab` (DA-5.4), включая исчезновение
опустевших групп и окон.

**DA-5.8** [Core] `SetDocumentWindowBounds(id, bounds)`: запоминает границы окна документов,
содержащего вкладку `id` (аналог `SetFloatingBounds`, TW-5.9). UI обязан вызывать при
перемещении/ресайзе окна; троттлинг допустим.

**DA-5.9** [Core] `RotateSplit(id)`: переворачивает ориентацию сплита-родителя группы, содержащей
вкладку `id` (`Row` ↔ `Column`), сохраняя порядок и доли детей; затем нормализация (DA-3.1) —
уровни, совпавшие по ориентации после поворота, сливаются по N3. Если группа вкладки — корень
дерева (родителя-сплита нет) — ошибка. Адресация вкладкой намеренно повторяет охват эталона:
поворачивается сплит, непосредственно держащий группу. (≈ IDEA `ChangeSplitOrientation`: для
узла из двух детей совпадает дословно, включая сохранение долей — DA-E25; наш узел n-арный, и
поворот действует на весь ряд/стопку разом; слияния N3 необратимы, поэтому последовательность
из нескольких поворотов может дать иную структуру, чем перманентная вложенность эталона —
DA-E26)

Следствие каноничности: поворот без слияний возможен только у сплита, у которого ни родитель, ни
дети — не сплиты (DA-E25). У внутреннего сплита повёрнутая ориентация совпадает и с
родителем-сплитом, и с детьми-сплитами сразу — N3 сливает каждую совпавшую границу, вплоть до
двойного слияния за один поворот (DA-E31, DA-E32); повторный поворот после слияний действует уже
на объединённый узел вместе с бывшими соседями. Отмена необратимого поворота — приложение поверх
`Snapshot` (раздел 11).

## 6. Активность и фокус

**DA-6.1** [Core] Текущая вкладка — свойство **хоста**: каждый хост док-зоны помнит свою, и эта
память не затрагивается ни работой в других хостах, ни фокусом в панелях. **Текущая группа**
хоста — группа его текущей вкладки (у главного окна при null — пустая корневая). **Активный
хост** (`ActiveDockHost`) — хост, чья вкладка активировалась последней (DA-5.3). **Эффективный
хост** — приёмник `OpenDocument`: при `ActiveToolWindowId ≠ null` — главное окно, иначе активный
хост. Обоснование: панель — не хост док-зоны, а её прописка в v1 — всегда слот главного окна,
в том числе у `Float`/`Window`-панелей с физически отдельным окном; открытие из панели адресуется
рабочей области, которой панель принадлежит. Так открытие файла из панели-«проекта» попадает в
группу, где работали в главном окне, — даже если последней активировалась вкладка отсоединённого
окна. (= IDEA: per-host `currentWindow` + живой резолв активного хоста из фокуса;
`ActiveDockHost` — тот же резолв, переведённый в явное поле)

**DA-6.2** [Core] Взаимодействие с активностью панелей — по TW-6.5, где «активация документа»
означает активацию любой вкладки хоста док-зоны, включая панельные вкладки, живущие там:
`ActivateTab` в док-зоне сбрасывает `ActiveToolWindowId`; активация панели не трогает ни текущие
вкладки хостов, ни `ActiveDockHost`. «Активный документ» для индикации [UI] — текущая вкладка
активного хоста при `ActiveToolWindowId = null`.

**DA-6.3** [Core] Фолбэки — локальные для хоста, детерминированные. Когда текущая вкладка хоста
исчезает (закрыта, перенесена):

- её группа осталась непустой — текущей становится новая активная вкладка группы (правило DA-5.2);
- группа исчезла — активная вкладка соседней группы **того же дерева**: предыдущей в обходе в
  глубину слева направо, иначе следующей; соседство определяется по дереву на момент перед
  исчезновением группы (нормализация того же вызова могла перекроить структуру — DA-E8),
  активная вкладка соседки — из итогового состояния;
- дерево главного окна опустело — текущая вкладка главного окна = null.

Опустевшее окно документов исчезает целиком (INV-D6); если оно было активным хостом, активным
становится главное окно — с его сохранённой текущей вкладкой (память хоста, DA-E18). Правило —
детерминированный фолбэк ядра; UI вправе сразу назначить свою цель командой `ActivateTab` по
собственной истории фокуса. (≈ IDEA: там выбирает живой фокус ОС и MRU-ранжирование хостов;
история — забота UI, не ядра — ср. DA-5.2)

**DA-6.4** [UI] Проводка активности: любое реальное получение фокуса контентом вкладки в любом
хосте UI обязан свести к `ActivateTab` — включая восстановление фокуса при активации окна как
целого (Alt-Tab, клик по заголовку окна: фокус возвращается последней сфокусированной вкладке
этого окна), а не только явный клик по заголовку вкладки (DA-E20). Возврат фокуса из панели в
док-зону (`Esc`, TW-6.3; скрытие панели шорткатом) ведёт в текущую вкладку активного хоста
(DA-E21). (= IDEA: фокус-наблюдатель каждого хоста реагирует на сырой `FOCUS_GAINED`, не на клики)

Обратная сторона проводки (зеркало TW-6.6): активация вкладки жестом UI — клик по заголовку,
выбор в меню — переносит клавиатурный фокус в контент вкладки: первый фокусируемый элемент
построенного вида, фолбэк — хост вкладки; фокус, уже находящийся внутри, не переставляется.
Прямое присвоение состояния приложением фокус не двигает. Перенос фокуса в хост,
живущий в другом окне воркспейса (окно документов, плавающая панель), предварительно
активирует это окно: настоящее окно ОС или поднятие псевдоокна в z-порядке (DA-7.5, TW-7.7) —
зеркало TW-6.6. На платформе, где окна документов не материализуются вовсе (мобильные
best-effort; ни настоящих окон, ни оверлеев), эффективный активный хост для целей фокуса и
индикации — главное окно: хранимый `ActiveDockHost`, указывающий на окно документов
восстановленной раскладки, не меняется — деградирует только представление (ср. TW-7.6).

## 7. Окна документов

**DA-7.1** [Core] Окно документов = собственные `Bounds` + дерево той же модели (ADR-0005) +
текущая вкладка (DA-2.5). Список окон — часть раскладки; z-порядок и фокус окон — забота UI.

**DA-7.2** [Core] Окно существует, пока его дерево содержит вкладки (INV-D6): создаётся командой
`MoveTabToNewWindow` (DA-5.7), исчезает, когда последняя вкладка закрыта или перенесена.

**DA-7.3** [UI, desktop] Окно документов — независимое окно верхнего уровня (аналог режима
`Window` панелей, TW-7.2). Закрытие окна системной кнопкой = `CloseTab` всех его вкладок
(≠ панелей TW-7.3: у документов нет «режима», к которому можно вернуться). (= IDEA, = VSC)
Заголовок — текущая вкладка окна, дополненная суффиксом приложения по правилу TW-7.2
(`WindowTitleSuffix`), если тот задан.

**DA-7.4** [Core] При `Apply`/восстановлении `Bounds` окон валидируются аналогично TW-7.4
(порог пересечения с экранами задаёт UI-слой); невалидные — позиционирование относительно
главного окна, состояние обновляется. (≈ IDEA: тот же наблюдаемый контракт через
общеплатформенный кэш границ окон эталона)

**DA-7.5** [UI, browser] В браузере окна документов — псевдоокна в оверлей-слое workspace
(ADR-0006): та же проекция дерева и тот же единый кэш хостов вкладок, что у
настоящих окон (DA-9.6). Псевдоокно несёт собственный title bar — заголовок (текущая
вкладка) и «×» = `CloseTab` всех вкладок (DA-7.3); перемещение за title bar и ресайз за
края/углы — чистая визуализация до отпускания, коммит одним `SetDocumentWindowBounds`
(DA-5.8), `Esc` откатывает без команды; z-порядок собственный — клик или клавиатурный
фокус внутри поднимает псевдоокно (не персистится, DA-7.1). «Экран» браузера — рабочая
область workspace: `Bounds` интерпретируются в её координатах, валидатор DA-7.4
поставляется по её границам, рендер клемпит псевдоокно в пределах workspace, не меняя
состояние (DA-2.2). Эффективное представление деградирует, хранимое состояние не меняется
(ср. TW-7.6, TW-7.7).

**DA-7.6** [UI] Закрытие главного окна: раскладка снапшотится, затем окна документов закрываются;
их вкладки остаются в сохранённом состоянии и откроются при следующем запуске (аналог TW-7.5;
= IDEA: save-before-teardown).

## 8. Вкладки панелей и док-зона

Нормативная база — спека tool-windows, раздел 9 (TW-9.5…TW-9.10); здесь — следствия в терминах
дерева.

**DA-8.1** [Core] Таблица допуска (`canHost`, TW-9.7, INV-D5):

| Приёмник | Документы | Вкладки панели P | Вкладки панели Q ≠ P |
|---|---|---|---|
| Хосты док-зоны (главное окно, окна документов) | да | да | да |
| Панель P | нет | да | нет |

(≈ IDEA для направления «панель → док-зона»: живой прецедент — мост Terminal, сессия переезжает
в редакторную вкладку и обратно без пересоздания. Общего реестра владельцев у эталона нет;
ячейки «нет» — наша конструкция, у эталона обеспечены раздельностью моделей контента)

**DA-8.2** [Core] Перемещения панель ↔ док-зона выполняются теми же командами
`MoveTab`/`SplitTab`/`MoveTabToNewWindow` (TW-9.8); из док-зоны панельная вкладка может уехать
только в панель-владельца (DA-8.1).

**DA-8.3** [Core] Панельные вкладки в док-зоне независимы от панели-владельца: её закрытие, смена
режима, перемещение и вытеснение деревьев док-зоны не меняют (TW-9.10; E26 tool-windows).
`Unregister(P)` удаляет вкладки P из деревьев хостов док-зоны с нормализацией (собственное
дерево P инертно и спит — TW-9.4); повторная регистрация их не восстанавливает (TW-9.4, TW-9.10).

**DA-8.4** [Core] Дерево панели с единственной группой из одной вкладки — вырожденное: полоса
табов не отображается [UI], только если эта единственная вкладка — тело панели (id = `Id`
панели, TW-9.5) — панель выглядит обычной. Одиночная вкладка с собственным id отображается:
иначе у неё не было бы ни одной UI-ручки — close, split, перенос (= IDEA: Run и Terminal
показывают единственную вкладку, Project — панель одного тела — вкладок не показывает).
Пустое дерево панели (все вкладки уехали или закрыты) легально (DA-2.3); отображение пустой
панели — заглушка слоя UI; открытость панели от состояния контента не зависит (TW-9.3).

## 9. Персистентность

**DA-9.1** [Core] Деревья всех хостов, список окон документов, текущие вкладки хостов и
`ActiveDockHost` — часть области `Full` (TW-10.6). Область `Arrangement` док-зону не затрагивает
вовсе: деревья, окна документов, текущие вкладки и активный хост остаются текущими (TW-10.6,
TW-10.7; E20/E26 tool-windows).

**DA-9.2** [Core] Нормализация при `Apply` (дополняет TW-10.4, тот же контракт отчёта):

- дубликат вкладки между группами/деревьями — остаётся первое вхождение в порядке обхода:
  док-зона главного окна → окна документов по порядку → панели в порядке следования в документе;
  внутри дерева — обход в глубину слева направо, внутри группы — по порядку вкладок (правило —
  целиком следствие INV-D2/ADR-0005: у эталона дубль — не дефект, механизма починки нет);
- структурные и числовые дефекты чинят правила N1–N5 (DA-3.1);
- вкладка с подтверждённым иным владельцем в дереве панели (INV-D5) переезжает в конец текущей
  группы главного окна (в пустое дерево — корневой группой): данные не теряются (ср. DA-9.4),
  активность не переназначается — кроме вынужденного назначения инвариантами, когда дерево
  главного окна было пустым (INV-D4); конфликтная заявка владельца не подтверждает — вкладка
  остаётся на месте (TW-9.11); в области `Arrangement` док-зона неприкосновенна (DA-9.1),
  переезд недоступен — такая вкладка удаляется из новой спящей записи с отчётом (TW-10.7);
  то же правило переезда действует при примирении на регистрации — без отчёта (TW-10.3);
- невалидные текущие вкладки хостов — по правилу DA-6.3; невалидный `ActiveDockHost` — главное
  окно.

Каждое исправление отражается в отчёте ровно один раз; вторичные переназначения активности,
вызванные починкой (смена текущей вкладки хоста после дедупликации, ремап `ActiveDockHost`
после удаления опустевшего окна), отдельных записей не порождают — их покрывает запись о
первичной починке.

Сериализация самого ядра проходит `Apply` без исправлений (round-trip-инвариант, property-тест).

**DA-9.3** [Core] Восстановление контента лениво (TW-9.3): дерево восстанавливается целиком,
фабрика/приложение вправе не восстановить вкладку — тогда она закрывается штатной `CloseTab` с
нормализацией (TW-9.9). Для документов контракт тот же: контент — забота приложения (ADR-0003).
Отказ единообразен для восстановленной и свежеоткрытой вкладки — ядро их не различает (память
«restored vs fresh» запрещена TW-9.3); отказ фабрики вкладке, только что открытой самим
приложением, — признак несогласованности приложения, обрабатываемый так же мягко.

**DA-9.4** [Core] Спящие вкладки. Если в дереве раскладки — хосте док-зоны или дереве панели
(INV-D5) — оказывается вкладка, чей владелец не имеет регистрации (панель-владелец «спит»,
TW-10.2), вкладка остаётся в дереве «спящей»: не материализуется (отображение-заглушка — [UI]),
но остаётся обычной вкладкой для структуры и операций — занимает место в порядке группы (N1),
может быть активной и текущей, закрывается `CloseTab`, перемещается в пределах хостов док-зоны
(в панель — нельзя: владелец не подтверждён, INV-D5) — и сериализуется обратно (round-trip, как
TW-10.2); в частности, окно документов, все вкладки которого спят, восстанавливается как есть
(DA-E27). Фолбэки активности (DA-5.2, DA-6.3) спящих и живых не различают — активной вкладкой
группы и текущей вкладкой хоста может стать спящая (DA-E28). Появление регистрации владельца
материализует её штатно лениво (DA-9.3: фабрика вправе отказаться — тогда `CloseTab`).
Граница с TW-9.4/TW-9.10 намеренная: явный `Unregister` в живой сессии закрывает вкладки
владельца безвозвратно; пассивное отсутствие регистрации при восстановлении — усыпляет.
Симметрично снятие регистрации контента док-зоны в живой сессии (`UnregisterDockContent`) —
та же граница: закрывает документы снятой фабрики в хостах док-зоны безвозвратно (`CloseTab`
в порядке обхода DA-9.2), освобождая их контент; опустевшее окно документов исчезает (INV-D6),
повторная регистрация фабрики документы не восстанавливает (DA-E42). Пассивное отсутствие
фабрики при восстановлении по-прежнему усыпляет её документы.
(≈ IDEA: вкладка с исчезнувшим провайдером остаётся видимой, но пустой; полного удаления нет
ни в одном механизме эталона)

**DA-9.5** [Core] Формат и версия — общие с tool-windows: JSON, `SchemaVersion`, golden-тесты,
явная ошибка загрузки при неподдерживаемой версии (TW-10.1, TW-10.5).
Граница между «ошибкой загрузки» и нормализацией — по представимости, общая с tool-windows
(TW-10.5), там же политика неизвестных полей и правило эволюции схемы. Для док-зоны: нечисловая
доля сплита — канонический пример леченого поля (→ NaN → N4 с отчётом, DA-E23); значение вне
домена модели — отсутствующая или пустая текущая вкладка окна документов, нечисловая или
отсутствующая составляющая его `Bounds`, отрицательный индекс `ActiveDockHost` — ошибка
загрузки: у окна документов нет представимого состояния «без текущей вкладки или без границ»
(INV-D4, INV-D6, ADR-0002).

**DA-9.6** [UI] Материализация хостов вкладок инкрементальна (зеркало TW-9.13) — для всех
материализуемых деревьев: хостов док-зоны и деревьев контента открытых панелей (TW-9.5;
окна документов материализуются настоящими окнами на десктопе и псевдоокнами в браузере —
DA-7.5; на платформе без обоих их вкладки удерживают кэшированные виды; идентичности у окон
нет — живые окна сопоставляются состоянию пересечением множеств вкладок, DA-1.3).
*Хост вкладки* — визуальный узел на id вкладки, владеющий построенным видом её контента;
создаётся однократно и обновляется на месте. Кэш хостов един на раскладку: id вкладки уникален
во всех деревьях (INV-D2) и служит глобальным ключом, поэтому перенос панель ↔ док-зона —
переприсоединение **того же** хоста, вид контента переезжает вместе с ним. Проход проекции
над этими кэшами нереентерабелен — команда, порождённая внутри прохода, проецируется
отложенным перепроходом (TW-9.14). Правила:

- смена состояния раскладки не пересоздаёт и не переприсоединяет хосты остающихся вкладок;
  клавиатурный фокус и view-state не теряются как побочный эффект материализации;
- переприсоединение хоста допустимо только по семантике команды: перенос вкладки в другую
  группу или хост (`MoveTab`, `SplitTab`, `MoveTabToNewWindow`), смена активной вкладки
  группы (отображается контент активной — хост прежней снимается, оставаясь в кэше) и
  структурная перестройка узла, адресованного командой (`SplitTab` поперёк оси,
  `RotateSplit`, слияния N3 того же вызова); соседние поддеревья, пережившие перестройку,
  не переприсоединяются;
- командный канал UI восстанавливает клавиатурный фокус в контент вкладки, потерявшей его
  из-за легального переприсоединения, если до команды фокус был внутри её хоста (зеркало
  TW-6.6); транзиентная потеря фокуса от реатача — не потеря фокуса для TW-6.1. Фокус,
  повисший после закрытия сфокусированной вкладки, возвращается преемнику: внутри
  оставшейся открытой панели — новой активной вкладке её группы (DA-5.2: скачок в док-зону
  деактивировал бы панель, а unpinned захлопнул бы автоскрытием TW-6.1); для вкладки
  док-зоны и вкладки, закрывшейся вместе со своей панелью, — текущей вкладке активного
  хоста (DA-6.3, DA-E21). Фокус, ушедший самим жестом на живой хром (нажатие по «×»
  переводит его на ближайший фокусируемый предок — декоратор), не повисал и не
  переставляется (TW-6.6);
- контент материализуется лениво и только для активных вкладок видимых групп (TW-9.3,
  DA-9.3): групп главного окна, материализованных окон документов и деревьев **открытых**
  панелей — вкладки закрытой панели не материализуются, момент создания тела следует
  политике TW-9.2; вкладка без контента — в том числе спящая (DA-9.4) — отображается
  заглушкой; отказ фабрики (DA-9.3) обрабатывается вне прохода проекции: закрытие
  отказанной вкладки — переход координатора, жизненному циклу повторно не докладывается;
- хост удерживает построенный вид, пока id вкладки присутствует в раскладке: уход
  активности, закрытие панели-хозяйки и перенос в нематериализованный хост вид не роняют;
  `CloseTab` и любой другой уход id из раскладки освобождают контент и вид — «вида без
  вкладки» не существует (TW-9.2). Исключение — освобождение контента при живом id: тело
  `DisposeOnClose`-панели на переходе из открытости (TW-9.2, в собственном дереве — щит
  DA-8.3 не действует) и контент вкладок при `Unregister` владельца (TW-9.4) освобождаются,
  не покидая раскладку, — хост роняет вид вместе с контентом (удержание вида не должно
  продлевать жизнь контенту, TW-9.13) и возвращается к заглушке до следующей материализации;
- вид строится однократно на объект контента — тот же ручной срез ContentPresenter, что у
  тела панели (TW-9.13, раздел 12 спеки tool-windows): живая подмена DataTemplates не
  перевыбирает шаблон уже построенных видов, содержимое темизируется штатно; ре-селект —
  кандидат v2;
- лист-хром — заголовки вкладок и их меню, полосы вкладок, сплиттеры — вправе
  пересоздаваться при любой смене состояния. Заголовок вкладки — строка от необязательного
  провайдера приложения (id → отображаемая строка; ср. подсказки TW-5.5), фолбэк — сам id;
  для вкладки-тела (id = `Id` панели, TW-9.5) фолбэк — `Title` дескриптора, затем id:
  раскладка хранит только идентификаторы (ADR-0003), а заголовок обязан существовать до
  материализации контента и у спящих вкладок. (≈ IDEA)

**DA-9.7** [UI] Перетаскивание вкладок — жест TW-5.17 для заголовков вкладок: общий
контракт жеста (порог старта, образ-ghost и маркер цели как чистая визуализация, захват
указателя на стабильном корне окна-источника, пространство жеста и приоритет верхнего
окна в точке, перестройка целей при внешнем изменении состояния, отмена — `Esc`, потеря
захвата — без следа: ни команды, ни активации, ни переноса фокуса, DA-E22) наследуется
оттуда дословно; жест отменяется, когда перетаскиваемая вкладка покидает раскладку.
Специфика вкладок:

- **источники** — заголовки вкладок полос всех материализованных деревьев (DA-9.6) во
  **всех окнах воркспейса**: групп док-зоны главного окна, деревьев открытых панелей —
  включая полосу корневой группы в ряду заголовка декоратора (TW-9.5) и декораторы в
  плавающих окнах и псевдоокнах, — и групп окон документов, настоящих и псевдо (DA-7.5).
  Одиночное тело панели источником не является — у него нет заголовка (DA-8.4: ручек у
  вырожденного тела нет намеренно);
- **цели и их приоритет** (= эталон: полоса главнее клиньев, клинья главнее центра):
  - *полоса вкладок* — вставка в позицию между серединами соседних заголовков →
    `MoveTab(id, группа, index)` (DA-5.4); маркер — живой reorder-превью полосы
    (визуальный язык ниже);
  - *краевые клинья группы* — область группы делится диагоналями на четыре сектора,
    зона клина ограничена долей 0.2 протяжённости группы от кромки (константа UI,
    = эталон): клин собственной группы → `SplitTab(id, direction)` (DA-5.5), клин чужой —
    композиция `MoveTab` в конец приёмника + `SplitTab` (атомарный вариант — раздел 11);
    маркер — превью результата, половина группы по направлению (= эталон);
  - *центр группы* → `MoveTab` в конец группы; маркер — вся группа. Пустая корневая
    группа (DA-2.3) даёт только центр — `MoveTab` в корень хоста (DA-1.3), клиньев у
    пустой группы нет;
- **визуальный язык вкладочного жеста**: ghost — плашка с заголовком вкладки у курсора на
  протяжении всего жеста, над любой целью и вне полосы — единый язык со стрип-целями
  TW-5.17. Над полосой вкладок к плашке добавляется **живой reorder-превью**: заголовки
  полосы расступаются, зазор подсвечен рамкой-заливкой — местом, которое займёт вкладка
  при отпускании (ширина — с заголовка-источника на старте; текста в рамке нет —
  заголовок уже на плашке у курсора), место утаскиваемого заголовка схлопывается — в
  том числе в полосе-доноре при кросс-полосном переносе (= эталон: плавающая вкладка у
  курсора, расступание, подсвеченное место вставки). Превью чисто визуально: сдвиги —
  трансформы поверх проекции, `Bounds` и геометрия зон хит-теста не меняются, реальные
  заголовки не переупорядочиваются (≠ эталон: тот двигает сам TabLabel, потому его
  отмена грязна — DA-E22; у нас отмена бесплатна по построению), реаплай оверрайдов
  после каждой внешней репроекции (контракт раздела 12 tool-windows). **Вне всех целей**
  плашку сменяет полупрозрачная миниатюра контента вкладки — предвестие выноса в окно
  документов (см. «отпускание вне всех целей»); миниатюра снимается однократно при
  старте жеста и только у вкладки, чей вид был построен и присоединён на старте —
  активной вкладки видимой группы (DA-9.6): press-фокус отложен, у неактивной вкладки
  вида нет — остаётся плашка (зеркало правила видимости панелей TW-5.17). Над клиньями
  и центрами — плашка и закраска зоны;
- **недопустимая зона целью не является** (canHost, INV-D5, DA-8.2): цели строятся из
  дерева, где вкладка живёт сейчас (перемещения внутри хоста легальны всегда), деревьев
  **всех хостов док-зоны** — главного окна и материализованных окон документов (хосты
  док-зоны принимают любые вкладки) — и дерева панели-владельца — при подтверждённом
  резолве владельца (TW-9.11). Деревья чужих панелей зон не дают; у спящей вкладки нет
  целей в панелях (владелец не подтверждён); закрытая панель не цель — спринг-лоадинг не
  поддерживается (раздел 12 tool-windows); иконка владельца не цель — перенос в закрытую
  панель покрыт меню (DA-E39);
- **отпускание вне всех целей** — не отмена (зеркало TW-7.8):
  `MoveTabToNewWindow(id, bounds у точки отпускания)` (DA-5.7) — вкладка уезжает в новое
  окно документов у точки сброса, настоящее на десктопе или псевдоокно в координатах
  workspace (DA-7.5); активация и фокус вслед, как у прочих кросс-хостовых дропов
  (= эталон: drag-out вкладки создаёт окно с bounds от точки сброса). Размер — текущий
  размер группы-донора; тождества нет: единственная вкладка окна документов, брошенная
  вне целей, переезжает в новое окно у точки — старое исчезает (INV-D6), наблюдаемо это
  перенос окна;
- **тождественный дроп** — без команды и без активации (продолжение правила TW-5.17):
  своя позиция в полосе, зазор сразу после себя, центр собственной группы (DA-E40);
- **активация и фокус вслед за дропом** — как у пунктов меню (DA-5.4, DA-6.4): команда
  делает перенесённую вкладку активной в приёмнике, UI переносит клавиатурный фокус
  следом; при смене хоста активность назначается `ActivateTab` (DA-5.3);
- **press-фокус заголовка вкладки откладывается на отпускание** — расширение TW-6.6 с
  полосы заголовка панели на заголовки вкладок: нажатие, могущее стать перетаскиванием,
  не паркует клавиатурный фокус и не активирует панель; клик-семантика — активация
  вкладки и перенос фокуса (DA-5.3, DA-6.4) — выполняется на отпускании, нажатие,
  ставшее перетаскиванием, её не выполняет (TW-5.17). Кнопка «×» заголовка не
  затронута: её нажатие паркует фокус штатно (DA-9.6).

(= эталон по механике зон и приоритетов — трассировка DnD в reference/idea-tracing.md;
≠ эталона по чистоте отмены вкладочного drag — DA-E22)

## 11. Вне рамок v1

- Жест «Split and Duplicate» (второе представление документа одним действием). Сама модель второй
  вид поддерживает — как вкладку с собственным `Id` от приложения (DA-2.4); рецепт: клон-`Id` →
  `OpenDocument` → `SplitTab`. Готовый UI-жест с хуком «выдай клон-`Id`» — кандидат v2.
- MRU-история фокуса и переключатель Ctrl+Tab — приложение/UI поверх `ActivateTab`; ядро держит
  только детерминированные фолбэки (DA-5.2, DA-6.3).
- Undo-стек закрытых вкладок (аналог `ReopenClosedTabAction` эталона) — приложение поверх
  `Snapshot`/`OpenDocument`/`MoveTab`; ядро истории закрытий не держит — та же линия, что с MRU.
  Сюда же — отмена необратимых слияний `RotateSplit` (DA-5.9): снимок перед командой.
- Хостинг тул-панелей в окнах документов (эталон умеет: окно с докированной панелью переживает
  пустое дерево вкладок) — вне рамок v1. Цена включения известна заранее: INV-D6 ослабнет до
  «вкладка или докированная панель»; окнам понадобится лёгкий персистентный id — адресация
  «любой вкладкой» не покрывает окно без вкладок (DA-1.3); правило эффективного хоста DA-6.1
  потребует обобщения (фокус в панели перестанет означать главное окно).
- Сплит с целевой группой другого хоста одним жестом (интерактивный chooser эталона) —
  выражается композицией `MoveTab` + `SplitTab`; атомарное расширение `SplitTab` целевой
  группой — кандидат v2.
- Pinned- и preview-вкладки, ограничение числа вкладок в группе.
- Maximize/«зум» группы; выравнивание долей (equalize); перемещение целых групп одним жестом
  (VSC «Move Editor Group») — достигается по-вкладочно.
- Позиция вставки новой вкладки и политика выбора после закрытия как настройки (у эталонов —
  настройки, DA-5.1/DA-5.2 фиксируют дефолт).
- Пустые группы как постоянные жители сетки (VSC) — не поддерживаются (DA-2.3).
- Переполнение полосы вкладок (суммарная ширина заголовков больше ширины группы): скролл,
  мультиряд и сжатие не поддерживаются — та же линия, что с переполнением стрипа (раздел 12
  tool-windows); graceful-обработка — кандидат v2.
- Реордер вкладок пунктами меню («Move Tab Left/Right») — эталонный аналог terminal-специфичен,
  платформенное меню вкладки панели реордера не содержит; ядро покрывает его `MoveTab` внутри
  группы, UI-жест — перетаскивание вкладки (DA-9.7).
