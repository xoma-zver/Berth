# Спецификация поведения: tool windows (тул-панели)

**Статус: v0.23 — принята: guard домена enum в программных входах ядра, решение владельца (2026-07).**

История:

- v0.22 → v0.23 — guard домена enum в программных входах ядра (хозяйственная задача по ревью
  2.1; прецедент — guard конечности bounds TW-5.9), решение владельца (2026-07): раздел 5
  дополнен контрактом — значение перечисления вне домена в программном входе (команде либо
  конструкторе состояния/дескриптора) — ошибка вызывающей стороны (`ArgumentOutOfRangeException`);
  иначе кастованное недоменное значение доживало бы до сериализации и роняло `Serialize` на
  состоянии, произведённом штатной операцией (`Move`, `SetMode`). Файловый вход уже защищён
  `Deserialize` (TW-10.5); значение, инъецированное в иммутабельную модель через `with` в обход
  этих входов, остаётся ошибкой вызывающей стороны. Вариант «валидирующие init-аксессоры»
  (correct-by-construction, закрывает и `with`) отклонён в пользу пропорционального guard'а на
  входах — по прецеденту finite-bounds; уточнение к формулировке пункта бэклога: NRE в UI-сайте
  (`SidePane`) снят каркасом 3.0, остался класс «несериализуемое состояние». Нормативные пункты
  не менялись, `PublicAPI` без изменений (guard — поведение, не поверхность).

Документ определяет *поведение* подсистемы тул-панелей (не API и не визуальный дизайн).
Документная зона — документы, дерево таб-групп, их отдельные окна — описывается отдельной спецификацией document-area (задача 1.4 бэклога); настоящий документ покрывает только тул-панели и ссылается на общую модель дерева там, где панель его хостит (раздел 9).

- Метки: **[Core]** — проверяется в ядре без UI; **[UI]** — проверяется в слое Avalonia (headless).

## 1. Термины и слоты

- **Тул-панель (tool window)** — вспомогательное представление с заголовком, привязанное к краю рабочей области (в отличие от документов в центральной зоне).
- **Сторона (Side)**: `Left`, `Right`, `Bottom` — где открывается контент панели.
- **Группа (Group)**: `Primary`, `Secondary` — половина стороны.
- **Слот** — пара (Side, Group). Слотов ровно шесть.
- **Стрип (stripe)** — вертикальная полоса иконок у левого/правого края главного окна.
- **Прописка (placement)** — слот + порядковый номер, закреплённые за панелью. Определена всегда, в любом режиме.

**TW-1.1** [Core] Существует ровно шесть слотов. Соответствие слотов, стрипов и зон контента:

| Слот | Иконка на стрипе | Где открывается контент | Термины IDEA |
|---|---|---|---|
| Left.Primary | левый стрип, верхний сегмент | левая панель, верхняя часть | anchor=left, split=false |
| Left.Secondary | левый стрип, под разделителем | левая панель, нижняя часть | anchor=left, split=true |
| Right.Primary | правый стрип, верхний сегмент | правая панель, верхняя часть | anchor=right, split=false |
| Right.Secondary | правый стрип, под разделителем | правая панель, нижняя часть | anchor=right, split=true |
| Bottom.Primary | левый стрип, нижний сегмент | нижняя панель, левая часть | anchor=bottom, split=false |
| Bottom.Secondary | правый стрип, нижний сегмент | нижняя панель, правая часть | anchor=bottom, split=true |

**TW-1.2** [UI] Порядок сегментов левого стрипа сверху вниз: иконки Left.Primary → разделитель → иконки Left.Secondary → кнопка быстрого доступа «⋯» (при `QuickAccessSide = Left`, раздел 8) → растяжимое пустое место → иконки Bottom.Primary. Правый стрип зеркально: Right.Primary → разделитель → Right.Secondary → «⋯» (при `QuickAccessSide = Right`) → растяжка → Bottom.Secondary. (= IDEA, new UI; уточнение владельца, 2026-07)

**TW-1.3** [UI] Разделитель между сегментами Primary и Secondary отображается, только если оба сегмента непусты.

**TW-1.4** [UI] Внутри сегмента иконки упорядочены по `Order` панелей (см. TW-3.1). Нижние сегменты (Bottom.Primary на левом стрипе, Bottom.Secondary на правом) растут снизу вверх: иконка с `Order = 0` — ближайшая к нижнему краю (= IDEA, new UI — нижний сегмент читается от угла). Порядок меняется пользователем: перетаскивание иконки (фаза DnD) и пункты меню сводятся к команде `Move` (TW-5.7).

**TW-1.5** [UI] Отображение видимой позиции вставки в индекс плотного порядка (при перетаскивании иконки или вставке через меню): элемент вставляется сразу после ближайшего **видимого** предшественника; если такого нет — в начало слота. Скрытые иконки (TW-5.10) остаются на своих местах плотного порядка. Команда ядра `Move` при этом однозначна — конвенция целиком живёт в UI-отображении. ## 2. Геометрия рабочей области

**TW-2.1** [UI] Компоновка рабочей области (все элементы опциональны, кроме док-зоны):

```
┌──┬─────────────────────────────────┬──┐
│с │ ┌───────┬───────────┬─────────┐ │с │
│т │ │ левая │ док-зона  │ правая  │ │т │
│р │ │ панель│(документы)│ панель  │ │р │
│и │ ├───────┴───────────┴─────────┤ │и │
│п │ │        нижняя панель        │ │п │
└──┴─┴─────────────────────────────┴─┴──┘
```

Нижняя панель занимает всю ширину между стрипами; боковые панели располагаются над ней. (= IDEA при выключенной опции «Widescreen tool window layout»)

**TW-2.2** Вариант «widescreen» (боковые панели полной высоты, нижняя — между ними) в v1 не реализуется; возможная опция v2. (решение владельца, 2026-07)

**TW-2.3** [UI] Боковая панель (левая/правая), когда открыты обе её группы, делится **вертикальным** стеком: контент Primary сверху, Secondary снизу, между ними сплиттер. Если открыта одна группа — она занимает всю панель. (= IDEA)

**TW-2.4** [UI] Нижняя панель, когда открыты обе группы, делится **горизонтально**: Primary слева, Secondary справа, между ними сплиттер. Если открыта одна — занимает всю панель. (= IDEA)

**TW-2.5** [Core] Размеры описываются долями, а не пикселями:

- `Weight` стороны — доля рабочей области (ширина для Left/Right, высота для Bottom); по одному числу на сторону, **общему для докированного слоя и Undock-оверлея** (TW-3.3). Дефолт 0.33 (дефолт IDEA `WindowInfoImpl.DEFAULT_WEIGHT`; решение владельца, 2026-07).
- `PairRatio` панели ∈ (0..1) — её собственное предпочтение доли стороны в паре, дефолт 0.5; единственный геометрический атрибут панели.

**Эффективная доля Primary-контента** при открытых обеих группах стороны не хранится, а **выводится** нормализацией предпочтений пары (правило R1, TW-2.7): рассинхронизация «позиция сплиттера ↔ веса пары» невыразима — тот же приём, что доля в паре с ребёнком сплита (DA-2.1). (= IDEA new UI по наблюдаемому поведению при дефолтных настройках; по устройству — наша замена двух синхронизируемых состояний эталона, `sideWeight` окон + транзиентный кэш пропорции, одним выводимым значением; трассировка владельца, 2026-07)

**TW-2.6** [Core] Открытие, закрытие и переключение панелей не меняют `Weight` стороны — панель отображается в текущей ширине (высоте) стороны, в том числе унаследованной от ранее открытой панели. `Weight` записывается только ресайзом внешней кромки стороны.

**TW-2.7** [Core] Соотношение пары групп — четыре правила:
- **R1** (эффективная доля пары): при открытых обеих группах стороны доля Primary-контента = `P.PairRatio / (P.PairRatio + S.PairRatio)` — нормализация двух предпочтений. Величина выводится из состояния (TW-2.5), поэтому не зависит ни от пути образования пары (`Open`, `Move`, будущий DnD), ни от порядка прибытия панелей — геометрия пары есть функция от пары. При согласованной паре (сумма предпочтений 1 — её поддерживает R2; дефолты 0.5 + 0.5 согласованы) вырождается в `P.PairRatio` — «Primary диктует». (= IDEA при согласованной паре и при прибытии Secondary — там та же формула `oldInfo.sideWeight / (oldInfo.sideWeight + info.sideWeight)`; ≠ IDEA в одном случае: прибывающий Primary с несогласованным весом у эталона применяет чистое предпочтение, то есть итог зависит от порядка прибытия — асимметрия двух независимых путей `addAndSplitDockedComponentCmd`, признана случайной, не заимствуем; принцип нормализации един с N4 док-зоны; трассировка владельца, 2026-07)
- **R2** (ресайз пары): сдвиг сплиттера в позицию p (доля Primary) → `PairRatio` Primary-панели = p, Secondary-панели = 1 − p — «учим обе», пара становится согласованной и R1 воспроизводит p точно. (= IDEA: при драге `componentResized` срабатывает у обоих декораторов и `sideWeight` пишется каждому — симметричная запись подтверждена трассировкой владельца, 2026-07, вопреки прежнему прочтению «пишется только двигаемая»)
- **R3** (закрытие из пары): оставшаяся панель занимает всю сторону; предпочтения `PairRatio` не меняются — переоткрытие пары восстанавливает прежнюю долю по R1;
- **R4** (одиночное открытие): предпочтения в геометрии не участвуют.

**TW-2.8** [UI] Минимальные размеры контента панелей и док-зоны задаются константами слоя UI; при нехватке места веса клемпируются на этапе рендера, состояние не изменяется.

## 3. Модель состояния

**TW-3.1** [Core] Состояние панели:

| Поле | Тип/диапазон | Смысл |
|---|---|---|
| `Id` | непустая строка | стабильный идентификатор регистрации |
| `Side`, `Group` | слот | прописка |
| `Order` | int ≥ 0 | позиция в слоте (и на стрипе) |
| `Mode` | см. TW-3.2 | режим отображения |
| `LastInternalMode` | {`DockPinned`, `DockUnpinned`, `Undock`} | последний внутренний режим — цель возврата из `Float`/`Window` (TW-5.6); связан с `Mode` инвариантом INV-7, дефолт `DockPinned` |
| `IsOpen` | bool | открыта/скрыта (в любом режиме) |
| `IsIconVisible` | bool | иконка присутствует на стрипе |
| `PairRatio` | (0..1) | предпочтение доли стороны в паре (TW-2.5) |
| `FloatingBounds` | Rect? | сохранённые границы для Float/Window |
| `ContentTree` | дерево таб-групп | контент панели (раздел 9); вырожденное — одна группа с одной вкладкой |

Плюс на уровне раскладки: состояние сторон (`Weight` — по одному на каждую из трёх сторон, TW-2.5), `QuickAccessSide ∈ {Left, Right}` (TW-8.1), `ActiveToolWindowId : string?` — активная панель, а также корень дерева док-зоны (спецификация document-area).

**TW-3.2** [Core] Режимы (`Mode`):

| Режим | Занимает место в раскладке | Автоскрытие при потере фокуса | Отдельное окно |
|---|---|---|---|
| `DockPinned` | да | нет | нет |
| `DockUnpinned` | да | да | нет |
| `Undock` | нет — оверлей поверх | да | нет |
| `Float` | нет | нет | да, поверх главного (owned) |
| `Window` | нет | нет | да, независимое |

**TW-3.3** [UI] Панель в режиме `Undock` — оверлей, прижатый к стороне прописки и занимающий её **полную протяжённость**: боковые — всю высоту рабочей области, включая область нижней панели, если та открыта; нижняя — всю ширину между стрипами, поверх нижних частей боковых панелей. Толщина оверлея — `Weight` стороны (TW-2.5): докированный слой и оверлей делят одну ширину, оверлей ровно накрывает докированную соседку по стороне. Показ оверлея рисуется поверх док-зоны и других панелей, не меняя ничьих размеров, и в правилах пары R1–R2 не участвует; ресайз оверлея (ручка появится в последующих фазах) — это `SetSideSize` стороны и потому меняет и ширину докированного слоя. Оверлей непрозрачен — панели под ним не просвечивают. (= IDEA new UI при дефолтных настройках: толщина sliding-оверлея фактически равна весу стороны — `UnifiedToolWindowWeights` затирает персональный вес окна при каждой активации, а ресайз пишет в оба места; полная протяжённость — по наблюдению владельца, 2026-07; источник толщины — трассировка владельца, 2026-07)

## 4. Инварианты

Проверяются после каждой операции; нарушение — ошибка ядра (property-тесты).

**INV-1** [Core] Каждая зарегистрированная панель имеет ровно одно состояние; прописка всегда валидна.

**INV-2** [Core] Слот имеет два независимых слоя: докированный и оверлейный. В каждом слоте открыто не более одной панели из {`DockPinned`, `DockUnpinned`} и, независимо, не более одной панели в режиме `Undock`. Открытые `Float`/`Window` не ограничены. (= IDEA: `ToolWindowManagerImpl.doShowWindow` скрывает только окна того же типа — условие `otherInfo.type == info.type` при совпадении anchor+isSplit; sliding сосуществует с docked)

**INV-3** [Core] `Order` образует плотную последовательность 0..n−1 внутри каждого слота (по всем панелям слота, независимо от открытости). (≈ IDEA: там плотность по стороне — Primary и Secondary делят одну нумерацию, а сегменты разделяет компаратор стрипа; плотность на слот строже при том же наблюдаемом порядке и делает TW-5.11 точным)

**INV-4** [Core] `Weight` сторон и `PairRatio` панелей ∈ (0..1); после R2 пара согласована (предпочтения Primary и Secondary дают в сумме 1). Эффективная доля пары (R1) лежит в (0..1) по построению — выводимая величина в проверке не нуждается (TW-2.5).

**INV-5** [Core] `ActiveToolWindowId` ссылается на существующую открытую панель либо null.

**INV-6** [Core] Открытая панель имеет `IsIconVisible = true` — в любом режиме (следствие TW-5.10 и TW-5.13).

**INV-7** [Core] Если `Mode` внутренний ({`DockPinned`, `DockUnpinned`, `Undock`}), то `LastInternalMode = Mode`; поле расходится с `Mode`, только пока панель во `Float`/`Window`. (= IDEA: `internalType` плюс неизменность `isAutoHide` при уходе в плавающие режимы — та же память возврата одним явным полем)

## 5. Операции

Все операции — команды ядра. UI (клики, меню, жесты DnD) только вызывает их (ADR-0004).
Значение перечисления вне своего домена в **программном** входе — команде либо конструкторе
состояния/дескриптора (`ToolWindowSide`/`ToolWindowGroup` слота, `ToolWindowMode`,
`QuickAccessSide`) — ошибка вызывающей стороны (`ArgumentOutOfRangeException`), как конечность
экранных границ (TW-5.9): иначе кастованное недоменное значение доживало бы до сериализации и
роняло `Serialize` на состоянии, произведённом штатной операцией. Файловый вход защищён
`Deserialize` (TW-10.5); значение, инъецированное в иммутабельную модель через `with` в обход
конструктора, остаётся ошибкой вызывающей стороны и не гарантируется. Команды документной зоны
следуют тому же контракту (`SplitTab` отвергает недоменное `SplitDirection`).

**TW-5.1** [Core] `Open(id, activate = true)`: устанавливает `IsOpen = true`. Если в слоте уже открыта другая панель **того же слоя** (докированного {`DockPinned`, `DockUnpinned`} либо оверлейного {`Undock`}) — та закрывается (её `Mode` и прочие поля не меняются). Панель другого слоя не затрагивается: `Undock`-оверлей сосуществует с докированной соседкой (INV-2). При `activate = true` панель становится `ActiveToolWindowId`. (= IDEA)

**TW-5.2** [Core] `Open` уже открытой панели с `activate = true` только активирует её; без активации — no-op.

**TW-5.3** [Core] `Close(id)`: `IsOpen = false`; если панель была активной, `ActiveToolWindowId = null`. Режим, веса, прописка сохраняются.

**TW-5.4** [UI] Клик по иконке на стрипе: если панель открыта — `Close`, иначе `Open(activate = true)`. Независимо от того, активна ли панель. (= IDEA)

**TW-5.5** [UI] Шорткат активации панели (назначается приложением): панель закрыта → `Open + activate`; открыта, но не активна → активировать; открыта и активна → `Close`. (= IDEA) Примечания (задача 3.2, 2026-07):

- шорткаты и кеймап — целиком у приложения (= IDEA: шорткат живёт в кеймапе, не в
  регистрации тул-окна); библиотека даёт публичную UI-команду с этой трёхпозиционной
  логикой поверх командного канала: активация переносит фокус (TW-6.6), автоскрытие
  достижимо (TW-6.1), доклад жизненному циклу — один на команду (ADR-0004);
- «открыта и активна» UI определяет по клавиатурному фокусу — фокус логически внутри
  панели или её иконки (предикат TW-6.1), — а не по хранимому `ActiveToolWindowId`:
  у эталона активная панель вычисляется из фокуса, и его правило «редактор активен или
  панель не активна → активировать, иначе закрыть» в наших терминах — ровно «фокус вне
  панели → активировать, внутри → закрыть»; хранимое поле отстаёт от фокуса — при фокусе
  в другой панели оно всё ещё называет прежнюю активную (сбрасывает его только активация
  документа, TW-6.5/DA-6.4), и проверка по нему закрывала бы панель без фокуса —
  расхождение с эталоном;
- подсказка шортката в тултипе иконки стрипа — необязательный провайдер приложения
  `id → отображаемая строка` (TW-6.4); без провайдера или для id без подсказки тултип —
  только `Title`.

**TW-5.6** [Core] `SetMode(id, mode)` допустим в любом состоянии, открытость панели не меняет, с переходами:
- в `Float`/`Window`: используются сохранённые `FloatingBounds`, а при их отсутствии — текущие экранные границы контента (передаёт UI); (= IDEA)
- из `Float`/`Window` в докируемые режимы: панель возвращается в слот прописки; если открыта — вытесняет по правилу TW-5.1; `FloatingBounds` сохраняются для будущих переходов;
- `Float` ↔ `Window`: окно пере-хостится с теми же границами;
- переход в любой внутренний режим записывает его в `LastInternalMode`; уход в `Float`/`Window` поле не трогает (INV-7). «Вернуть в док» из меню/жеста = `SetMode(id, LastInternalMode)` — панель возвращается в тот внутренний режим, из которого уходила; (= IDEA: `internalType`)
- переходы открытой панели между слоями (`Dock*` ↔ `Undock`) применяют вытеснение TW-5.1 в слое-приёмнике;
- смена режима закрытой панели просто меняет поле — проявится при следующем `Open`.

**TW-5.7** [Core] `Move(id, side, group, index)`: меняет прописку и вставляет панель в позицию `index` слота-приёмника (с нормализацией `Order` обоих слотов). Если панель открыта во внутреннем режиме ({`DockPinned`, `DockUnpinned`, `Undock`}) — остаётся открытой в новом слоте и вытесняет тамошнюю открытую панель своего слоя, докированного или оверлейного («движущаяся побеждает», TW-5.1); то есть `Undock`-панель вытесняет оверлейную соседку по слоту-приёмнику, но не докированную (INV-2). Для `Float`/`Window` меняется только прописка. Перемещение в свой же слот = переупорядочивание.

**TW-5.8** [Core] `Move` не затрагивает `Weight` сторон; `PairRatio` панели сохраняется как есть — предпочтение доли не зависит от ориентации стороны. Эффективная доля пары в слоте-приёмнике следует из R1 автоматически (TW-2.7): переезд открытой панели в пару наблюдаемо эквивалентен открытию в пару. (= IDEA: там move = hide + повторный show, прибытие проходит тот же путь образования пары; трассировка владельца, 2026-07)

**TW-5.9** [Core] Ресайзы: `SetSideSize(side, weight)` пишет `Weight` стороны (TW-2.6) — им же выражается будущий ресайз Undock-оверлея (TW-3.3); `SetSideRatio(side, p)` выполняет правило R2 (TW-2.7). `SetFloatingBounds(id, rect)` — для `Float`/`Window` (UI обязан вызывать при перемещении/ресайзе окна, допустим троттлинг). Экранные границы, передаваемые любой команде ядра — `SetFloatingBounds`, `screenBounds` в `SetMode` (TW-5.6), команды окон документов (DA-5.7, DA-5.8), — обязаны состоять из конечных чисел; NaN/∞ — ошибка вызывающей стороны, как выход дроби из диапазона (INV-4): состояние, произведённое ядром, всегда сериализуемо (TW-10.1).

**TW-5.10** [Core] `SetIconVisible(id, false)`: иконка исчезает со стрипа; открытая панель закрывается независимо от режима — `Float`/`Window`-окна тоже. Панель не покидает порядок слота: её `Order` сохраняется (INV-3). (= IDEA; решение владельца, 2026-07)

**TW-5.11** [Core] `SetIconVisible(id, true)`: иконка появляется на прежнем месте — скрытая панель не выводилась из порядка слота (TW-5.10), поэтому позиция сохраняется, в том числе относительно переупорядочиваний, случившихся за время скрытия. (= IDEA в простом случае; уточнение владельца, 2026-07)

**TW-5.12** [Core] `HideAll()`: закрывает все открытые панели внутренних режимов ({`DockPinned`, `DockUnpinned`, `Undock`}); открытые `Float`/`Window` не затрагиваются — их пользователь разместил сознательно, нередко вне главного окна. Обратный toggle «Restore» — вне ядра, собирается приложением из `Snapshot`/`Apply` (раздел 12). (= IDEA «Hide All Tool Windows»: `type.isInternal`; решение владельца, 2026-07)

**TW-5.13** [Core] Открытие панели с `IsIconVisible = false` (из быстрого доступа, по шорткату или из кода) автоматически устанавливает `IsIconVisible = true` — в любом режиме.

**TW-5.14** [Core] `Snapshot()` → неизменяемое состояние раскладки; `Apply(state, scope)` — атомарная замена с нормализацией (раздел 10) в границах области `scope` (TW-10.6); `ResetToDefaults()` — сборка состояния из дескрипторов регистрации. Поскольку состояние раскладки иммутабельно, снапшотом служит само удержанное состояние — отдельный копирующий API не требуется; `Apply` принимает любое состояние: сохранённое, сконструированное или удержанное ранее.

**TW-5.15** [Core] `SetQuickAccessSide(side)`, `side ∈ {Left, Right}` — перемещение кнопки быстрого доступа (TW-8.1).

**TW-5.17** [UI] Жесты перетаскивания — тонкий слой над командами ядра (ADR-0004; жест
не умеет ничего, чего не умеет команда). Правила общие для всех источников
перетаскивания; источники слот-жестов — иконка стрипа и полоса заголовка панели (кроме
заголовка в псевдоокне — там он ручка перемещения самого псевдоокна, TW-7.7), вкладки
деревьев — DA-9.7. Жест мультиоконен (задача 6.2): координаты, источники и цели живут в
**пространстве жеста** — экранные координаты на платформе с настоящими окнами, координаты
workspace в оверлее («экран» браузера — workspace, TW-7.7):

- перетаскивание начинается, когда указатель уходит от точки нажатия дальше порога
  (константа UI; = IDEA: 7 px, `MouseDragHelper.DRAG_START_DEADZONE`); движение в
  пределах порога — обычный клик со своим штатным действием (TW-5.4, TW-6.6). Нажатие,
  ставшее перетаскиванием, клик-действия источника не выполняет;
- жест до отпускания — чистая визуализация: образ у курсора (ghost) и маркер цели —
  лист-хром (TW-9.13), состояние не меняется, фокус не двигается. Захват указателя — на
  стабильном корне **окна-источника** (корень workspace для главного окна и псевдоокон,
  TopLevel плавающего окна), а не на пересобираемом лист-хроме: пересборка проекции жеста
  не рвёт, а отпускание всегда приходит владельцу захвата — в каком бы окне ни оказался
  указатель. Ghost на платформе с настоящими окнами — отдельное окно ОС поверх всех окон
  воркспейса (= эталон `DialogDragImageView`, opacity 0.85), в оверлее — слой workspace;
  маркер цели рисуется в оверлей-слое окна, содержащего цель;
- источники и цели — во всех окнах воркспейса: главное окно, плавающие панели, окна
  документов, псевдоокна (задача 6.2). Зона перекрытого окна не срабатывает: хит-тест
  отдаёт точку верхнему окну воркспейса, содержащему её; z-порядок настоящих окон
  аппроксимируется — плавающие окна выше главного (owned-Float у ОС всегда выше, показ
  нового окна без активации кладёт его поверх тоже), между собой — по последним
  активациям (точный z-порядок ОС платформа не отдаёт, допущение v1); псевдоокна —
  порядок детей оверлея, точный;
- цели иконки и заголовка — шесть сегментов стрипов (TW-1.2): зоны сегментов покрывают
  столбец стрипа целиком, свободное пространство делится между смежными сегментами,
  пустой сегмент получает зону своей нулевой позиции — в том числе пустой Primary над
  стоящим впритык Secondary: верхняя кромка стрипа резервируется под неё (= IDEA:
  скрытый разделитель позиции 0 остаётся живым как DnD-подсказка — трассировка TW-1.3);
  позиция вставки внутри сегмента —
  по ближайшей середине соседней иконки (= IDEA, `AbstractDroppableStripe`), отображение
  в плотный порядок — по TW-1.5 (вставка после ближайшего видимого предшественника);
  нижние сегменты растут снизу вверх (TW-1.4) — зоны отражены. Дроп панели внутреннего
  режима = ровно один `Move(id, side, group, index)` (TW-5.7); дроп в текущую позицию —
  без команды; дроп панели плавающего режима (`Float`/`Window`) — докировка: композиция
  `Move` + `SetMode(id, LastInternalMode)` (TW-5.6; = эталон: перетаскивание плавающего
  тул-окна на стрип докирует его; каждый вызов канала — одна команда с одним докладом,
  ADR-0004), тождества у неё нет — та же позиция при плавающем режиме всё равно докирует;
  активация при дропе в стрип не выполняется (= IDEA: дроп в стрип не активирует);
- цели отражают текущее состояние раскладки: внешнее изменение состояния во время жеста
  (прямое присвоение `State`, `Apply`, живая регистрация) жест не прерывает — проекция
  обновляется штатно, цели перестраиваются по обновлённой раскладке; жест отменяется,
  только если перетаскиваемый элемент покинул раскладку либо проекция сброшена целиком
  (смена `Registry`/`Lifecycle`). (≈ IDEA: там цели пересчитываются на каждое движение,
  а универсальной инвалидации сессии нет — защита точечная; у нас моменты пересчёта
  перечислимы, потому что визуал во время жеста не мутирует);
- отмена — `Esc` и потеря захвата указателя — не оставляет следа в состоянии: ни
  команды, ни активации, ни переноса фокуса (ADR-0004, ср. DA-E22; ≠ IDEA у вкладок
  редактора, = IDEA у кнопок стрипа и заголовков — там состояние тоже не мутируется до
  коммита). Отпускание вне всех целей — с задачи 6.2 не отмена, а команда «вынести»:
  для иконки и заголовка — переход во `Float` у точки отпускания (TW-7.8), для
  вкладки — новое окно документов у точки (DA-9.7); исключение-тождество — панель, уже
  открытая в плавающем слое (TW-7.8).

## 6. Автоскрытие и фокус

**TW-6.1** [UI] Панель в режиме `DockUnpinned` или `Undock` закрывается (`Close`), когда клавиатурный фокус перемещается **из неё** к элементу вне её визуального дерева — закрывается панель, потерявшая фокус. Открытая панель, не имеющая фокуса (восстановленная раскладкой, открытая программно без переноса фокуса — TW-6.6), сторонними перемещениями фокуса не закрывается: её закрывает клик по TW-6.2 (= IDEA: `hideIfAutoHideToolWindowLostFocus` реагирует на потерю фокуса самим окном; заострение против прочтения «любое перемещение наружу» — иначе первичная расстановка фокуса при показе окна закрывала бы восстановленные панели). Исключения — фокус ушёл, но панель не закрывается:
- фокус ушёл в попап/флайаут/контекстное меню, логический владелец которого находится внутри панели либо является её иконкой на стрипе (меню иконки — ручка самой панели, TW-5.16: без этого «Move to» открытой unpinned-панели из её же меню был бы недостижим);
- деактивировалось само приложение (переключение в другое приложение не закрывает панель; деактивация — уход активности из **всех** окон воркспейса: перенос фокуса между собственными окнами — плавающими панелями, окнами документов — обычная потеря фокуса и закрывает, задача 6.0); (= IDEA)
- идёт операция перетаскивания, инициированная из панели;
- открыт модальный диалог, принадлежащий панели.

**TW-6.2** [UI] Клик указателем по области вне панели (док-зона, другая панель, стрип) для unpinned-панели равносилен потере фокуса — панель закрывается, сам клик обрабатывается штатно. Уточнения:
- клик по иконке самой панели — жест TW-5.4 (toggle), автоскрытием он не дублируется: перехват вырождал бы toggle открытой панели в «закрыть + открыть»;
- перетаскивание сплиттера — жест ресайза, а не клик: автоскрытие не срабатывает ни на захвате, ни на отпускании (= IDEA: драг разделителя не двигает фокус и не является action — иначе ресайз пары, содержащей unpinned-панель, закрывал бы её же);
- жест перетаскивания DnD (TW-5.17) — не клик: отпускание, завершившее или отменившее перетаскивание, не закрывает автоскрываемые панели pointer-путём — тот же прецедент, что сплиттер; фокус-путь TW-6.1 жестом не затрагивается структурно — захват указателя фокус не двигает (≈ IDEA: у эталона исключение эмерджентно — кнопки нефокусируемы, drag-хелпер фокус не трогает; у нас исключение явное, задача 5.0);
- для цели, не принимающей фокус, закрытие фиксируется по завершении клика (на отпускании): закрытие на нажатии пересобирало бы лист-хром (TW-9.13) под уже начатым жестом и рвало пару «нажатие/отпускание»; цель, принимающая фокус, закрывает панель фокус-путём TW-6.1 уже на нажатии. Конечные состояния обоих путей совпадают с «закрыть, затем обработать клик». Принадлежность цели клика панелям — и множество открытых автоскрываемых панелей — фиксируются по нажатию, закрытие применяется на отпускании: команды, выполняемые самим жестом на отпускании (активация вкладки, toggle, перенос в закрытого владельца DA-E39), пересобирают лист-хром под ещё идущим событием (TW-9.13, DA-9.6) — проверка по отпусканию приняла бы клик внутри панели за внешний, — и могут открыть панель посреди жеста, которую то же отпускание не должно тут же захлопнуть: закрывается только панель, открытая на момент нажатия (задача 4.1).

**TW-6.3** [UI] `Esc` внутри тул-панели переводит фокус в текущую вкладку активного хоста док-зоны (DA-6.4, DA-E21). Для unpinned/undock-панелей это влечёт закрытие по TW-6.1. Когда у активного хоста нет текущей вкладки (пустое главное окно; окна документов пустыми не бывают, INV-D6), фокус остаётся на месте и закрытия нет — `Esc` без цели ничего не делает (= IDEA: семантика `Esc` — «вернуть фокус в документ», закрытие — лишь следствие его потери; наблюдение владельца, 2026-07). На платформе, где окна документов не материализуются вовсе, эффективный активный хост — главное окно (DA-6.4); цель в окне документов активирует его окно — настоящее окно ОС (задача 6.0) или поднятие псевдоокна в z-порядке (TW-7.7, задача 6.1) — по TW-6.6.

**TW-6.4** [UI] Открытые панели индицируются подсветкой их иконок на стрипе. Дополнительная индикация активной панели — на усмотрение темы; заголовок панели спецификацией не регламентируется. (= IDEA: заголовок при активации не меняется; уточнение владельца, 2026-07) Тултип иконки — `Title` панели; при заданном провайдере подсказки шортката (TW-5.5) дополняется его отображаемой строкой, оформление — усмотрение темы (задача 3.2, 2026-07).

**TW-6.5** [Core] Активация панели любым способом обновляет `ActiveToolWindowId`; активация документа в док-зоне сбрасывает его в null.

**TW-6.6** [UI] Активация ↔ фокус — две стороны одной проводки:

- активация панели командой (TW-5.1, TW-5.4, TW-5.5, TW-8.3) переводит клавиатурный фокус в её контент: первый фокусируемый элемент построенного вида, фолбэк — хост панели; цель, живущая в другом окне воркспейса (плавающая панель, окно документов), предварительно активирует своё окно ОС (задача 6.0), а цель в псевдоокне поднимает его в z-порядке (TW-7.7, задача 6.1). Без реального переноса фокуса автоскрытие TW-6.1 было бы недостижимо. Фокус, уже находящийся внутри панели, не трогается — активация не переставляет фокус внутри неё;
- команда, фактически переприсоединившая хост активной панели — другой слот либо другой слой, белый список TW-9.13 (`Move`, `SetMode`: id активной не меняется, а реатач теряет клавиатурный фокус побочно), — завершается тем же переносом фокуса в контент, **если до команды фокус был внутри панели**: командный канал восстанавливает потерянное, не отнимая фокус у стороннего владельца. Транзиентная потеря фокуса от белосписочного реатача — не потеря фокуса для TW-6.1: фокус уходит в никуда, а не к внешнему элементу (headless-проба, 2026-07), и автоскрытие не срабатывает — иначе `Move` открытой unpinned-панели закрывал бы её же (задача 3.2);
- реальное получение фокуса контентом панели UI сводит к активации (`Open` открытой панели = только активация, TW-5.2, TW-6.5) — зеркало DA-6.4 для панелей; клик по заголовку или нефокусируемому телу панели переводит фокус в неё и тем самым активирует. Для полосы заголовка — источника перетаскивания TW-5.17 — перенос фокуса фиксируется на отпускании: нажатие, ставшее перетаскиванием, панель не активирует, и отменённый header-жест не оставляет следа (= IDEA: `startDrag` не активирует, активация — только у успешного дропа во Float; для пользователя клик неотличим — активация на отпускании; задача 5.0). Фокус в попапах активационно нейтрален: не активирует панель-владельца попапа и не деактивирует прочие (симметрия с исключением TW-6.1);
- фокус переносит только командный канал UI: прямое присвоение состояния приложением (биндинг `State`, `Apply`) фокус не двигает — активация из кода без фокуса легальна (TW-9.3), такая панель не закрывается автоскрытием, пока сама не получит и не потеряет фокус (TW-6.1).

Сброс `ActiveToolWindowId` фокусом вне панелей выполняет проводка док-зоны: получение фокуса контентом вкладки сводится к `ActivateTab` (DA-6.4), который сбрасывает поле (TW-6.5). При фокусе в другой панели поле по-прежнему означает «последняя активная» — активность для TW-5.5 определяется по фокусу, не по полю.

## 7. Float и Window

**TW-7.1** [UI, desktop] `Float`: окно с владельцем-главным окном — всегда поверх главного, сворачивается вместе с ним, не отображается в панели задач ОС.

**TW-7.2** [UI, desktop] `Window`: независимое окно верхнего уровня — собственная позиция в z-порядке ОС и в панели задач.

**TW-7.3** [UI] Закрытие плавающего окна системной кнопкой = `Close(id)`. Режим панели не меняется — следующий `Open` снова откроет окно. (= IDEA)

**TW-7.4** [Core] При `Apply`/восстановлении `FloatingBounds` валидируются: если пересечение с видимой областью экранов меньше порога (задаёт UI-слой), окно позиционируется относительно главного окна с дефолтным размером. Состояние при этом обновляется. Готовый валидатор по экранам главного окна поставляет UI-слой библиотеки — приложение передаёт его в `Apply` (задача 6.0); для платформы без окон «экран» — рабочая область workspace, и валидатор поставляется по её границам (TW-7.7, задача 6.1) — им же лечится раскладка, принесённая с десктопа с экранными координатами. Без валидатора (headless, тесты) сохранённые границы принимаются как есть.

**TW-7.5** [UI] Закрытие главного окна закрывает все плавающие окна; раскладка перед этим снапшотится (панели остаются `IsOpen = true` в сохранённом состоянии и откроются при следующем запуске).

**TW-7.6** [Core] Платформенные возможности: `CanFloat`, `CanUseWindowed`. Эффективный режим отображения = деградация хранимого: `Window` → `Float` (если `CanUseWindowed = false`) → `Undock` (если и `CanFloat = false`); внутренние режимы не деградируют. Хранимое значение `Mode` не изменяется — раскладка, перенесённая обратно на «полную» платформу, восстанавливает исходное поведение. Деградация — чистая функция ядра (аргументы-возможности; ядро о платформе не знает, ADR-0002/0006); сами возможности объявляет UI-слой: публичные read-only `CanFloat`/`CanUseWindowed` на workspace, автодетект платформы (десктоп — обе true; браузер — `CanFloat = true` оверлеями TW-7.7, `CanUseWindowed = false`), настраиваемость приложением сознательно не даётся (задача 6.1, решение владельца). Семантика деградированной панели: эффективному режиму следует **представление** (где и как хостится контент), поведение — автоскрытие TW-6.1 — определяется хранимым `Mode`. ⚠ На v1-платформах жив только путь `Window → Float` (браузер); путь `Float → Undock` реализуется тем же правилом, но не наблюдаем до платформы без оверлеев — полная Undock-семантика деградированной панели (в т.ч. её неубираемость без автоскрытия) не специфицируется до появления такой платформы.

**TW-7.7** [UI, browser] В браузере `CanUseWindowed = false`, `CanFloat = true`: открытая `Float`-панель (и `Window`-панель, деградировавшая по TW-7.6) — псевдоокно в оверлей-слое workspace, поверх докированной раскладки и Undock-оверлея, под слоем drag-визуализации. Правила (задача 6.1):
- псевдоокно хостит тот же кэшированный декоратор — реатач слоя по белому списку TW-9.13; отдельного title bar у панельного псевдоокна нет: заголовок декоратора служит ручкой перемещения, «—» декоратора уже даёт `Close` (= TW-7.3), рамка псевдоокна — ресайз за края и углы;
- перемещение и ресайз — жесты в стиле TW-5.17: до отпускания — чистая визуализация, состояние не меняется; отпускание после фактического движения — ровно один `SetFloatingBounds` из итоговых границ (TW-5.9); `Esc` откатывает визуал без команды; клик без движения — без команды;
- z-порядок собственный: клик или получение клавиатурного фокуса внутри псевдоокна поднимает его наверх — оверлейный эквивалент активации окна ОС (TW-6.3, TW-6.6); z-порядок не персистится (ср. DA-7.1);
- «экран» браузера — рабочая область workspace: `FloatingBounds` интерпретируются в её координатах; экранные границы, которые UI передаёт командам (TW-5.6, TW-5.9), — тоже; валидатор TW-7.4 для платформы без окон поставляется по границам workspace; рендер клемпит псевдоокно в пределах workspace, не меняя состояние (TW-2.8);
- пункты меню недоступных режимов скрываются (TW-5.16): в браузере скрыт только `Window`.

Жесты DnD (TW-5.17, DA-9.7) внутри псевдоокон включены (задача 6.2): заголовки вкладок псевдоокон — источники, деревья псевдоокон — цели, зоны перекрытых псевдоокном областей не срабатывают (z-порядок — порядок детей оверлея, TW-5.17). Заголовок панельного псевдоокна остаётся ручкой перемещения самого псевдоокна и слот-жеста TW-5.17 не начинает: возврат в док — меню «View Mode»/«Dock» или перетаскивание вкладок.

**TW-7.8** [UI] Вытаскивание панели перетаскиванием = переход во `Float` (задача 6.2): отпускание слот-жеста TW-5.17 — заголовка **или иконки стрипа** (= эталон: оба источника ведёт один drag-хелпер, `processDragFinish` не различает; решение владельца, 2026-07) — вне всех целей, когда платформа умеет `Float` (TW-7.6), выполняет `SetFloatingBounds(id, bounds у точки отпускания)` + `SetMode(id, Float)` + `Open(id, activate)` — панель открывается плавающей у точки отпускания, активируется и получает фокус (TW-6.6; = эталон: активация — только у успешного дропа во Float). Размер bounds — текущий размер контента панели; у панели без размещённого контента (закрытой) — дефолт по правилу TW-7.4. Координаты — пространство жеста: экранные на десктопе, workspace в оверлее (псевдо-Float, TW-7.7). Для панели, уже открытой в плавающем слое, отпускание вне целей — тождество: ни команды, ни активации (перемещение её окна — жест самого окна: системный титул на десктопе, заголовок псевдоокна в оверлее). Обратное направление — дроп иконки/заголовка плавающей панели в зону стрипа — докировка `Move` + `SetMode(LastInternalMode)` (TW-5.17).

## 8. Быстрый доступ «⋯»

**TW-8.1** [UI] Кнопка «⋯» располагается в конце сегмента Left.Secondary; настройкой `QuickAccessSide` (TW-5.15) перемещается в конец Right.Secondary. (= IDEA; уточнение владельца, 2026-07)

**TW-8.2** [Core] Список быстрого доступа = все зарегистрированные панели с `IsIconVisible = false`, отсортированные по `Title` без учёта регистра (сравнение ordinal — порядок не зависит от локали машины, golden-тесты стабильны), при равенстве — по `Id`. По INV-6 все они закрыты. Прописка как ключ отвергнута: у скрытой панели она невидима пользователю, а задача списка — поиск инструмента по имени. (= IDEA по существу: там мнемоника, затем id алфавитно, но мнемоники почти всегда отсутствуют, а в нашей модели их нет вовсе; решение владельца, 2026-07)

**TW-8.3** [Core] Выбор элемента списка: `SetIconVisible(id, true)` + `Open(id, activate = true)` — панель возвращается на стрип и открывается.

**TW-8.4** [UI] Кнопка «⋯» скрывается, когда список пуст. (решение владельца, 2026-07)

## 9. Жизненный цикл контента

**TW-9.1** [Core] Дескриптор регистрации: `Id`, `Title`, `Icon`, дефолтные слот/`Order`/`Mode`/`PairRatio`, политика жизненного цикла, фабрика контента.

**TW-9.2** [Core] Политики — две независимые оси:
- создание: `OnFirstOpen` (дефолт) | `Eager` (при регистрации);
- удержание: `KeepWhileRegistered` (дефолт) | `DisposeOnClose` (контент освобождается при каждом закрытии и пересоздаётся при открытии).

Моменты: `Eager` создаёт контент в момент регистрации независимо от открытости панели;
`DisposeOnClose` освобождает на каждый переход панели из открытого состояния в закрытое
**любым путём** — `Close`, вытеснение (TW-5.1), скрытие иконки (TW-5.10), `HideAll`
(TW-5.12), применение макета (TW-10.7). Следствие: контент, созданный `Eager` у закрытой
панели, живёт до первого закрытия из открытого состояния либо до `Unregister` (TW-9.4) —
перехода «из открытости» у него не случалось.

Тело панели, живущее вкладкой в хосте док-зоны (TW-9.5, DA-8.1), из-под оси удержания
выходит: закрытие панели его не освобождает (DA-8.3) — освобождение происходит, когда
id покидает раскладку: `CloseTab` в любом дереве освобождает контент тела при любой
политике («контента без вкладки» не существует), а перенос — не освобождение (DA-5.4).
Следствие по образцу `Eager`: тело, перенесённое обратно в дерево уже закрытой
`DisposeOnClose`-панели, в момент переноса не освобождается — перехода из открытости
не было — и живёт до следующего цикла открытие → закрытие либо до `Unregister`.

**TW-9.3** [Core] Состояние раскладки не зависит от того, создан ли контент: панель может быть `IsOpen = false` с живым контентом и `IsOpen = true` с ещё не созданным (создание — обязанность слоя материализации).

**TW-9.4** [Core] `Unregister(id)`: панель закрывается, контент освобождается — при любой политике удержания, включая контент вкладок собственного дерева, — состояние сохраняется как «спящее» (раздел 10) — повторная регистрация того же `Id` подхватывает его. Собственное дерево панели инертно: структура сохраняется в спящей записи как есть (TW-9.9, TW-10.2), её вкладки спят до повторной регистрации (DA-9.4). Вкладки владельца в деревьях хостов док-зоны закрывает правило TW-9.10.

### Дерево таб-групп внутри панели

**TW-9.5** [Core] Контент панели — дерево таб-групп той же модели, что и док-зона (структура, операции и нормализация — спека document-area). Вырожденный случай — одна группа с единственной вкладкой-телом: полоса табов не отображается, панель выглядит обычной. Правила тела:

- **Тело панели** — вкладка с id, равным `Id` самой панели; её материализация делегируется фабрике тела (`IToolWindowContentFactory`, TW-9.1) — мост фабрик: пути отказа у тела нет (ср. DA-9.3), заявку на id тела даёт сама регистрация (TW-9.11).
- **Посев тела**: при регистрации и при примирении `Apply(Full)` (TW-10.3) панель с фабрикой тела получает тело корневой группой дерева, если id тела отсутствует во всей раскладке, а дерево панели пусто. Непустое дерево без тела не трогается — пользователь закрыл тело осознанно; id тела, живущий в другом хосте, посев отменяет (INV-D2). Посев — примирение, не починка: в отчёт `Apply` не попадает. Область `Arrangement` не сеет: контент вне её (TW-10.6), закрытое в живой сессии тело до следующего `Full`-примирения не воскресает.
- Панель **без фабрики тела** тела не получает: её дерево пусто, пока вкладки не принесут заявки `TabFactory` (TW-9.11) или переносы; пустое дерево легально (DA-8.4), контент такой панели материализует оболочка приложения вне ядра (TW-9.1).

**TW-9.6** [Core] Слой слотов оперирует панелью как атомом: перемещения по слотам, смены режимов и вытеснения переносят панель целиком вместе с её деревом; операции слотов внутрь дерева не заглядывают.

**TW-9.7** [Core] Владелец вкладки определяется реестром по её `id` (дескриптором), само дерево хранит только идентификаторы. Правило допуска `canHost`: док-зона принимает документы и панельные вкладки; панель P — только вкладки с владельцем P. Следствия: документ не может попасть в панель; вкладка чужой панели — тоже; панель внутри панели невыразима.

**TW-9.8** [Core] Перемещения вкладки: внутри своей панели; из панели в док-зону (включая отдельные окна документов); из док-зоны — только обратно в панель-владельца. Каждое перемещение — команда ядра с валидацией `canHost` (ADR-0004).

**TW-9.9** [Core] Структура дерева панели персистится как часть раскладки; содержимое вкладок восстанавливают фабрики (ADR-0003) — фабрика вправе не восстановить вкладку, тогда её id удаляется из дерева с нормализацией.

**TW-9.10** [Core] Вкладки панели, перенесённые в док-зону, не зависят от открытости и режима панели-владельца: закрытие или скрытие панели их не отзывает. `Unregister` владельца (TW-9.4) закрывает его вкладки в деревьях хостов док-зоны (с нормализацией; собственное дерево панели инертно — TW-9.4); повторная регистрация их не восстанавливает автоматически.

**TW-9.11** [Core] Владение вкладками — заявочное. Каждая регистрация контента вкладок —
контента док-зоны (документы) либо панели (её `TabFactory`) — заявляет свои `Id`
предикатом владения; владелец `Id` — единственная заявившая живая регистрация (TW-9.7,
DA-1). Правила:

- предикат заявки чист и стабилен, пока регистрация жива: владелец `Id` не меняется
  иначе как снятием или появлением регистраций;
- заявка одного `Id` двумя живыми регистрациями — ошибка приложения; обнаруживается при
  резолве владельца конкретного `Id` (единственность — контракт времени резолва, не
  регистрационный инвариант: домен `Id` открыт, заявки предиката неперечислимы);
- `Id` без заявки — вкладка спящего владельца (TW-10.2, DA-9.4), в том числе в живой
  сессии: документ, открытый до регистрации контента док-зоны, спит до её появления;
- регистраций контента док-зоны может быть несколько (модульное приложение) — при
  непересекающихся заявках;
- регистрация панели с фабрикой тела — **неявная заявка** на id тела (= `Id` панели,
  TW-9.5); заявки одной регистрации — неявная и предикат её `TabFactory` —
  объединяются: конфликт считается только между **разными** регистрациями, внутри
  одной приоритет у моста тела (материализация тела не получает пути отказа);
- регистрация панели без `TabFactory` иных заявок владения не даёт: её ранее спящие
  вкладки продолжают спать;
- в `Apply` и валидации инвариантов резолв владельца **не бросает**: конфликтный id
  считается неподтверждённым — вкладка остаётся на месте и спит, ошибка всплывает при
  первой операции или материализации этого id (канал отчёта `Apply` — для дефектов
  раскладки, не ошибок приложения).

**TW-9.13** [UI] Материализация инкрементальна. *Хост панели* — визуальный узел, владеющий её
декоратором и построенным видом контента; *view-state* — состояние вида, не отражённое в модели
раскладки (позиция скролла, каретка, выделение, раскрытые узлы). Правила:

- смена состояния раскладки не пересоздаёт и не переприсоединяет хосты остающихся панелей;
  клавиатурный фокус и view-state не теряются **как побочный эффект материализации** — фокус
  переносится только семантикой команд (активация: TW-5.1, TW-5.5);
- в частности, никого не переприсоединяют: смена активности, ресайзы (веса и доли — только
  перекладка геометрии), операции над иконками других панелей, переупорядочивание внутри
  слота, а также перестройка компоновки вокруг остающихся панелей — образование и распад пары
  стороны, появление и исчезновение соседней стороны или нижней панели;
- переприсоединение хоста открытой панели допустимо только при её фактическом перемещении:
  в другой слот (TW-5.7) либо в другой слой ({`Dock*`} ↔ `Undock` ↔ `Float`/`Window`, TW-5.6);
  `DockPinned` ↔ `DockUnpinned` слоя не меняет и хост не трогает;
- закрытие (любым путём — перечень TW-9.2) снимает хост с раскладки, открытие присоединяет его
  вновь; при `KeepWhileRegistered` хост удерживает построенный вид до переоткрытия или
  `Unregister` (TW-9.4). При `DisposeOnClose` хост не удерживает ни освобождённый контент, ни
  построенный над ним вид (TW-9.2) — удержание вида не должно продлевать жизнь контенту;
- лист-хром — кнопки стрипов, «⋯», сплиттеры, меню — вправе пересоздаваться при любой смене
  состояния: он не фокусируем (сплиттеры — сознательно, задача 2.2) и не несёт view-state;
  подсветка кнопки открытой панели — отрисовка состояния раскладки, не фокус (TW-6.4).
  Граница пересматривается, если хром станет фокусируемым (раздел 12).

## 10. Персистентность

**TW-10.1** [Core] Сериализуется полное состояние раскладки + `SchemaVersion` (int). Формат — JSON. Стабильность формата фиксируется golden-тестами (эталонные файлы в репозитории).

**TW-10.2** [Core] Состояния с `Id`, не имеющими зарегистрированного дескриптора («спящие»), сохраняются при десериализации и записываются обратно при сериализации (round-trip без потерь) — регистрация может произойти позже (плагины).

**TW-10.3** [Core] Примирение при `Apply` и при регистрации: сохранённое состояние ⊃ дескриптор → применяется сохранённое; зарегистрирован, но не сохранён → дефолты дескриптора, `Order` — после существующих панелей слота. Примирение при регистрации (панели или контента док-зоны) дополнительно сеет тело (TW-9.5) и выполняет переезд INV-D5 (DA-9.2): вкладка в дереве панели, чьего владельца заявки новой регистрации подтверждают иным, переезжает в конец текущей группы главного окна; канала отчёта у регистрации нет — асимметрия с `Apply` сознательная.

**TW-10.4** [Core] Нормализация при `Apply` (валидный результат из любого входа, без исключений):
- дубликаты `Id` — остаётся первый;
- несколько открытых панелей в одном **слое** слота — докированном или оверлейном (INV-2) — остаётся открытой панель с минимальным `Order`;
- `Order` уплотняются с сохранением относительного порядка;
- невалидные дроби (вне (0..1), NaN) — `PairRatio` панелей и `Weight` сторон — заменяются дефолтом поля (в духе N4 док-зоны; «клемп» открытого интервала потребовал бы произвольного эпсилона и производил бы вырожденные размеры);
- открытая панель со скрытой иконкой получает `IsIconVisible = true` (следствие TW-5.13, INV-6);
- нечисловые `FloatingBounds` сбрасываются в null — следующий переход в `Float`/`Window` возьмёт экранные границы от UI (TW-5.6);
- невалидный `ActiveToolWindowId` → null; проверяется после разрешения конфликтов открытости (актив мог ссылаться на панель, закрытую предыдущим правилом);
- `LastInternalMode`, нарушающий INV-7 или невалидный, исправляется: при внутреннем `Mode` — самим `Mode`, иначе `DockPinned`.

Нормализация не молчалива: `Apply` возвращает отчёт о применённых исправлениях — приложение решает, логировать их или игнорировать (приложение, желающее «правильный макет или ничего», строит строгий режим поверх проверки пустоты отчёта). Каждое исправление отражается в отчёте ровно один раз; вторичные переназначения активности, вызванные починкой (например, смена текущей вкладки хоста после дедупликации), отдельных записей не порождают. Штатное поведение — вытеснение TW-5.1, примирение TW-10.3, сброс `ActiveToolWindowId` областью `Arrangement` (TW-10.7) — исправлением не является и в отчёт не попадает. Сериализация, произведённая самим ядром, проходит `Apply` без единого исправления (round-trip-инвариант, property-тест).

**TW-10.5** [Core] `SchemaVersion` новее поддерживаемой либо неразбираемый документ → ошибка загрузки, вызывающая сторона получает её явно и решает (типично — `ResetToDefaults`). Старые версии — через цепочку миграций.

Граница «лечим ↔ отвергаем» проходит по представимости: лечится с отчётом (TW-10.4, DA-9.2) только дефект, для которого в типизированной модели существует представимое невалидное значение и правило починки — нечисловое значение числового поля (→ NaN), висячая ссылка (несуществующий id в активности или текущей вкладке), отрицательный `Order`. Значение вне домена модели неразбираемо наравне с битым документом: неизвестный член перечисления, пустой или нестроковый идентификатор сущности (панели, вкладки), небулево значение флага, отрицательный индекс окна документов, отсутствующая или пустая обязательная составляющая — текущая вкладка окна документов, составляющая его `Bounds` (`null` эквивалентен отсутствию). Для таких значений нет ни представления, ни правила починки, а канала отчёта у загрузки нет — молчаливая подстановка нарушала бы контракт TW-10.4.

Неизвестные поля при чтении игнорируются. Добавление поля без смены `SchemaVersion` допустимо, только если документ **без** поля корректен для новой версии (безопасный дефолт) и документ **с** полем корректен для старой (поле игнорируемо без изменения смысла соседних); изменение смысла существующего поля или неаддитивная перестройка — всегда `SchemaVersion + 1`. Асимметрия с TW-10.2 сознательная: неизвестные **сущности** (спящие панели, спящие вкладки) переживают round-trip без потерь, неизвестные **поля** — нет.

**TW-10.6** [Core] Области применения состояния (`scope` в TW-5.14):
- `Full` — всё состояние: размещение панелей, геометрия сторон, деревья контента панелей и док-зоны. Используется для сохранения/восстановления сессии (перезапуск приложения).
- `Arrangement` — только размещение: слоты/`Order`/режимы/открытость/иконки панелей, геометрия сторон, `QuickAccessSide` — **без** деревьев контента и открытых вкладок. Используется именованными макетами: смена макета переставляет панели, но не закрывает открытые документы и вкладки.

`Apply(state, Arrangement)` берёт из `state` только размещение, сохраняя текущие деревья; контент открывшихся панелей материализуется лениво как обычно (TW-9.3).

**TW-10.7** [Core] Слияние `Arrangement` с текущим состоянием. Макет частичен: затрагиваются только id, упомянутые в нём.

| Случай | Поведение |
|---|---|
| id в макете, зарегистрирован | размещение (слот/`Order`/режим/открытость/иконка) — из макета; `ContentTree`, `PairRatio` и `FloatingBounds` остаются текущими — `Arrangement` переносит размещение, не геометрию панели (TW-10.6); конфликт открытости в слоте решает штатное вытеснение TW-5.1 |
| id в макете, не зарегистрирован | новая «спящая» запись создаётся из макета целиком — текущих значений не существует (TW-10.2); её `FloatingBounds` проходят валидацию TW-7.4 |
| id зарегистрирован, в макете отсутствует | собственные атрибуты не переписываются; коллективные свойства пересчитываются — см. ниже |
| вкладки панелей, живущие в док-зоне | вне области `Arrangement` — остаются где были (TW-9.8) |
| `ActiveToolWindowId` | сбрасывается в null; фокус назначает UI |

У не упомянутой панели различаются свойства двух природ. **Собственные атрибуты** (слот прописки, режим и `LastInternalMode`, `IsIconVisible`, `PairRatio`, `FloatingBounds`, `ContentTree`) принадлежат панели — макет их не переписывает. **Коллективные свойства** (позиция в плотном порядке, единственность открытой в слое) принадлежат слоту, а не панели, — они пересчитываются:

- порядок каждого слота пересобирается: сначала упомянутые макетом панели в порядке макета, затем не упомянутые — в прежнем относительном порядке между собой (тот же принцип «неизвестное макету — в конец», что и в TW-10.3);
- открытость меняется только косвенно: если макет открывает панель в слое, где открыта не упомянутая, — штатное вытеснение TW-5.1 (E24); если макет этот слой не занимает — открытая панель остаётся открытой.

Несогласованности внутри самого макета чинит нормализация TW-10.4 (с отчётом); вытеснение TW-5.1 и пересборка порядка — штатное поведение слияния, не исправление: применение макета без внутренних дефектов возвращает пустой отчёт (E24). Новых механизмов слияние не вводит — каждый случай сводится к уже определённым правилам.

## 12. Вне рамок v1

- Widescreen-геометрия (TW-2.2), скрытие стрипов целиком, анимации выезда.
- Именованные раскладки с UI управления и toggle «Hide All / Restore» (TW-5.12) — ядро покрывает Snapshot/Apply, приложение может строить поверх.
- Цепочка миграций схемы персистентности (TW-10.5): до первого повышения `SchemaVersion` миграций не существует — документ с любой версией, кроме текущей, отвергается явной ошибкой загрузки, и приложение реагирует штатно (типично `ResetToDefaults`; образец — мини-IDE демо, задача 7.0). Каркас цепочки строится вместе со `SchemaVersion = 2` — раньше он был бы мёртвым кодом (решение владельца, 2026-07).
- Визуальные drag-эффекты сверх ghost-плашки и маркера цели (TW-5.17): живой reorder-превью иконок/вкладок, скрытие утаскиваемого элемента, морфинг образа под целевой контейнер (эталон: `DragSession` переключает иконку ↔ миниатюру контента, `processDropOver` подменяет образ превью цели). Чисто визуальные оверрайды поверх проекции архитектуре не противоречат — состояние не трогают, отмена откатывает их бесплатно, — но требуют реаплая после каждой пересборки лист-хрома (второй источник визуальной правды на время жеста); кандидат v2. Мутация самой раскладки посреди жеста — как у эталонного reorder'а вкладок редактора — запрещена всегда: `State` наблюдаем приложением, фантомные переходы утекали бы в персистентность и жизненный цикл (ADR-0004, DA-E22).
- Спринг-лоадинг (открытие закрытой панели наведением жеста на её иконку) — отсутствует и у эталона (целевой поиск по обоим drag-путям, трассировка 2026-07); открытие панели — команда ядра, то есть изменение состояния посреди жеста, ломающее TW-5.17/DA-E22. Не поддерживается.
- Восстановление вытесненной панели при закрытии заменившей (SideStack старого UI IDEA): в new UI IDEA выключено реестром `ide.enable.toolwindow.stack` — не поддерживается и у нас.
- Растяжка стрипа с подписями под иконками (драг края стрипа, icon + text — new UI IDEA; включается пунктом «Show Tool Window Names» меню иконки — трассировка владельца, 2026-07): опция представления уровня приложения, не состояние раскладки (ср. TW-2.2); `Title` для подписей уже в модели — кандидат v2.
- Секция «Resize» полного меню (Stretch to Left/Right/Top/Bottom с недоступностью неприменимых направлений, Maximize Tool Window — = IDEA; трассировка владельца, 2026-07): команд ядра нет — Stretch-шаги выразимы поверх `SetSideSize`/`SetSideRatio`, Maximize потребует нового пункта спеки; кандидат последующих фаз.
- Переполнение стрипа иконками (суммарная высота сегментов больше высоты окна) не обрабатывается — ни скролла, ни сжатия, ни переноса лишних в «⋯» (= IDEA: там это необработанный вырожденный случай Swing-раскладки — эталон полагается на то, что реалистичное число панелей на сторону не превысит высоту экрана; трассировка владельца, 2026-07). Graceful-обработка — кандидат v2, если допущение опровергнется.
- Подсказка шортката в пунктах меню панели и в списке «⋯» (эталон показывает шорткаты в меню): тултип иконки покрыт TW-5.5/TW-6.4 (задача 3.2); меню — кандидат v2.
- Пер-панельные размеры докированного/Undock-представления (настройка IDEA `rememberSizeForEachToolWindowNewUI`, в new UI дефолт `false` — трассировка владельца, 2026-07): v1 фиксирует один режим — вес общий на сторону (TW-2.5, TW-3.3); пер-оконные размеры — кандидат v2, возврат аддитивен по правилу эволюции схемы (TW-10.5).
- Транзиентный кэш пропорции пары (`idToSplitProportion` эталона: точное восстановление позиции сплиттера при повторном показе Secondary в рамках сессии, умирает при пересоздании `ToolWindowPane`) — не берём: R2 «учим обе» кодирует живую пропорцию в персистентных `PairRatio` точно, поэтому восстановление у нас переживает и рестарт, и полную замену layout — строже эталона (трассировка владельца, 2026-07).

  `ToolWindowContentUiType` tabs|combo, у IDEA персистится пер-окно —
