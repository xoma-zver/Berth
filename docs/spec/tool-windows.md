# Спецификация поведения: tool windows (тул-панели)

**Статус: ЧЕРНОВИК v0.1 — на ревью.**

Документ определяет *поведение* подсистемы тул-панелей (не API и не визуальный дизайн).
Соглашения:

- Нормативные пункты имеют стабильные идентификаторы вида `TW-<раздел>.<номер>`. Идентификаторы не переиспользуются: удалённый пункт помечается «отменён», новый получает следующий номер.
- Каждый пункт — будущий тест (или группа тестов) с тем же идентификатором в имени.
- Метки: **[Core]** — проверяется в ядре без UI; **[UI]** — проверяется в слое Avalonia (headless).
- **(= IDEA)** — соответствует наблюдаемому поведению IntelliJ IDEA; **(≈ IDEA)** — наше упрощение/отклонение; **⚠** — решение под вопросом, требуется мнение ревьюера.

---

## 1. Термины и слоты

- **Тул-панель (tool window)** — вспомогательное представление с заголовком, привязанное к краю рабочей области (в отличие от документов в центральной зоне).
- **Сторона (Side)**: `Left`, `Right`, `Bottom` — где открывается контент панели.
- **Группа (Group)**: `Primary`, `Secondary` — половина стороны.
- **Слот** — пара (Side, Group). Слотов ровно шесть.
- **Стрип (stripe)** — вертикальная полоса иконок у левого/правого края главного окна.
- **Прописка (placement)** — слот + порядковый номер, закреплённые за панелью. Определена всегда, в любом режиме.

**TW-1.1** [Core] Существует ровно шесть слотов. Соответствие слотов, стрипов и зон контента:

| Слот | Иконка на стрипе | Где открывается контент | Термины IDEA |
|---|---|---|---|
| Left.Primary | левый стрип, верхний сегмент | левая панель, верхняя часть | anchor=left, split=false |
| Left.Secondary | левый стрип, под разделителем | левая панель, нижняя часть | anchor=left, split=true |
| Right.Primary | правый стрип, верхний сегмент | правая панель, верхняя часть | anchor=right, split=false |
| Right.Secondary | правый стрип, под разделителем | правая панель, нижняя часть | anchor=right, split=true |
| Bottom.Primary | левый стрип, нижний сегмент | нижняя панель, левая часть | anchor=bottom, split=false |
| Bottom.Secondary | правый стрип, нижний сегмент | нижняя панель, правая часть | anchor=bottom, split=true |

**TW-1.2** [UI] Порядок сегментов левого стрипа сверху вниз: иконки Left.Primary → разделитель → иконки Left.Secondary → растяжимое пустое место → иконки Bottom.Primary → кнопка быстрого доступа «⋯» (см. раздел 8). Правый стрип зеркально: Right.Primary → разделитель → Right.Secondary → растяжка → Bottom.Secondary. (= IDEA, new UI)

**TW-1.3** [UI] Разделитель между сегментами Primary и Secondary отображается, только если оба сегмента непусты.

**TW-1.4** [UI] Внутри сегмента иконки упорядочены по `Order` панелей (см. TW-3.1).

## 2. Геометрия рабочей области

**TW-2.1** [UI] Компоновка рабочей области (все элементы опциональны, кроме док-зоны):

```
┌──┬─────────────────────────────────┬──┐
│с │ ┌───────┬───────────┬─────────┐ │с │
│т │ │ левая │ док-зона  │ правая  │ │т │
│р │ │ панель│(документы)│ панель  │ │р │
│и │ ├───────┴───────────┴─────────┤ │и │
│п │ │        нижняя панель        │ │п │
└──┴─┴─────────────────────────────┴─┴──┘
```

Нижняя панель занимает всю ширину между стрипами; боковые панели располагаются над ней. (= IDEA при выключенной опции «Widescreen tool window layout»)

**TW-2.2** ⚠ Вариант «widescreen» (боковые панели полной высоты, нижняя — между ними) в v1 не реализуется; зафиксировать как возможную опцию v2. Подтвердить на ревью.

**TW-2.3** [UI] Боковая панель (левая/правая), когда открыты обе её группы, делится **вертикальным** стеком: контент Primary сверху, Secondary снизу, между ними сплиттер. Если открыта одна группа — она занимает всю панель. (= IDEA)

**TW-2.4** [UI] Нижняя панель, когда открыты обе группы, делится **горизонтально**: Primary слева, Secondary справа, между ними сплиттер. Если открыта одна — занимает всю панель. (= IDEA)

**TW-2.5** [Core] Размеры описываются весами, а не пикселями:
- `Weight` панели — доля рабочей области: для Side=Left/Right — доля ширины, для Bottom — доля высоты. Диапазон (0..1), дефолт задаёт дескриптор (рекомендуемые дефолты: 0.25 для боковых, 0.30 для нижней).
- `SideWeight` — доля Primary-контента внутри пары групп, когда открыты обе. Дефолт 0.5.

**TW-2.6** [Core] Ширина (высота) стороны определяется весом *последней открытой или последней ресайзнутой* панели этой стороны. Ресайз стороны пользователем записывает новый `Weight` во **все открытые** докированные панели этой стороны. ⚠ (≈ IDEA — точная семантика IDEA не верифицирована; правило выбрано за детерминизм)

**TW-2.7** [Core] Ресайз сплиттера между группами записывает новый `SideWeight` в обе открытые панели пары. Читается `SideWeight` открытой Primary-панели (fallback 0.5).

**TW-2.8** [UI] Минимальные размеры контента панелей и док-зоны задаются константами слоя UI; при нехватке места веса клемпируются на этапе рендера, состояние не изменяется.

## 3. Модель состояния

**TW-3.1** [Core] Состояние панели:

| Поле | Тип/диапазон | Смысл |
|---|---|---|
| `Id` | непустая строка | стабильный идентификатор регистрации |
| `Side`, `Group` | слот | прописка |
| `Order` | int ≥ 0 | позиция в слоте (и на стрипе) |
| `Mode` | см. TW-3.2 | режим отображения |
| `IsOpen` | bool | открыта/скрыта (в любом режиме) |
| `IsIconVisible` | bool | иконка присутствует на стрипе |
| `Weight`, `SideWeight` | (0..1) | см. TW-2.5 |
| `FloatingBounds` | Rect? | сохранённые границы для Float/Window |

Плюс на уровне раскладки: `ActiveToolWindowId : string?` — активная (сфокусированная) панель.

**TW-3.2** [Core] Режимы (`Mode`):

| Режим | Занимает место в раскладке | Автоскрытие при потере фокуса | Отдельное окно |
|---|---|---|---|
| `DockPinned` | да | нет | нет |
| `DockUnpinned` | да | да | нет |
| `Undock` | нет — оверлей поверх | да | нет |
| `Float` | нет | нет | да, поверх главного (owned) |
| `Window` | нет | нет | да, независимое |

**TW-3.3** [UI] Панель в режиме `Undock` отображается оверлеем ровно в той области, которую заняла бы в `DockPinned` (по своим весам), поверх док-зоны и соседних панелей, не влияя на их размеры. (= IDEA)

## 4. Инварианты

Проверяются после каждой операции; нарушение — ошибка ядра (property-тесты).

**INV-1** [Core] Каждая зарегистрированная панель имеет ровно одно состояние; прописка всегда валидна.

**INV-2** [Core] В каждом слоте открыто не более одной панели с режимом из {`DockPinned`, `DockUnpinned`, `Undock`}. Открытые `Float`/`Window` не ограничены. ⚠ включение `Undock` в правило вытеснения — наш выбор ради единообразия; поведение IDEA для sliding-режима не верифицировано.

**INV-3** [Core] `Order` образует плотную последовательность 0..n−1 внутри каждого слота (по всем панелям слота, независимо от открытости).

**INV-4** [Core] `Weight`, `SideWeight` ∈ (0..1).

**INV-5** [Core] `ActiveToolWindowId` ссылается на существующую открытую панель либо null.

**INV-6** [Core] Открытая панель с режимом из {`DockPinned`, `DockUnpinned`, `Undock`} имеет `IsIconVisible = true` (следствие TW-5.13).

## 5. Операции

Все операции — команды ядра. UI (клики, меню, жесты DnD) только вызывает их (ADR-0004).

**TW-5.1** [Core] `Open(id, activate = true)`: устанавливает `IsOpen = true`. Если в слоте панели уже открыта другая панель с режимом из INV-2 — та закрывается (её `Mode`, веса и прочие поля не меняются). При `activate = true` панель становится `ActiveToolWindowId`. (= IDEA)

**TW-5.2** [Core] `Open` уже открытой панели с `activate = true` только активирует её; без активации — no-op.

**TW-5.3** [Core] `Close(id)`: `IsOpen = false`; если панель была активной, `ActiveToolWindowId = null`. Режим, веса, прописка сохраняются.

**TW-5.4** [UI] Клик по иконке на стрипе: если панель открыта — `Close`, иначе `Open(activate = true)`. Независимо от того, активна ли панель. (= IDEA)

**TW-5.5** [UI] Шорткат активации панели (назначается приложением): панель закрыта → `Open + activate`; открыта, но не активна → активировать; открыта и активна → `Close`. (= IDEA)

**TW-5.6** [Core] `SetMode(id, mode)` допустим в любом состоянии, открытость панели не меняет, с переходами:
- в `Float`/`Window`: используются сохранённые `FloatingBounds`, а при их отсутствии — текущие экранные границы контента (передаёт UI); (= IDEA)
- из `Float`/`Window` в докируемые режимы: панель возвращается в слот прописки; если открыта — вытесняет по правилу TW-5.1; `FloatingBounds` сохраняются для будущих переходов;
- `Float` ↔ `Window`: окно пере-хостится с теми же границами;
- смена режима закрытой панели просто меняет поле — проявится при следующем `Open`.

**TW-5.7** [Core] `Move(id, side, group, index)`: меняет прописку и вставляет панель в позицию `index` слота-приёмника (с нормализацией `Order` обоих слотов). Если панель открыта в докируемом режиме — остаётся открытой в новом слоте и вытесняет тамошнюю открытую панель («движущаяся побеждает»). Для `Float`/`Window` меняется только прописка. Перемещение в свой же слот = переупорядочивание.

**TW-5.8** [Core] При `Move` между сторонами разной ориентации (боковая ↔ нижняя) `Weight` сохраняет числовое значение и переинтерпретируется (доля ширины ↔ доля высоты). (≈ IDEA)

**TW-5.9** [Core] Ресайзы: `SetSideSize(side, weight)` и `SetSideRatio(side, ratio)` — семантика записи по TW-2.6/TW-2.7. `SetFloatingBounds(id, rect)` — для `Float`/`Window` (UI обязан вызывать при перемещении/ресайзе окна, допустим троттлинг).

**TW-5.10** [Core] `SetIconVisible(id, false)`: иконка исчезает со стрипа; открытая панель в докируемом режиме закрывается; открытые `Float`/`Window` остаются открытыми. ⚠ (≈ IDEA — поведение IDEA для плавающих не верифицировано; выбрано, чтобы не закрывать внезапно окно на втором мониторе)

**TW-5.11** [Core] `SetIconVisible(id, true)`: иконка возвращается в конец своего сегмента (Order = максимальный в слоте).

**TW-5.12** [Core] `HideAll()`: закрывает все открытые панели во всех режимах. (= IDEA «Hide All Tool Windows»)

**TW-5.13** [Core] Открытие панели в докируемом режиме с `IsIconVisible = false` (например, из быстрого доступа или по шорткату) автоматически устанавливает `IsIconVisible = true`.

**TW-5.14** [Core] `Snapshot()` → неизменяемое состояние раскладки; `Apply(state)` — атомарная замена с нормализацией (раздел 10); `ResetToDefaults()` — сборка состояния из дескрипторов регистрации.

## 6. Автоскрытие и фокус

**TW-6.1** [UI] Панель в режиме `DockUnpinned` или `Undock` закрывается (`Close`), когда фокус перемещается к элементу вне её визуального дерева, **кроме** случаев:
- фокус ушёл в попап/флайаут/контекстное меню, логический владелец которого находится внутри панели;
- деактивировалось само приложение (переключение в другое приложение не закрывает панель); (= IDEA)
- идёт операция перетаскивания, инициированная из панели;
- открыт модальный диалог, принадлежащий панели.

**TW-6.2** [UI] Клик указателем по области вне панели (док-зона, другая панель, стрип) для unpinned-панели равносилен потере фокуса — панель закрывается, затем обрабатывается сам клик.

**TW-6.3** [UI] `Esc` внутри тул-панели переводит фокус в док-зону (в последний активный документ). Для unpinned/undock-панелей это влечёт закрытие по TW-6.1. (= IDEA)

**TW-6.4** [UI] Активная панель визуально выделяет заголовок; активация панели снимает выделение с предыдущей.

**TW-6.5** [Core] Активация панели любым способом обновляет `ActiveToolWindowId`; активация документа в док-зоне сбрасывает его в null.

## 7. Float и Window

**TW-7.1** [UI, desktop] `Float`: окно с владельцем-главным окном — всегда поверх главного, сворачивается вместе с ним, не отображается в панели задач ОС.

**TW-7.2** [UI, desktop] `Window`: независимое окно верхнего уровня — собственная позиция в z-порядке ОС и в панели задач.

**TW-7.3** [UI] Закрытие плавающего окна системной кнопкой = `Close(id)`. Режим панели не меняется — следующий `Open` снова откроет окно. (= IDEA)

**TW-7.4** [Core] При `Apply`/восстановлении `FloatingBounds` валидируются: если пересечение с видимой областью экранов меньше порога (задаёт UI-слой), окно позиционируется относительно главного окна с дефолтным размером. Состояние при этом обновляется.

**TW-7.5** [UI] Закрытие главного окна закрывает все плавающие окна; раскладка перед этим снапшотится (панели остаются `IsOpen = true` в сохранённом состоянии и откроются при следующем запуске).

**TW-7.6** [Core] Платформенные возможности: `CanFloat`, `CanUseWindowed`. Эффективный режим отображения = деградация хранимого: `Window` → `Float` (если `CanUseWindowed = false`) → `Undock` (если и `CanFloat = false`). Хранимое значение `Mode` не изменяется — раскладка, перенесённая обратно на «полную» платформу, восстанавливает исходное поведение.

**TW-7.7** [UI, browser] В браузере `CanUseWindowed = false`; `Float` реализуется псевдоокном в оверлей-слое приложения (перемещение за заголовок, ресайз за края, свой z-порядок). Пункты меню недоступных режимов скрываются.

**TW-7.8** [UI] Вытаскивание панели перетаскиванием за заголовок = `SetMode(id, Float)` c `FloatingBounds` у точки отпускания (фаза DnD; до неё — через меню).

## 8. Быстрый доступ «⋯»

**TW-8.1** [UI] Кнопка «⋯» размещается в нижнем сегменте левого стрипа, после иконок Bottom.Primary. (= IDEA new UI)

**TW-8.2** [Core] Список быстрого доступа = все зарегистрированные панели с `IsIconVisible = false`, отсортированные по заголовку. Открытые в данный момент (например, плавающие) помечаются.

**TW-8.3** [Core] Выбор элемента списка: `SetIconVisible(id, true)` + `Open(id, activate = true)` — панель возвращается на стрип и открывается.

**TW-8.4** [UI] Кнопка «⋯» отображается всегда, в том числе при пустом списке (пустой список — с поясняющим текстом). ⚠ Альтернатива: скрывать при пустом списке.

## 9. Жизненный цикл контента

**TW-9.1** [Core] Дескриптор регистрации: `Id`, `Title`, `Icon`, дефолтные слот/`Order`/`Mode`/`Weight`, политика жизненного цикла, фабрика контента.

**TW-9.2** [Core] Политики — две независимые оси:
- создание: `OnFirstOpen` (дефолт) | `Eager` (при регистрации);
- удержание: `KeepWhileRegistered` (дефолт) | `DisposeOnClose` (контент освобождается при каждом закрытии и пересоздаётся при открытии).

**TW-9.3** [Core] Состояние раскладки не зависит от того, создан ли контент: панель может быть `IsOpen = false` с живым контентом и `IsOpen = true` с ещё не созданным (создание — обязанность слоя материализации).

**TW-9.4** [Core] `Unregister(id)`: панель закрывается, контент освобождается, состояние сохраняется как «спящее» (раздел 10) — повторная регистрация того же `Id` подхватывает его.

## 10. Персистентность

**TW-10.1** [Core] Сериализуется полное состояние раскладки + `SchemaVersion` (int). Формат — JSON. Стабильность формата фиксируется golden-тестами (эталонные файлы в репозитории).

**TW-10.2** [Core] Состояния с `Id`, не имеющими зарегистрированного дескриптора («спящие»), сохраняются при десериализации и записываются обратно при сериализации (round-trip без потерь) — регистрация может произойти позже (плагины).

**TW-10.3** [Core] Примирение при `Apply` и при регистрации: сохранённое состояние ⊃ дескриптор → применяется сохранённое; зарегистрирован, но не сохранён → дефолты дескриптора, `Order` — после существующих панелей слота.

**TW-10.4** [Core] Нормализация при `Apply` (валидный результат из любого входа, без исключений):
- дубликаты `Id` — остаётся первый;
- несколько открытых докируемых в одном слоте — остаётся открытой панель с минимальным `Order`;
- `Order` уплотняются с сохранением относительного порядка; веса клемпируются в допустимый диапазон; невалидный `ActiveToolWindowId` → null.

**TW-10.5** [Core] `SchemaVersion` новее поддерживаемой либо неразбираемый документ → ошибка загрузки, вызывающая сторона получает её явно и решает (типично — `ResetToDefaults`). Старые версии — через цепочку миграций.

| E2 | A открыта в Left.Primary; B открыта в Left.Secondary | `Open(C)` в Left.Primary | вытеснена только A; B открыта |
| E3 | A открыта (pinned), единственная на стороне | `Close(A)` | сторона пуста, док-зона расширяется; `Weight` A сохранён |
| E4 | A открыта в Left.Primary, B открыта в Right.Primary | `Move(A, Right, Primary, 0)` | A открыта в Right.Primary, B закрыта («движущаяся побеждает»), прописки корректны |
| E5 | A в режиме Float, открыта | `Move(A, Bottom, Secondary, 0)` | окно A не тронуто; изменилась только прописка |
| E6 | A в Float, в слоте прописки открыта B | `SetMode(A, DockPinned)` | A докируется в свой слот, B закрыта |
| E7 | A открыта (pinned) | `SetIconVisible(A, false)` | A закрыта, иконки нет, A в списке «⋯» |
| E8 | A в Window, открыта | `SetIconVisible(A, false)` | окно A живо; иконки нет; A в списке «⋯» с пометкой ⚠ см. TW-5.10 |
| E9 | A открыта unpinned | фокус в документ | A закрывается (TW-6.1) |
| E10 | A открыта unpinned, из неё открыт комбобокс | выбор в комбобоксе | A остаётся открытой |
| E11 | сохранёнка: A и B открыты в одном слоте | `Apply` | открыта одна с меньшим `Order` (TW-10.4) |
| E12 | сохранёнка содержит неизвестный id X | `Apply`; затем `Serialize` | X «спит» и присутствует в выводе (TW-10.2) |
| E13 | сохранёнка: `Mode = Window`, платформа browser | `Open` | эффективно Float-оверлей; `Mode` в состоянии — `Window` (TW-7.6) |
| E14 | `FloatingBounds` вне всех экранов | `Apply` + `Open` | окно у главного окна, bounds обновлены (TW-7.4) |
| E15 | зарегистрирована панель во время работы, слот непуст | — | встала в конец слота, закрыта, если дескриптор не велит иное (TW-10.3) |

## 12. Вне рамок v1

- Вкладки-контенты внутри одной тул-панели (аналог ContentManager в IDEA: Run/Services с внутренними табами) — кандидат в v2, модель не должна ему противоречить.
- Widescreen-геометрия (TW-2.2), скрытие стрипов целиком, анимации выезда.
- Именованные раскладки с UI управления (ядро покрывает Snapshot/Apply — приложение может строить поверх).
- Переупорядочивание/перенос иконок перетаскиванием — фаза DnD (операции ядра те же).

## 13. Трассировка к эталону
