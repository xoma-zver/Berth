# Эталон: как устроена документная зона в IntelliJ IDEA

Настольный справочник по образцу (открытые исходники `intellij-community`, Apache 2.0):
устройство редакторских сплитов, окон документов и моста «панель ↔ редакторы» в IDEA;
дополняет [idea-tracing.md](idea-tracing.md). Не нормативен — поведение Berth определяет
`docs/spec/document-area.md`. Построчная трассировка каждого пункта спеки (вердикты с
цитатами `файл:строка`, включая арифметические проверки эквивалентности бинарного и
n-арного представлений), по которой принимались решения, — в git-истории этого файла
(версии до 2026-07).

## Модель соответствия терминов

| Термин Berth | В IDEA | Замечание |
|---|---|---|
| Вкладка | `VirtualFile` (документ) либо `Content` (панельная) | у IDEA две разные вселенные контента |
| Таб-группа | `EditorWindow` | |
| Сплит | `Splitter` | **строго бинарный**: ровно 2 ребёнка, одна доля, вторая = 1 − доля |
| Хост дерева | экземпляр `EditorsSplitters` | один класс для главного окна и каждого окна документов |
| Текущая вкладка | per-host `currentWindow` (свой `MyFocusWatcher` у каждого хоста) + живой резолв активного хоста из фокуса | у нас — per-host поле + явный `ActiveDockHost` |
| Адресация узлов | чистая identity объекта: ни id, ни `equals` | = наша адресация вкладками и путями (DA-1.3); счётчик-id есть только у самих окон документов, для bounds |
| Окно документов | `DockWindow` (extends `FrameWrapper`) | может независимо хостить свой `ToolWindowPane` — у нас такой способности нет |
| Дедупликация (INV-D2) | **не существует**: один файл легально открыт в N группах | наш ключ — аналог composite, не файла; дубль у нас невыразим |

## Устройство по подсистемам

### Дерево: бинарность без канонической формы

`Splitter` всегда о двух детях с одним `myProportion`; «сумма долей ≠ 1» у IDEA
невыразима структурно. Клемп `[0.1, 0.9]` зашит в сеттер **состояния** каждого узла (наш
INV-D3 разрешает весь интервал (0..1), клемп — на рендере). Канонической формы нет:
одноориентационная вложенность легальна и перманентна — аналога нашего N3 (вливание) не
существует, `removeFromSplitter()` такую вложенность рутинно создаёт и оставляет
навсегда. Схлопывание пустых групп и однодетных сплитов (наши N1+N2) выполняется одной
атомарной заменой в `removeFromSplitter()`. Наша n-арная модель воспроизводит видимый
результат бинарных операций IDEA после мысленного применения N3 — проверено
арифметически, включая точные доли (git-история этого файла, разборы DA-E13/E14).

### Операции

Открытие: вставка **после активной** вкладки (`openTabsAtTheEnd` дефолтно false);
уже открытый файл только активируется — в пределах окна (для отсоединённых окон IDEA
повторное открытие клонирует composite; у нас граница «активировать или клонировать» —
политика приложения через клон-`Id`, DA-2.4). Настоящий дубликат — только явный жест:
Split Right/Down (`closeSource=false`) или Ctrl/Cmd+drag. Закрытие: активной становится
**предыдущая соседка** (`computeIndexToSelect`; MRU и «правая соседка» — опции).
Единого примитива переноса нет — три места независимо реализуют «закрыть здесь + открыть
там» (`SplitAction`, `MoveEditorToOppositeTabGroupAction`, drag-delegate); наш атомарный
`MoveTab` — архитектурно чище при том же наблюдаемом эффекте, перенос у IDEA маркируется
флагом `CLOSING_TO_REOPEN` («перенос ≠ закрытие+открытие», DA-5.4). `split()` всегда
оборачивает лист новым бинарным сплитом c `proportion = 0.5` (наша «половина доли
донора»). `ChangeSplitOrientation` → `EditorWindow.changeOrientation()` — прямой аналог
нашего `RotateSplit`; у IDEA поворот всегда no-op-безопасен (канонической формы нет),
у нас он каскадит слияния N3 — цена INV-D1.

### Активность и фокус

`FileEditorManagerImpl` не хранит единого «текущего окна»: каждый `EditorsSplitters`
помнит своё через приватный `MyFocusWatcher`, реагирующий на сырой `FOCUS_GAINED`
(включая восстановление фокуса при Alt-Tab), а «какой хост активен» вычисляется живьём
из реального фокуса при каждом запросе (`getActiveSplitterSync`). Фокус в тул-панели не
резолвится ни в один хост редакторов — потому «текущая группа переживает фокус в панели»
(DA-6.1). Отсюда два наших решения: `ActiveDockHost` — то же живое значение, переведённое
в явное поле, и требование проводки DA-6.4 — **любое** реальное получение фокуса
контентом вкладки сводится к `ActivateTab`, не только клик по заголовку. Фолбэк после
исчезновения хоста у IDEA — MRU по `lastFocusGainedTime` поперёк хостов; наш DA-6.3 —
детерминированное упрощение (история фокуса — забота UI).

### Окна документов

`DockWindow` extends `FrameWrapper` — тот же предок, что у Window-режима панелей.
Создание — `createNewDockContainerFor`: drag-путь вычисляет bounds от точки сброса,
программный путь полагается на `suggestChildFrameBounds` (прямоугольник главного окна с
отступом, без каскада — наш дефолт «Move to New Window»). Bounds живут в
общеплатформенном `WindowStateService` (вписывание в экран → фолбэк к главному окну —
наблюдаемо наш TW-7.4/DA-7.4). Системное закрытие = `closeAll()` → настоящий `closeFile`
каждой вкладки (DA-7.3); «режима» для возврата у окна нет. Окно IDEA может независимо
хостить тул-окна (свой `ToolWindowPane`) и переживает пустое дерево вкладок, пока хостит
хоть одну панель — наш INV-D6 строже и безопасен лишь потому, что Berth такой
способности не даёт (раздел 11 спеки фиксирует цену включения). Закрытие проекта —
save-before-teardown: состояние сохраняется, пока окна ещё открыты (DA-7.6).

### Мост «панель ↔ док-зона»

Общего реестра владельцев у IDEA нет — владение выражено раздельностью моделей контента.
Но существует точечный живой мост: SPI `ToolWindowInEditorSupport`
(`canOpenInEditor`/`openInEditor`), единственная реализация — Terminal. Живая сессия
**не пересоздаётся**: `detachTab` → обёртка `TerminalViewVirtualFile` → штатный
`openFile`; обратно — `TerminalDockContainer.attachTab`, приём типизирован строго под
терминальный контент. Это прецедент, поднявший ячейку «панель → док-зона: да» таблицы
DA-8.1 с «нашей выдумки» до «≈ эталона»; ячейки «нет» у IDEA обеспечены раздельностью
моделей, у нас — проверкой владельца (canHost, INV-D5).

### Персистентность

Тот же XML-механизм, что у тул-окон; числового SchemaVersion нет (`IdeFingerprint` +
толерантный парсинг). Исчезнувший владелец контента: живой layout **не удаляет** вкладку —
сперва пробует актуальный список провайдеров, при неудаче оставляет вкладку видимой, но
пустой (в исходниках IDEA у этого места до сих пор `TODO`); MRU-история вычищает только
мёртвую часть записи. Оба прецедента мягче удаления — основание наших «спящих вкладок»
(DA-9.4). Контрпример устойчивости: один нечисловой атрибут доли роняет загрузку **всего**
layout редакторов (`try/catch` только на уровне компонента) — наши N1–N5 с поэлементной
починкой и отчётом надёжнее по построению.

## Сознательные расхождения Berth ↔ IDEA

1. **N-арное дерево с канонической формой** (INV-D1, N3) против перманентной бинарной
   вложенности — главное решение ADR-0005; видимые результаты эквивалентны.
2. **Уникальность `Id` по всей раскладке** (INV-D2): у IDEA дубль — не дефект, а жест;
   наш ключ соответствует composite, второй вид — клон-`Id` приложения (DA-2.4).
3. **INV-D3 разрешает состоянию весь (0..1)**, клемп — на рендере; у IDEA жёсткий
   `[0.1, 0.9]` в состоянии каждого узла (для n-арного узла фиксированный пол
   несостоятелен: k детей требуют k·пол ≤ 1).
4. **`ActiveDockHost` — явное поле** вместо живого резолва из фокуса; плата — требование
   проводки DA-6.4 (любой focus-gain = `ActivateTab`).
5. **Детерминированные фолбэки активности** (DA-5.2, DA-6.3) вместо MRU по истории
   фокуса; MRU — забота UI/приложения.
6. **Атомарный `MoveTab`** вместо трёх независимых реализаций close-then-open.
7. **Окна документов — чистые деревья вкладок**: хостинг панелей в них вне рамок v1,
   потому INV-D6 жёсткий («окно живо ⇔ есть вкладка»).
8. **«Спящие вкладки» с round-trip** (DA-9.4) — формализация обоих мягких прецедентов
   IDEA вместо «видимой, но пустой» вкладки.
9. **Явная ошибка загрузки + отчёт починки** (DA-9.5, TW-10.5) против «никогда не падать
   на workspace».
10. **Чистая отмена вкладочного drag** (DA-E22): эталонная отмена у вкладок редактора
    оставляет следы.
