# Трассировка спеки document-area к эталону (IntelliJ IDEA)

Отчёт: как **каждый** нормативный пункт `docs/spec/document-area.md` реализован в открытых
исходниках IntelliJ Platform (`intellij-community-master`, Apache 2.0). Дополняет
`docs/reference/idea-tracing.md` (та же трассировка для `tool-windows.md`) — раздел 9
tool-windows.md переиспользуется document-area.md почти целиком, поэтому там, где эта спека
ссылается на `TW-9.x`, здесь стоит короткая пометка «см. idea-tracing.md», а не повторная
трассировка с нуля.

*Отчёт готовился по спеке **v0.1**; правки v0.2–v0.3 (per-host текущая вкладка и `ActiveDockHost`,
спящие вкладки, `RotateSplit`, переформулировка DA-2.4) сделаны по его итогам — отмеченные ниже
расхождения по DA-2.5/DA-6.1/DA-9.4 и «пробелы» (поворот сплита, undo-стек) в актуальной спеке уже
адресованы. Разборы механизмов IDEA остаются точными.*

## Как читать

- **Вердикт**: `=` поведение совпадает с IDEA; `≈` наше упрощение/обобщение при том же наблюдаемом
  эффекте; `≠` **расхождение** (IDEA делает иначе — важно для ревью спеки); `—` вне модели IDEA
  (наша конвенция, эталона нет).
- Ссылки вида `Файл.kt:NNN` — на исходники под `platform/`/`plugins/`, если не указано иное; пути в
  тексте сокращены до basename там, где файл один раз назван в разделе — полные пути см. таблицу
  ниже или заголовок раздела.
- Пути к файлам в этом отчёте относительны корня `intellij-community-master`.
- Все выводы — из чтения исходников (нет запущенной IDE в среде подготовки отчёта), но каждая
  цитата проверена: файл открыт и строки перечитаны, а не угаданы по имени метода.

## Модель соответствия (самое важное)

| Термин спеки | В IDEA | Замечание |
|---|---|---|
| **Вкладка (tab)** | `VirtualFile` (документ) либо `Content` (панельная вкладка, см. TW-9.7 в idea-tracing.md) | две разные вселенные; см. DA-8.1 про единственный мост между ними |
| **Таб-группа (Group)** | `EditorWindow` | `tabbedPane: TabInfo`-список + `selectedInfo`/`currentCompositeFlow` |
| **Сплит** | `Splitter` (`platform/util/ui`) | **строго бинарный** — ровно 2 ребёнка, доля — **одно** `myProportion`, доля второго = `1 − proportion` всегда, никогда отдельно не хранится |
| **Дерево** | рекурсивная вложенность `Splitter`/`EditorWindow` | без ограничения на «ребёнок не той же ориентации» — см. INV-D1 |
| **Хост дерева** (док-зона / окно документов) | экземпляр `EditorsSplitters` | `mainSplitters` — один на приложение; ещё один на каждый открытый `DockableEditorTabbedContainer` (окно документов) — **тот же класс**, не параллельная реализация |
| **`CurrentTabId`** | **не единое поле.** Каждый `EditorsSplitters` хранит свой `currentWindow: StateFlow<EditorWindow?>` (приватный `MyFocusWatcher`); «какой хост активен» вычисляется **живьём** из реального AWT-фокуса при каждом запросе (`getActiveSplitterSync`) | структурно двухуровневая величина — см. DA-6.1, это самая важная находка всей трассировки |
| **Адресация узлов** (DA-1.3) | чистая identity объекта — ни у `Splitter`, ни у `EditorWindow` нет id/`equals`/`hashCode` нигде | **но** само окно документов (не узлы внутри) получает мелкий персистентный числовой id (`DockManagerImpl.windowIdCounter`) — только для восстановления bounds/tool-pane после рестарта, не для адресации дерева |
| **Окно документов** | `DockWindow` (`extends FrameWrapper`) | тот же базовый класс, что и `WindowedDecorator` тул-окон (TW-7.2); создаётся `DockManagerImpl.createNewDockContainerFor`; может **независимо** хостить свой `ToolWindowPane` — возможность, которой в нашей модели нет вовсе |
| **N-арность vs бинарность** | `Splitter` всегда ровно 2 ребёнка | наша n-арная вставка «вдоль оси» воспроизводит тот же видимый эффект только после мысленного применения N3 к бинарному дереву IDEA — арифметически проверено точно для DA-E13/DA-E14 |
| **Дедупликация (INV-D2)** | **не существует вообще** — ни живьём (один файл легально открыт в N группах по дизайну), ни при восстановлении (`UiBuilder` строит каждый лист независимо, без проверки пересечений) | DA-E9 → `—`, не `≈`: чинить нечего, потому что нет самого понятия дефекта |
| **`canHost` (INV-D5)** | нет общего реестра владельцев; **но** есть один настоящий, живой, двунаправленный opt-in мост — `ToolWindowInEditorSupport`, реализован только Terminal-плагином | поднимает вердикт DA-8.1 (направление панель→докзона) с `—` до `≈` |
| **Формат персистентности** | тот же механизм, что у тул-окон: XML/JDOM в `PRODUCT_WORKSPACE_FILE` | нет явного `SchemaVersion` — версионирование полагается на `IdeFingerprint` + толерантный JDOM-парсинг; MRU-история — **отдельный** `PersistentStateComponent` со своим циклом жизни записей |

**Топ находок**, которые стоит держать в поле зрения при ревью спеки (детали — в соответствующих
пунктах ниже):

1. **`CurrentTabId` — двухуровневая величина у IDEA, наша спека делает её плоской (DA-6.1).**
   Каждый хост (главное окно, каждое окно документов) независимо помнит своё «текущее окно» через
   собственный `MyFocusWatcher`; какой хост «активен» — вычисляется живьём из реального фокуса, а
   не хранится. Плоское поле `CurrentTabId` воспроизводит это **только если** UI-слой шлёт
   `ActivateTab`-эквивалент на **любое** реальное получение фокуса редактором в любом хосте, а не
   только на явный клик по вкладке — это сейчас незафиксированное требование к проводке UI.
2. **INV-D3 — у IDEA жёсткий `[0.1, 0.9]` клемп на каждом сплите на уровне *состояния***
   (`Splitter.setProportion`), а не только на этапе рендера. Наша спека разрешает состоянию
   `(0,1)` целиком, откладывая клемп в UI (DA-2.2) — мы строго permissive-нее, чем реальная IDEA.
3. **INV-D6/DA-7.1/DA-7.2 — окно документов у IDEA может независимо хостить тул-окна** (свой
   `ToolWindowPane`, `DockWindow.kt:63,112-157`) и переживает опустошение дерева вкладок, пока там
   есть хотя бы одно тул-окно (`closeIfEmpty()`). Наш инвариант «окно живо ⇔ дерево непусто» строже
   той модели, которую он описывает — безопасно только потому, что Berth вообще не даёт окнам
   документов хостить панели.
4. **DA-8.1 — существует настоящий, живой (session-preserving), двунаправленный, DnD-подключённый
   мост панель↔докзона** (`ToolWindowInEditorSupport`, единственная реализация — Terminal). Ячейка
   «панель → док-зона: да» — не изобретение без прецедента; направление подтверждено конкретным
   работающим кодом.
5. **DA-9.4 — оба реальных прецедента IDEA мягче нашего «удалить с отчётом».** Живой layout
   пытается переоткрыть исчезнувший провайдер актуальным списком раньше, чем сдаться (и даже тогда
   оставляет вкладку видимой, но пустой — сами разработчики IDEA пометили это `TODO`); MRU-история
   хирургически чистит только мёртвую часть записи. Ни один механизм не удаляет вкладку/запись
   целиком.
6. **DA-E9/INV-D2 — дедупликации при восстановлении у IDEA нет вообще**, потому что дублей не
   существует как концепции. Правильный вердикт — `—`, не `≈` (нечего заимствовать или сверять).
7. **DA-2.4 сформулирована шире, чем факт.** Дефолт открытия («уже открыт где-то → активация, без
   переноса») у IDEA и Berth совпадает **дословно**. Расходится не дефолт, а сама *возможность*
   дубликата: у IDEA это два явных, редких, специально запрашиваемых жеста (Split Right/Down,
   Ctrl/Cmd+drag-out), у нас — структурно невозможно (INV-D2).
8. **N3 (вливание) у IDEA нет вообще** — бинарность делает вложенность перманентной, ничего никогда
   не сливает. Проверено арифметически на DA-E13/DA-E14: наша n-арная модель после нормализации
   воспроизводит тот же видимый эффект и те же доли, что и оба возможных бинарных представления
   IDEA одного и того же ряда групп.
9. **DA-5.4/`MoveTab` — у IDEA нет единого атомарного примитива переноса.** Три места
   (`SplitAction`, `MoveEditorToOppositeTabGroupAction`, drag-delegate) заново реализуют
   «закрыть-здесь + открыть-там» с разной механикой. Наш единый `MoveTab` (ADR-0004) — архитектурно
   чище, а не расхождение в наблюдаемом эффекте.
10. **DA-E13 (новая, персистентность) — единственный нечисловой атрибут доли роняет загрузку ВСЕГО
    layout редакторов** целиком (нет `try/catch` на уровне узла, только на уровне всего компонента,
    `ComponentStoreImpl.kt:238-246`) — контраст в пользу нашего дизайна: N1–N5 чинят дефекты
    поэлементно с отчётом, а не падают целиком.

## Раздел 1. Термины и хосты деревьев

**DA-1.1 — единая модель дерева для всех хостов дока-зоны.** `=`

Главная зона главного окна и плавающее/окно-документов — буквально **один и тот же** класс
`EditorsSplitters`, не параллельные реализации. Плавающий контейнер строится как анонимный
подкласс, переопределяющий только `isFloating` и два колбэка жизненного цикла:
`createEditorDockContainer` (`DockableEditorContainerFactory.kt:42-74`). `EditorWindow` (группа) и
`Splitter` (узел сплита) переиспользуются между хостами без изменений. Разница в жизненном цикле
совпадает с DA-1.1 дословно: плавающие контейнеры создаются с `disposeWhenEmpty = true`
(`DockableEditorContainerFactory.kt:72`), тогда как `mainSplitters` создаётся напрямую
`FileEditorManagerImpl` через голый `createCurrentWindow()` (`EditorsSplitters.kt:863-869`) без
такого флага и всегда оставляет минимум пустой корень.

**DA-1.2 — ориентация: `Row` = слева направо, `Column` = сверху вниз; `Left`/`Right`→`Row`,
`Up`/`Down`→`Column`.** `=`

`Splitter.isVertical()` однозначен: `true` — компоненты друг над другом (наш `Column`), `false` —
бок о бок (наш `Row`) (`Splitter.java:37-40, 617-632`). Направление↔ориентация подтверждено
независимо через `SplitterService.split()`: `UP`/`DOWN` → `JSplitPane.VERTICAL_SPLIT` (`Column`),
всё остальное → `HORIZONTAL_SPLIT` (`Row`) (`SplitterService.kt:98-104`), и через привязку
шорткатов в `PlatformActions.xml:855-865` (`TW.SplitRight` разделяет шорткат с `SplitVertically`,
`TW.SplitDown` — с `SplitHorizontally`) — те же самые (запутанно названные по линии разделителя, а
не по раскладке) команды.

**DA-1.3 — адресация без id (⚠, открытый вопрос владельцу).** `≈` (спека уже сама помечает так)

Подтверждено по всем пунктам, с одним уточнением:

- **Сплит: id нет никогда.** `Splitter.java` не имеет ни id/uuid-поля, ни персистентной идентичности.
  Позиция в XML-дереве — это и есть адрес, ровно наша идея «хост + путь индексов», просто выраженная
  вложенностью XML, а не явным вектором.
- **Группа (`EditorWindow`): id нет никогда.** Ни `equals`/`hashCode` не переопределены нигде в
  файле на 1358 строк — идентичность голая, по ссылке JVM. `isValid` буквально
  `owner.containsWindow(this)` (`EditorWindow.kt:175-177`) — «этот же объект всё ещё в множестве
  хоста», а не поиск по ключу.
- **In-memory идентичность — только на время сессии, не переживает `readExternal`.** При
  восстановлении `UiBuilder.process` строит для каждого листа **новый** `EditorWindow`
  (`EditorsSplitters.kt:1286-1293`) — попытки сопоставить восстановленный лист с «тем же самым»
  довосстановочным окном нет вовсе.
- **Уточнение к формулировке спеки: само окно документов (не узлы внутри него) получает
  персистентный строковый id.** `DockManagerImpl` присваивает
  `val windowId = id ?: (windowIdCounter++).toString()` (`DockManagerImpl.kt:486-487`, счётчик —
  `:89`) и сохраняет как XML-атрибут (`:526`), читает обратно через `getOrCreateWindowFor(id, ...)`
  (`:506`) — чтобы переприсоединить сохранённые bounds/tool-pane состояние конкретного окна после
  рестарта. Тривиальный инкрементный счётчик, не семантическая идентичность, но реальный и
  переживающий рестарт — ближайшая вещь к персистентному id во всей этой области. Формулировка
  спеки «нет id» технически верна для узлов дерева, но не учитывает эту разницу
  окно-vs-узел. **Для решения владельца:** это не меняет расчёт по DA-1.3 как таковой (адресация
  узлов id-ами не нужна, IDEA сама обходится без них) — но если Berth когда-либо захочет минимальный
  persistent-id именно для окна (не для узлов внутри), у IDEA есть рабочий прецедент именно такого,
  предельно лёгкого масштаба.

## Раздел 2. Модель дерева

**DA-2.1 — узлы: сплит (ориентация + ≥2 упорядоченных детей + доли), группа (упорядоченные Id +
активная).** `≈`

Поля отображаются один в один с оговоркой о форме (раздел 12 спеки уже фиксирует бинарность как
осознанное расхождение). Сплит: ориентация — `myVerticalSplit`; ровно 2 упорядоченных ребёнка —
`myFirstComponent`/`mySecondComponent` (`Splitter.java:54-55`, **всегда** ровно 2, никогда «≥2»);
доля — **одно** `myProportion`, доля второго ребёнка — `1 − proportion` по построению, никогда не
хранится отдельно. Группа: `EditorWindow.tabbedPane` держит упорядоченный список `TabInfo`
(`:137`), каждый оборачивает `VirtualFile`; активная — `tabbedPane.editorTabs.selectedInfo` /
`currentCompositeFlow` (`:184-185, 224-227`). Отдельное структурное следствие бинарности: раз у
сплита всегда ровно 2 ребёнка с одной производной долей, у него никогда не возникает нужды в
*векторе* долей — «сумма N долей равна 1» это свойство, которое представление IDEA физически не
может даже сформулировать, не то что нарушить (см. N4/INV-D3 ниже).

**DA-2.2 — доли не знают о пикселях; минимум клемпится на рендере, не в состоянии.** `=`

Разделение реальное, не только концептуальное. `Splitter.setProportion` (`:592-607`) клемпит только
к **фиксированному** диапазону `[myMinProp, myMaxProp]` (см. INV-D3 ниже) и никогда не смотрит на
пиксельный размер компонентов. Учёт минимального пиксельного размера — исключительно в
`computeFirstComponentSize`/`getMinProportion` (`:505-539, 699-719`), вызываются только из
`doLayout` (`:423-503`) — вычисление на этапе рендера, результат используется только для выставления
bounds компонентов и никогда не пишется обратно в `myProportion`. Состояние и рендер по-настоящему
разделены, ровно `[клемп — UI]`. (Оговорка: `reshape()` *может* писать обратно в `myProportion` при
нестандартной `DividerPositionStrategy` — `:294-326`, но редакторские сплиты создаются через
`createSplitter` без смены стратегии от дефолтной `KEEP_PROPORTION` (`:69`), так что для документной
зоны этот путь не активен.)

**DA-2.3 — только корневая группа может быть пустой (главное окно, панель); остальные пустые
устраняет N1.** `=`

`createCurrentWindow()` создаёт голый `EditorWindow` с нулём вкладок прямо как корень
(`EditorsSplitters.kt:863-869`) — пустой корень достижим и легален, ровно «минимум — пустая
корневая группа». Непустотность вне корня устраняется мгновенно: `closeFile` вызывает
`removeFromSplitter()` ровно при `disposeIfNeeded && tabCount == 0` (`EditorWindow.kt:735-738`),
и `removeFromSplitter()` (`:791-823`) безусловно продвигает соседа на место всего 2-детного
сплита — пути, оставляющего бездетный нерут-`EditorWindow` в дереве, нет. `restoreEditors()`
дополнительно подчищает оставшиеся пустые окна после загрузки состояния (`:452-462`) — защита даже
от испорченной сохранёнки.

**DA-2.4 — не более одного места для Id; операции переносят, а не копируют.** `≠` (сам инвариант,
уже принято) — но формулировка расходится **шире**, чем факт

Инвариант — подтверждённое, осознанное расхождение (таблица раздела 12 спеки его уже фиксирует). Но
разбор по существу показывает, что реальное поведение IDEA ýже, чем «≠ IDEA: там умеет держать файл
в нескольких группах» читается на первый взгляд:

**(a) Дефолт открытия уже открытого где-то файла — активация, без дубликата, и это совпадает с IDEA
дословно.** Цепочка из четырёх слоёв: обычная навигация идёт через `openEditorImpl`
(`FileEditorManagerImpl.kt:1585-1608`), где `reuseOpen = !isUseCurrentWindow`
(`OpenFileDescriptor.isUseCurrentWindow` дефолт `false` → `reuseOpen=true`, `:1602`); `getWindowToOpen`
(`:1144-1158`) с `forceUseActiveSplitter=false` по умолчанию (advanced setting
`editor.open.inactive.splitter` дефолт `true`, `PlatformExtensions.xml:1768`) уходит в
`findWindowInAllSplitters(file)` (`:1151`), которая обходит **все** сплиттеры и **все** окна
документов; `doOpenInEdt` (`:1419-1468`) вычисляет `isNewEditor = (window.getComposite(file) ==
null)` — если композит уже есть, `EditorWindow.addComposite` (`:352-446`) целиком пропускает блок
вставки таба и просто выделяет существующую вкладку. Второй независимый предохранитель на этом же
уровне — `EditorTabbedContainer.insertTab` сам возвращает существующий `TabInfo`, если для файла в
**этом** окне таб уже есть (`:262-264`), до всякой попытки вставки.

*Уточнение (2026-07, наблюдение владельца): для файла, открытого в **отсоединённом окне**,
повторное открытие в главном создаёт новую вкладку (новый composite) вместо активации
существующей — вывод (a) подтверждён наблюдением только в пределах одного окна. Цепочка
`findWindowInAllSplitters` в коде прочитана, но её достижимость для хостов
`DockableEditorTabbedContainer` наблюдением не подтверждается — перепроверить достижимость, не
только наличие. На спеку не влияет: граница «активировать или клонировать» — политика приложения
(DA-2.4); ядро Berth по совпавшему `Id` всегда активирует (DA-E33).*

**(b) Что именно создаёт настоящий дубликат — два независимых, явных, не-дефолтных механизма:**

1. **`Split Right`/`Split Down`** (табовое меню) — `SplitVerticallyAction`/`SplitHorizontallyAction`,
   тонкие подклассы `SplitAction` с `closeSource=false` по умолчанию (`SplitAction.java:26-28`).
   `actionPerformed` (`:36-49`) при `closeSource==false` файл **не закрывает** в источнике, сразу
   зовёт `window.split(...)`, которая открывает тот же файл в соседнем окне
   (`openFileInNewCompositeInEdt`, `EditorWindow.kt:573-585`) — настоящее «Split and Duplicate».
   Симметрично — `OpenEditorInOppositeTabGroupAction` (`closeSource=false`).
2. **Ctrl/Cmd + вытягивание вкладки за пределы контейнера.**
   `EditorTabbedContainerDragOutDelegate.dragOutFinished` (`EditorTabbedContainer.kt:505-521`):
   `val copy = UIUtil.isControlKeyDown(event) || ...ACCEPT_COPY` (`:506`) — модификатор-клавиша (на
   Mac `⌘`, иначе `Ctrl`). Без модификатора (дефолт при drag) — `copy=false`, источник закрывается
   (`:513`) — настоящий перенос, ровно DA-5.5 Move. `ACCEPT_COPY` как альтернативный триггер
   объявлен в контракте (`DockContainer.java:19-25`), но ни одна реализация `getContentResponse` во
   всём репозитории его не возвращает (единственные — `DockableEditorTabbedContainer.kt:125-128`,
   отдаёт только `ACCEPT_MOVE`/`DENY`) — на практике единственный живой путь к дублю через DnD —
   модификатор.

**Рекомендация:** в прозе DA-2.4 стоит явно разделить «дефолт» (не расходится, совпадает с DA-5.1
дословно) от «возможности вообще» (расходится — у IDEA явный редкий жест, у нас структурно
невозможно, INV-D2). Смежное наблюдение для DA-5.5: команды с `closeSource=true`
(`MoveTabRightAction`/`MoveTabDownAction`) требуют `window.getTabCount() >= 2`
(`SplitAction.java:72-73`) — «Move» единственной вкладки окна из выделенного действия
**задизейблено**, в отличие от нашего `SplitTab`, который для этого случая полагается на
нормализацию (DA-E1, N1–N2); для «Split»/дубликата у IDEA такого ограничения нет.

**DA-2.5 — состояние: дерево главного окна + упорядоченный список окон документов (Bounds+дерево) +
`CurrentTabId`.** `≈`, с двумя конкретными уточнениями

- **«Упорядоченный список (порядок создания)» не выполняется для реестра окон IDEA.**
  `DockManagerImpl` держит контейнеры в `private val containers = HashSet<DockContainer>()`
  (`:81`) — обычное неупорядоченное множество. Небольшое, ранее не отмеченное расхождение: ничто в
  каноническом реестре IDEA не гарантирует и не сохраняет порядок создания окон документов, который
  требует DA-2.5.
- **`CurrentTabId` и `currentWindow` — взаимно-обратные производные одной информации.** Наша спека
  делает первичной *вкладку* (`CurrentTabId`), группа выводится из неё. IDEA — наоборот:
  `currentWindow` (`MutableStateFlow<EditorWindow?>`, `EditorsSplitters.kt:199-211`) первичен, а
  `currentCompositeFlow` **выводится** из него через `flatMapLatest` в собственный поток этого окна
  (`:332-348`). То же содержание информации, противоположное направление вывода — см. полный разбор
  структурной двухуровневости в DA-6.1.

## Раздел 3. Нормализация

**N1 (пустые группы удаляются, корень остаётся).** `≈`

Существует, но у IDEA никогда не бывает *изолированным* шагом — слит с N2 в одну атомарную замену
(см. N2 и DA-E8). Нет пути, на котором бездетный нерут-`EditorWindow` наблюдаем даже временно в
рамках одного вызова: проверка `closeFile`'а (`tabCount==0`) и `removeFromSplitter()` идут подряд в
одном и том же методе.

**N2 (сплит с одним ребёнком заменяется этим ребёнком, в т.ч. в корне).** `≈`

Присутствует напрямую и, поскольку сплиттеры IDEA всегда ровно 2-детные, это **единственный**
способ, которым узел-сплит вообще исчезает (нет случая «3→2», который надо было бы отличать от
«2→1»). `removeFromSplitter()` обрабатывает и внутренний случай (родитель — другой `Splitter`:
2-детный сплит заменяется выжившим соседом в дедушке, `EditorWindow.kt:803-812`), и корневой
(родитель — сама `EditorsSplitters`: сосед ставится новым голым корнем напрямую, `:813-819`) —
включая корневой сплит, схлопнутый повторными закрытиями до одного ребёнка (это ровно сценарий
DA-E4 на уровень выше по дереву).

**N3 (вливание одноориентационного дочернего сплита).** `≠`

Аналога нет вовсе, и это прямое следствие строгой бинарности, а не упущенная фича: `Splitter`, чей
ребёнок — сам `Splitter` той же ориентации, — совершенно нормальное, постоянное, сколь угодно
глубоко вложенное состояние IDEA; ничто его никогда не сливает. Продвижение в
`removeFromSplitter()` (см. N2) **активно создаёт** такую вложенность как побочный эффект (точный
разбор — DA-E8) и оставляет её там навсегда. Единственный поверхностно похожий механизм,
`normalizeProportionsIfNeed` (`EditorWindow.kt:603-636`, вызывается после каждого `split()` и после
каждого продвижения `removeFromSplitter()`), — совсем другая операция: спрятана за advanced-настройкой
`editor.normalize.splits`, дефолт `false` (`PlatformExtensions.xml:1776`), и даже включённая **не**
сохраняет отрендеренные пропорции (в отличие от N3, который сохраняет доли через `gᵢ → s·gᵢ`) — она
их активно *меняет*, продвигаясь вверх по цепочке одноориентационных предков и приводя каждый
уровень к `1/(2+i)` ради визуального баланса. IDEA не поставляется ни со сливающим шагом, ни с
включённой по умолчанию ребалансировкой — вложенные одноориентационные сплиттеры это просто то, как
выглядит дерево после нескольких закрытий, навсегда.

**N4 (невалидные доли → равные, затем перенормировка к сумме 1).** `≈`

Невалидные состояния, от которых защищает N4, у IDEA по большей части **невыразимы**, а не просто
запрещены: поскольку каждый `Splitter` хранит одно число и производит долю второго ребёнка как
`1 - proportion` (`Splitter.java:505-507`, `doLayout` то же самое на `:454`), «доли детей не
суммируются в 1» просто не может возникнуть — перенормировывать нечего. То, что IDEA реально
защищает — другое, более узкое: `setProportion` клемпит **каждую** запись к фиксированной паре
`[myMinProp, myMaxProp]` (`:601`), для каждого редакторского сплиттера это жёстко `[0.1, 0.9]`
(`EditorWindow.kt:550`, `EditorsSplitters.kt:1245-1246`, оба места создания), плюс `LOG.warn` при
входе вне `[0,1]` вообще (`:598-600`). При восстановлении битый/отсутствующий `split-proportion`
дефолтится в `0.5f` (`EditorsSplitters.kt:1153-1154`) и проходит через тот же клемп-конструктор.
Итог: перенормировки суммы не нужно (структурно гарантировано), но есть постоянно включённый клемп
диапазона на уровне состояния, которого у нашего INV-D3 нет — см. INV-D3 ниже, это реальное, ранее
не отмеченное расхождение.

**N5 (невалидная активная вкладка → первая; аналогично `CurrentTabId`).** `=`

Для живого состояния правило структурно не нужно по той же причине, что и N4: `selectedInfo` —
прямая ссылка внутрь той же коллекции `TabInfo`, откуда выбирает, «выбрана вкладка не из группы»
возникнуть не может. Но при *восстановлении* IDEA приходится выбирать selection из
сериализованных данных, и там фолбэк — точное совпадение: `val tabToSelect = openedFiles.firstOrNull
{ it.isCurrentTab }?.tab`, затем `window.selectTabOnStartup(tab = tabToSelect ?: tabs.first(), ...)`
(`FileEditorManagerImpl.kt:2409-2419`) — буквально «нет (или невалидна) отмеченная текущая — берём
первую вкладку группы».

**DA-3.2 — нормализация идемпотентна; канонические деревья проходят её без изменений.** `≈`

Подтверждено по духу, но не проверяемо как изолированная чистая функция так, как это формулирует
спека — у IDEA нет отдельной точки входа `normalize(tree)`, вызываемой дважды. Гарантия
поддерживается инкрементально: каждая мутирующая операция оставляет дерево в состоянии, которое
последующие операции переносят без спецкейсов, а `restoreEditors()` делает один явный проход после
загрузки специально для чистки пустых окон из недоверенного сохранённого состояния (`:452-462`) —
забота о round-trip-безопасности (ср. TW-10.4) у IDEA тоже есть, просто реализована как «почистить
один раз после загрузки», а не «прогнать fixed-point-правила после каждой операции». Раз у N3 нет
аналога вообще (выше), «каноническое» в смысле нашего INV-D1 — не то, что операции IDEA сохраняют
или должны сохранять.

## Раздел 4. Инварианты

**INV-D1 — деревья канонические (нет пустых нерут-групп, нет сплитов <2 детей, ребёнок сплита не
повторяет ориентацию родителя).** `≠`

Первые два конъюнкта выполняются (совпадает с N1/N2). Третий — «нет одноориентационной
вложенности» — **не** выполняется и вообще не отслеживается IDEA как инвариант. Как показано в N3 и
DA-E8, продвижение `removeFromSplitter()` рутинно производит именно такую форму
(`Splitter(Row){ first=A, second=Splitter(Row){ first=B, second=C } }`), и ничто в кодовой базе её
никогда не схлопывает. Это самое острое из уже принятых следствий n-арности-vs-бинарности: наш
INV-D1 — настоящий, проверяемый структурный инвариант канонической формы; у бинарного дерева IDEA
нет эквивалентного понятия «канонической формы» вообще — любая форма/глубина, возникшая из
какой-то истории сплитов-и-закрытий, легальна навсегда.

**INV-D2 — каждый Id встречается не более одного раза суммарно по всем деревьям.** `≠` (уже принято)
— вердикт для *механизма проверки* конкретно: `—`

Сам инвариант — осознанное, уже зафиксированное расхождение (раздел 12 спеки). Важное уточнение из
DA-E9: у IDEA нет **никакого** механизма проверки/починки дублей ни живьём (дубли легальны по
дизайну, см. DA-2.4), ни при восстановлении — `UiBuilder.process`/`processFiles`
(`EditorsSplitters.kt:1209-1354`) строит дерево рекурсивно, каждый лист получает свой `EditorWindow`
из своего собственного списка файлов (`:1213-1238`), без проверки пересечений на каком-либо уровне.
Единственное похожее место — `EditorsSplitters.openFileList` (`:275-281`, `.distinct()`) — это
**read-only** список для рассылки (напр. обновления иконок), не воздействует на дерево и не
затрагивает вкладки. Правило DA-9.2 «дубликат → первое вхождение, отчёт» — целиком следствие
ADR-0005 и нашего собственного запрета дублей, не заимствовано и не проверяемо эталоном.

**INV-D3 — доли каждого сплита ∈ (0,1) и суммируются в 1.** `≠`

Половина «сумма = 1» структурно гарантирована (см. N4 — нечего нарушить). Открытый интервал —
именно там, где расхождение конкретно: наша спека разрешает любое значение из **открытого**
интервала `(0,1)`, с клемпом, явно отложенным в UI/рендер (DA-2.2, `[клемп — UI]`) — то есть само
состояние может быть сколь угодно близко к 0 или 1. IDEA вместо этого зашивает **фиксированный
10%/90% пол-и-потолок прямо в сеттер состояния**: `Splitter.setProportion` безусловно клемпит к
`[myMinProp, myMaxProp]` (`:601`), и каждый редакторский сплиттер создан с `minProp=0.1f,
maxProp=0.9f` (`EditorWindow.kt:550`, `EditorsSplitters.kt:1245-1246`) — это не клемп минимального
размера по контенту (это отдельный, рендер-only механизм `getMinProportion`, см. DA-2.2), а
контент-независимый, всегда включённый 10%-й пол в **сохранённой** доле. Пользователь физически не
может утащить разделитель за 10%/90%, сколько бы места ни было — живой обработчик драга,
`OnePixelDivider.processMouseMotionEvent`, зовёт `mySplitter.setProportion(proportion)` на каждом
`MOUSE_DRAGGED` (`:252-271`), так что пол упирается постоянно в обычном использовании, а не только
как защитный крайний случай. **Реальное, конкретное, ранее не отмеченное расхождение**, достойное
внимания владельца: наш INV-D3 — строгое надмножество того, что реально допускает состояние IDEA.

**INV-D4 — активная вкладка группы принадлежит группе; `CurrentTabId` валиден или null.** `≈`

Половина «активная вкладка принадлежит группе» покрыта N5 выше (`=`). Половина про `CurrentTabId` —
см. DA-6.1: у IDEA нет хранимого `CurrentTabId`-эквивалента, который мог бы «стать невалидным» в
нашем смысле — он вычисляется живьём из фокуса при каждом запросе (`getActiveSplitterSync`), так
что вопрос «валиден ли сохранённый указатель» для IDEA просто не возникает так, как формулирует
инвариант. Наблюдаемо эквивалентно (после любой мутации следующий запрос «текущей вкладки» у IDEA
никогда не укажет в никуда), но достигнуто отсутствием хранимого состояния, а не проверкой его
валидности.

**INV-D5 — допуск вкладок (`canHost`).** `≈` (для «панель P → хосты док-зоны») / `—` (для остальных
ячеек)

Общего реестра `canHost`, проверяющего владельца по id внутри одного дерева, у IDEA нет — уже
установлено в TW-9.7 (idea-tracing.md): контент тул-окна живёт в `ContentManager`, контент
редакторов — в `FileEditorManager`, модели не смешаны в общем случае. **Но** прямая проверка
terminal-плагина показала, что для направления «панель → док-зона» у IDEA есть настоящий,
двунаправленный, DnD-интегрированный мост — не общий реестр, а opt-in SPI на конкретном тул-окне.
Полный разбор — DA-8.1 ниже. Ячейки «нет» (документ → панель, чужая панель Q → панель P) остаются
`—`: обратный мост typed строго под terminal-контент и в принципе не примет произвольный
документ; общего механизма «любой тул-window принимает контент другого/редактора» не существует.

**INV-D6 — окно документов существует ⇔ дерево содержит хотя бы одну вкладку.** `≈`

Прямое направление (вкладки есть ⇒ окно существует) тривиально верно. Обратное — там, где реальный
инвариант IDEA **строго слабее** заявленного: `DockWindow.closeIfEmpty()` (`DockWindow.kt:176-181`)
закрывает окно только при

```
container.isEmpty && (toolWindowPane == null || !toolWindowPane!!.buttonManager.hasButtons())
```

— т.е. **пустое дерево вкладок необходимо, но не достаточно**: окно документов, для которого хоть
раз вызывался `setupToolWindowPane()` (любое окно, созданное не с `isSingletonEditorInWindow=true`)
и всё ещё хостящее задокированное/пришпиленное тул-окно, переживает опустошение таб-дерева
бесконечно. `DockWindow` имеет поле `toolWindowPane: ToolWindowPane?` (`:63`) и метод
`setupToolWindowPane()` (`:112-157`), который строит полноценный `ToolWindowPane`, регистрирует его
в `ToolWindowManagerImpl.addToolWindowPane` (`:138-139`) и перевешивает редакторский сплиттер
**внутрь** него (`:141-144`) — тул-окна можно перетащить в стрипы отдельного окна документов точно
так же, как в главное. Комментарий в `DockManagerImpl.readStateFor` подтверждает, что это
осознанная живая функциональность: *«a floating editor container with a docked tool window will
remain open»* (`:562-566`). Спека не моделирует этот случай не как ошибочное решение, а как
отсутствующую возможность: раздел 7 не даёт окнам документов независимого хостинга тул-окон вовсе,
так что сценарий, нарушающий строгий INV-D6, у Berth просто не может возникнуть **сегодня**. Если
Berth когда-нибудь даст окнам документов эту способность, INV-D6 в буквальной формулировке
потребует правки на «≥1 вкладка ИЛИ ≥1 задокированное тул-окно».

## Раздел 5. Операции

**DA-5.1 — `OpenDocument(id)`: вставка после активной, активация вместо переноса.** `=`

Позиция вставки подтверждена дословно — `EditorWindow.addComposite` (`:352-446`):

```
var indexToInsert = options.index
if (indexToInsert == -1) {
  if (isPreviewMode) { indexToInsert = composites().indexOfLast { it.isPreview } }
  if (indexToInsert == -1) {
    indexToInsert = if (UISettings.getInstance().openTabsAtTheEnd) tabbedPane.tabCount else tabbedPane.selectedIndex + 1
  }
}
```

Дефолт `openTabsAtTheEnd` — `false` (`editor-ui-api/.../UISettingsState.kt:154`), рабочая ветка —
«после активной», ровно DA-5.1. «Уже открытый — только активация» — разобрано полностью в DA-2.4(a)
выше, тот же путь. Активация происходит через `setCurrentCompositeAndSelectTab` внутри
`options.selectAsCurrent` и `owner.setCurrentWindow(window = this)` (`EditorWindow.kt:413-427`) — это
и делает открытую-где-то-ещё вкладку одновременно активной в своей группе и (если группа в хосте
док-зоны) текущей группой хоста, соответствуя DA-6.1. Две несущественные для вердикта оговорки: (1)
preview-таб (`shouldReservePreview`, `:1199-1210`) может вычислить `indexToInsert` раньше ветки
«после активной» — наша спека сознательно не моделирует preview, чистая сноска; (2) `reuseOpen` —
соглашение вызывающей стороны у IDEA (`options.reuseOpen=false` даст второй композит), а не
архитектурный инвариант, как наш INV-D2 — наблюдаемого расхождения для конечного пользователя нет
(все стандартные точки входа просят `reuseOpen=true`).

**DA-5.2 — `CloseTab(id)`: активна предыдущая соседка; первая — новая первая.** `=`, через два слоя

Механизм в IDEA разложен на два независимых слоя. **Слой 1** — `EditorWindow.getTabToSelect`
(`:766-782`), вызывается из `closeFile` до удаления: сначала `tabBeingClosed.previousSelection`
(`:771-776`, если задан и виден — используется без обращения к порядковой логике); иначе —
**`computeIndexToSelect`** (`:825-855`): если закрываемая вкладка не была активной — активная не
меняется; иначе по приоритету — `activeMruEditorOnClose` (дефолт `false`,
`UISettingsState.kt:159`) → `activeRightEditorOnClose` (дефолт `PlatformUtils.isAppCode()` — т.е.
`false` для всех продуктов линейки кроме AppCode, включая саму IntelliJ IDEA) → **дефолт-путь**:
предыдущая соседка (`fileIndex - 1`), а при закрытии первой (`fileIndex==0`) — `-1`. Для обычного
пользователя рабочий путь — именно «предыдущая соседка», ровно DA-5.2.

`TabInfo.previousSelection` (слой, что «выигрывает» первым) документирован как механизм
IDEA-61536 и гасится на каждом `mouseReleased` (`TabLabel.kt:130-134,156`) — для обычного `CloseTab`
(меню/шорткат/крестик, не в разгар перетаскивания) он практически всегда `null`, и решает именно
`computeIndexToSelect`.

Когда `getTabToSelect` возвращает `null` (закрывается первая и активная одновременно) —
`editorTabs.removeTab(info, forcedSelectionTransfer=null)` передаёт `null` в **слой 2**,
независимый алгоритм `JBTabsImpl.removeTab`/`getToSelectOnRemoveOf`
(`platform-api/.../impl/JBTabsImpl.kt:2568-2624, 1973-2010`): сканирует назад (`findEnabledBackward`),
затем вперёд (`findEnabledForward`), пропуская **disabled**-вкладки — деталь, которой у
`computeIndexToSelect` нет вообще (чистый индекс, без проверки состояния). Для первой вкладки
(`index==0`) назад искать нечего — сразу вперёд, находит новую первую. Итог совпадает с DA-5.2, но
через второй, независимый слой — см. полный разбор в DA-E3 ниже.

**Композитные закрытия** («Close Others», «Close All») — подтверждено, что заявление спеки о
«композициях над `CloseTab`» верно для интерактивного пути:
`CloseAllEditorsButActiveAction`/`CloseAllEditorsAction` собирают список файлов и передают в
`closeFilesWithChecks`, которая после единой батчевой проверки закрывает **последовательным циклом**
`window.closeFile(...)` (`FileEditorManagerImpl.kt:909-930`). Отдельно есть архитектурно другой путь
— публичный `FileEditorManager.closeAllFiles()` → `EditorsSplitters.closeAllFiles`
(`EditorsSplitters.kt:537-562`), который сначала диспоузит объекты `EditorWindow`, потом гоняет
`closeFile` — не привязан ни к одному интерактивному действию (используется task-контекстом и
тестами). Конечное состояние то же, но в кодовой базе IDEA два разных «закрыть всё», и только один
эквивалентен N-кратному `CloseTab`.

**Диалог подтверждения при несохранённых изменениях отсутствует по умолчанию.**
`EditorWindow.closeFile` (`:706-764`) вообще не проверяет `isModified`. Единственный хук —
`VirtualFilePreCloseCheck` (EP, `core-api/resources/META-INF/Core.xml:36`) — без единой продакшен-
реализации в дереве репозитория (только тестовые). Подтверждает, а не опровергает нашу архитектуру
(ADR-0003): судьба контента — контракт фабрик, а не забота раскладки; у самой IDEA такого хука по
умолчанию попросту нет (тихий autosave).

**DA-5.3 — `ActivateTab(id)`: `CurrentTabId`/`ActiveToolWindowId` взаимно вытесняются.** `≈`

Наблюдаемый эффект совпадает с уже принятым TW-6.5 (idea-tracing.md) и независимо подтверждён здесь:
`activeToolWindowId` — вычисляемое свойство от `frame.mostRecentFocusOwner`
(`ToolWindowManagerImpl.kt:699-714`); триггер сброса — реальный перевод клавиатурного фокуса в
потомка редакторского композита, `FOCUS_GAINED`-ветка `handleFocusEvent`
(`ToolWindowManagerLifecycle.kt:84-92`), `activeStack.clear()`. Обычный клик по вкладке
действительно запрашивает фокус (`TabLabel.kt:140`, `requestFocus=true`) — для клика мышью эффект
идентичен нашему атомарному `ActivateTab`.

Вердикт `≈`, а не плоский `=`: у IDEA «выбор вкладки» (selection) и «перевод фокуса» (focus) —
**независимые** оси (`FileEditorOpenOptions.selectAsCurrent`/`.requestFocus` — разные поля,
`:11,14`); можно выбрать вкладку, не запрашивая фокус (`requestFocus=false`, например при
программном восстановлении раскладки на старте, `selectTabOnStartup`). В такой комбинации
`ActiveToolWindowId` у реальной IDEA **не** сбросился бы, тогда как наш `ActivateTab(id)` —
единая ядровая команда без понятия «фокус» отдельно от «активации» — сбрасывает его безусловно как
часть контракта команды. Для всех путей, которыми реально управляет пользователь, обе оси совпадают.

**DA-5.4 — `MoveTab(id, target, index)`: перенос с клемпом, активация в приёмнике.** `≈`

По отдельности почти все составляющие — `=`: клемп индекса (`JBTabsImpl.addTabWithoutUpdating`,
`platform-api/.../impl/JBTabsImpl.kt:1471-1499` — недопустимый индекс молча схлопывается в «в
конец», тот же принцип, что и TW-5.7); активация в приёмнике (`FileEditorOpenOptions.selectAsCurrent`
дефолт `true`, ни DnD, ни `MoveEditorToOppositeTabGroupAction` его не сбрасывают); активность в
доноре — по правилу DA-5.2 (DnD `dragOutStarted` зовёт буквально `window.getTabToSelect(...)`,
action-based move зовёт `window.closeFile(...)` напрямую — тот же код, что и обычное закрытие).

**Итоговый вердикт `≈`, а не `=`**: у IDEA **нет единого атомарного `moveTab`-примитива** на уровне
`FileEditorManager`/`EditorWindow`, доступного из кода/меню/DnD одинаково — ADR-0004 у нас это прямое
требование. У IDEA «перенос» реализован заново в каждом из трёх мест: `SplitAction.actionPerformed`
(close-then-split), `MoveEditorToOppositeTabGroupAction.actionPerformed` (close-then-open с ручным
сохранением состояния курсора через `currentStateAsFileEntry()`), `EditorTabbedContainerDragOutDelegate`
(hide-then-close-or-unhide). Наблюдаемый результат одного жеста совпадает с нашим `MoveTab` (выглядит
атомарным), но три независимые реализации «закрыть здесь + открыть там» — именно то «по-другому
упакованное», что определяет `≈`.

**DA-5.5 — `SplitTab(id, direction)`: сплит переносом, n-арная вставка вдоль оси, вложение
поперёк.** `≈` в целом (`=` по механике после перевода через N3)

`EditorWindow.split()` (`:509-601`) всегда делает одно и то же независимо от направления: заменяет
целевой лист, **на его текущем месте у текущего родителя**, новым 2-детным `Splitter`
(`swapComponents`, `:559`). Какая сторона — старое/новое окно — определяет `fileIsSecondaryComponent`
(`:560-567`): `true` (дефолт, используется `SplitAction`/`MoveTabRightAction`/`MoveTabDownAction`)
кладёт старую группу первой, новую второй — новая группа *после*, ровно Right/Down;
`SplitterService`/drag-drop вычисляют явно через `RelativePosition` (`SplitterService.kt:108`,
`DockableEditorTabbedContainer.kt:182`). Доля новой группы всегда жёстко `proportion=0.5f`
(`:550`) — «обе половина прежней доли», ровно как в DA-5.5.

Поскольку IDEA строго бинарна, у неё никогда нет выбора между «вставить n-арным соседом» (вдоль
оси) и «обернуть в новый перпендикулярный сплит» (поперёк) — она всегда оборачивает. N-арная
вставка вдоль оси восстанавливается из бинарного результата IDEA только *после* мысленного
применения N3 (которого сама IDEA никогда не делает). Проверено точно для обоих случаев:

- **Поперёк ориентации (форма DA-E13):** слияние не нужно вовсе — новый перпендикулярный `Splitter`
  сразу удовлетворяет «ориентация ребёнка ≠ ориентация родителя», сырой вывод IDEA уже совпадает с
  нашей канонической формой. Прямое `=` (детали — DA-E13 ниже).
- **Вдоль ориентации, сплит среднего из трёх (форма DA-E14):** бинарное представление IDEA для «три
  группы в ряд» — неизбежно какая-то вложенная пара одноориентационных `Splitter`. Сплит *средней*
  группы заменяет её собственный слот у родителя новым одноориентационным `Splitter`, который
  вкладывается **прямо внутрь** уже-одноориентационного внутреннего сплиттера. Развёртка
  получившейся 3-уровневой одноориентационной цепочки через N3 (с умножением долей, ровно как
  специфицирует N3) воспроизводит топологию `Row[A, B, New, C]` DA-E14 **и** точные доли для обоих
  возможных вложений IDEA — полный вывод см. DA-E14 ниже.

**DA-5.6 — `SetSplitShares(host, path, shares)`.** `≈`

Поведение «трогать только соседнюю пару» присутствует, но по структурно другой причине: наша спека
называет это *сознательным решением*, нужным из-за того, что настоящий n-арный сплит может иметь 3+
детей; IDEA в этом решении не нуждается — у `Splitter` вообще только пара, `getOtherComponent(comp)`
(`Splitter.java:674-679`) — это весь концепт соседа, «какую пару среди N» решать не из чего. Живой
обработчик драга, `OnePixelDivider.processMouseMotionEvent` (`:252-271`), зовёт
`mySplitter.setProportion(proportion)` синхронно на каждом `MOUSE_DRAGGED` без явного дебаунса в
коде приложения — «троттлинг допустим» совместимо (IDEA просто не утруждается), но полагается на
неявное сглаживание событий самой очередью AWT, а не на намеренный рейт-лимит. Валидации
длины/суммы-к-1 на стороне IDEA нет вообще для сравнения — форма валидации там другая (клемп одного
числа к фиксированному `[0.1,0.9]`, не проверка вектора).

**DA-5.7 — `MoveTabToNewWindow(id, bounds)`.** `≈`

Два независимых пути создания подтверждены, оба ведут в один общий приватный метод: drag-жест —
`createNewDockContainerFor(content, point)` (`DockManagerImpl.kt:399-433`) вычисляет целевой
`Rectangle` от точки сброса и предпочтительного размера, затем явно вписывает в текущий дисплей
(`ScreenUtil.moveRectangleToFitTheScreen`/`cropRectangleToFitTheScreen`, `:425-426`) — честное
мультимониторное позиционирование от точки отпускания. Программное «открыть в новом окне» —
`createNewDockContainerFor(file, fileEditorManager, ...)` (`:435-484`) **не** вычисляет и не вписывает
явный прямоугольник — зовёт `window.show(true)` без аргумента bounds, полагаясь целиком на
generic-путь восстановления/дефолта `FrameWrapper` (см. DA-7.4). Оба пути зовут один и тот же
приватный `createWindowFor(dimensionKey, id, container, canReopenWindow)` (`:486-504`), строящий
один `DockWindow` — подтверждает, что «аналог режима Window» DA-7.1-DA-7.3 опирается на один
реальный общий путь конструирования, а не на два независимо написанных двойника. Вердикт `≈`, а не
`=`: у Berth «UI даёт пиксели, ядро их не выдумывает» (ADR-0002) **не имеет аналога-разделения** у
IDEA вообще — `DockManagerImpl`, ближайшее к «владеющему раскладкой ядру» здесь, сам свободно
вычисляет и манипулирует пиксельной геометрией; слоя, свободного от пикселей так, как того требует
`Berth.Core`, у IDEA нет вовсе.

**DA-5.8 — `SetDocumentWindowBounds(id, bounds)`.** `≈`

Тот же наблюдаемый эффект (bounds переживают сессии, валидируются при восстановлении по DA-7.4), но
явной команды «задать bounds» у IDEA нет вовсе — механизм это **пассивный, per-window AWT-слушатель**
с **ленивой, pull-based записью**, полностью отделённый от модели дока/дерева: `WindowStateAdapter`
(`openapi/util/WindowStateAdapter.java:13-35`) лениво прикрепляется к целевому `Window` при первом
запросе состояния и с этого момента непрерывно отслеживает `componentMoved`/`componentResized` — без
троттлинга со стороны IntelliJ, полагаясь на естественное сглаживание событий AWT. Живое значение
пишется в персистентную карту лениво: `WindowStateServiceImpl.getStateFor` регистрирует
diff-check `Runnable` на ключ, срабатывающий только когда что-то в следующий раз спрашивает
состояние — нет явного вызова «на каждый ресайз — запиши в стор» ни в `FrameWrapper`, ни в
`DockWindow`. Функционально эквивалентно, но архитектурно это ближе к «общий OS-window-bounds кэш,
которым докинг-слой просто пользуется», чем к докинг-модельной команде.

## Раздел 6. Активность и фокус

**DA-6.1 — «Текущая группа» переживает фокус в тул-панели.** `≈` — **главная находка всей
трассировки**

Подтверждено для случая, для которого пункт буквально написан (один хост), но базовый механизм
IDEA **структурно двухуровневый**, и схлопывание его в одно плоское поле `CurrentTabId` безопасно
только при неявном допущении о проводке UI, которого спека сейчас не формулирует.

*Почему claim про один хост верен.* `EditorsSplitters` хранит своё «текущее окно» в приватном
`MutableStateFlow<EditorWindow?>` (`EditorsSplitters.kt:199`, геттер `:210-211`), обновляемом только
**приватным, per-instance** `MyFocusWatcher` (`:985-1019`), установленным в `init` (`:308`). Его
`focusedComponentChanged` (`:1000-1019`) зовёт `setCurrentWindow` только когда новополучивший фокус
компонент имеет предка `EditorWindowHolder` (`:1016-1018`) — т.е. когда фокус попадает внутрь дерева
сплитов *этого самого* инстанса. Контент тул-окна живёт вне любого дерева `EditorsSplitters`, так
что фокусировка панели никогда не трогает `_currentWindowFlow` — весь механизм «текущее окно
переживает фокус в панели» это не явная защита, а *отсутствие* подходящей ветки. Композиция с уже
проверенным TW-6.5 чистая (`activeStack.clear()` на editor `FOCUS_GAINED`).

*Где это перестаёт быть одним плоским значением.* `FileEditorManagerImpl` **не** хранит единое поле
«текущее окно/вкладка» для всего приложения. `currentWindow` (`:852-867`) и `getCurrentFile()`
(`:839-847`) оба делегируют в `getActiveSplitterSync()` (`:659-672`) — **живой** поиск по реальному
AWT/Swing focus owner на момент вызова (`getDockContainer`, `:674-677`, поднимается от focus owner
до содержащего `DockableEditorTabbedContainer`), с фолбэком на `mainSplitters` только когда фокус не
резолвится. Каждый `EditorsSplitters` — `mainSplitters` плюс по одному на каждое открытое окно
документов — независимо помнит своё `currentWindow` через свой `MyFocusWatcher`, и эта память
**никогда не трогается** активностью в другом хосте. Реальная модель: **N независимых per-host
значений «последнее текущее окно», плюс живой (не хранимый) выбор того, какой хост «активен» прямо
сейчас.**

*Конкретный сценарий расхождения.* Фокус в M1 главного окна (текущее окно main splitters = M1). Клик
по вкладке в окне документов A (текущее окно A = A2). Клик по вкладке в окне документов B (текущее
окно B = B2). Возврат к окну A **без клика по конкретной вкладке** — например, Alt-Tab или клик по
заголовку окна A; Swing/AWT автоматически восстанавливает клавиатурный фокус на последнем
сфокусированном потомке A. `getActiveSplitterSync()` теперь резолвится в сплиттеры A, и
`A.currentWindow` всё ещё A2 — корректно сохранён, потому что собственный watcher A никогда не
переставал его помнить, пока использовались B и M. Плоское поле `CurrentTabId`, если оно приводится
в движение только явной командой `ActivateTab`, вызываемой *по клику на заголовок вкладки*
(буквальное прочтение ADR-0004/DA-6.2), в этот момент всё ещё читало бы «вкладка в B2», потому что
в A клика по вкладке не было — произошла только переустановка фокуса на уровне окна. Собственный
`MyFocusWatcher` IDEA приводится в движение сырым `FocusEvent.FOCUS_GAINED` (охрана `cause.getID() !=
FocusEvent.FOCUS_GAINED` внутри `focusedComponentChanged`, `:1001`), который **срабатывает** и на
такое восстановленное Swing'ом внимание, не только на явные клики — включая документированную защиту
от «дребезга» при повторной активации окна кликом (`rollbackFocusGainedIfNecessary`, `:1021-1050`).

*Вывод.* Наш `CurrentTabId` (DA-2.5) воспроизводит это **только если** UI-слой Berth шлёт
`ActivateTab`-эквивалент на **любое** реальное получение фокуса редактором в *любом* хосте
док-зоны — не только на явные клики по заголовку вкладки. Это реальное, сейчас незафиксированное
требование к проводке, а не деталь, которую гарантирует само ядро. Помечено `≈`, не `=`: намерение
(«текущая группа переживает фокус в тул-панели») верно, и механизм корректно композируется для
одного хоста; обобщение на много хостов требует либо (a) явной пометки, что UI обязан трактовать
переключение фокуса на уровне хоста как событие активации, либо (b) структурной правки, делающей
«текущую вкладку» по-настоящему per-host (по образцу IDEA), с `getActiveSplitterSync()`-подобным
живым резолвом хоста как заботой UI-слоя (согласуется с уже существующими оговорками «UI вправе...»
в других местах спеки). Конкретный тестовый сценарий — кандидат DA-E-C1 в конце документа.

**DA-6.2 — `ActivateTab` сбрасывает `ActiveToolWindowId`; «активный документ» = `CurrentTabId` при
`ActiveToolWindowId=null`.** `=`

Переформулировка уже проверенного TW-6.5 (`activateToolWindow`, `activeStack.clear()` на editor
`FOCUS_GAINED`) в терминах документной зоны. Подтверждено структурно и здесь: контент панельной
вкладки живёт внутри дерева самой панели (вне любого `EditorsSplitters`), так что фокусировка на нём
никогда не триггерит `MyFocusWatcher.focusedComponentChanged` — ни у одного хоста дока «текущее
окно» не меняется, ровно «активация панели не трогает `CurrentTabId`».

**DA-6.3 — Фолбэк текущей вкладки при исчезновении `CurrentTabId`.** `≈`

Спека сама уже помечает это как `≈` (MRU, не точный алгоритм), и трассировка подтверждает
самооценку жёсткими фактами: **алгоритма фолбэка по порядку обхода дерева не существует нигде в
кодовой базе.** `FileEditorManagerImpl` никогда не хранит «текущую вкладку» — она вычисляется живьём
из реального focus owner при каждом запросе (`getActiveSplitterSync`, см. DA-6.1). Когда хост
исчезает (окно документов диспоузится, см. INV-D6), *следующий* запрос «текущего файла/окна» просто
отражает то, куда OS/AWT естественно перевели клавиатурный фокус — по нативной z-order-семантике
оконного менеджера и собственному MRU-учёту `IdeFocusManager`
(`getLastFocusedFor(lastFocusedIdeWindow)`), а не по правилу «предыдущая группа в порядке
главное-окно-затем-список-окон-документов». Единственный явный MRU-механизм, который реально
существует, `getLastFocusedSplitters()` (`FileEditorManagerImpl.kt:1796-1807`), сортирует **все**
сплиттеры (главное + каждое окно документов) по `lastFocusGainedTime` каждого — ранжирование по
недавности **поперёк хостов**, структурно не связанное с позицией в дереве. DA-6.3 — сознательное,
детерминированное упрощение генуинно недетерминированного (управляемого историей фокуса) эталонного
поведения; `≈` — уже правильный вызов.

## Раздел 7. Окна документов

**DA-7.1 — Окно документов = `Bounds` + дерево той же модели.** `≈`

Половина «та же модель» подтверждена сильно: дерево вкладок окна документов — настоящий экземпляр
`EditorsSplitters`/`EditorWindow`, тот же класс, что и главное окно, не параллельная реализация
(`createEditorDockContainer`, `DockableEditorContainerFactory.kt:42-74`). Где перестаёт совпадать
«Bounds + одно дерево, ничего больше»: окно документов IDEA может хостить **собственный независимый
`ToolWindowPane`** — настоящие стрипы с докируемыми/пришпиленными тул-окнами, полностью
ортогональные его дереву вкладок (подробности и цитаты — в INV-D6 выше). Этой возможности нет
аналога нигде в DA-7.1-DA-7.6 — наша спека делает окна документов чистыми деревьями вкладок без
концепции независимого хостинга тул-окон.

**DA-7.2 — Окно создаётся `MoveTabToNewWindow`, исчезает когда последняя вкладка закрыта/перенесена.**
`≈`

Создание подтверждено через два `createNewDockContainerFor`-пути (см. DA-5.7). Исчезновение при
опустошении реально, но точный гейт **не** просто «дерево пусто» — см. полный разбор в INV-D6 выше
(`closeIfEmpty()` учитывает и наличие тул-окон).

**DA-7.3 — Закрытие окна системной кнопкой = `CloseTab` всех вкладок; нет «режима» для возврата.**
`=`

Прямо подтверждено. `DockWindow.installListeners` вешает настоящий `WindowAdapter`
(`DockWindow.kt:255-271`): `windowClosing` зовёт `container.closeAll()` —
`DockableEditorTabbedContainer.closeAll()` (`:322-326`): `for (each in splitters.openFileList) close(each)`,
где `close(file)` (`:318-320`) зовёт `splitters.closeFile(file, false)` — настоящее закрытие для
каждой открытой вкладки, ровно «= `CloseTab` всех вкладок». И в отличие от тул-окон (которые
сохраняют `type` FLOAT/WINDOWED/DOCKED, восстанавливаемый следующим `Open`, TW-7.3), у `DockWindow`
нет такого поля режима: как только `closeIfEmpty()` его диспоузит, объект окна исчезает — ничего,
во что «переоткрыться». Ровно соответствует «≠ панелей TW-7.3» как и заявлено спекой.

**DA-7.4 — Валидация `Bounds` при `Apply`/восстановлении по правилу TW-7.4.** `≈`

Тот же наблюдаемый итог, но через **совершенно другой, не тул-окно-специфичный механизм** — не
буквальное переиспользование кода `isValidBounds`/`setExternalDecoratorBounds` TW-7.4
(`ToolWindowManagerDecorators.kt:212-268`), который для окон документов никогда не зовётся. Вместо
этого: `DockWindow extends FrameWrapper` (`DockWindow.kt:46-55`) — примечательно, **так же, как**
`WindowedDecorator` (Window-режим тул-окон, TW-7.2) — оба сиблинги под общим
`com.intellij.openapi.ui.FrameWrapper` (`FrameWrapper.kt:80`), но валидация bounds не реализована в
этом общем базовом классе — каждая иерархия подклассов делает её сама. Для `DockWindow`:
`FrameWrapper.show(restoreBounds=true)` → `loadFrameState(state)` (`:318-330`) — при `state==null`
берёт `ideFrame.suggestChildFrameBounds()`, иначе `state.applyTo(frame)`. `state` приходит из
**общего** `WindowStateServiceImpl` (используется бесчисленными не-редакторными плавающими окнами по
всей платформе, ключ — произвольная строка `dimensionKey`). `CachedState.get()`
(`WindowStateServiceImpl.java:354-416`) делает саму валидацию: если сохранённый экран отличается от
текущей конфигурации — `ScreenUtil.moveAndScale`+`ScreenUtil.moveToFit` вписывают в текущий дисплей,
и если результат всё ещё не виден — возвращает `null`, что уводит `loadFrameState` в ветку
«позиционировать у главного окна» (`ProjectFrameHelper.kt:560-567`: bounds главного окна с отступом
100px по каждому краю) — тот же итог, что описывает TW-7.4. Итог совпадает, механизм — нет;
формулировка спеки «по правилу TW-7.4» (подразумевающая общее правило) слегка преувеличивает связь
кода — стоит формулировки «аналогично TW-7.4», если для владельца важна точность.

**DA-7.5 — Окна документов в браузере — псевдоокна в оверлей-слое (ADR-0006).** `—`

Вне модели IDEA полностью — этот эталон только desktop Swing/AWT. Есть концепт «JetBrains
Client»/frontend-remote (`FrontendApplicationInfo`/`isJetBrainsClient`,
`EditorsSplitters.kt:513-516`), но это доставка того же Swing UI по протоколу удалённого рабочего
стола, не браузерный рендер-слой с нуля, сравнимый с оверлей-абстракцией ADR-0006. Сравнение здесь
не имеет смысла; корректно наша собственная конвенция.

**DA-7.6 — Закрытие главного окна: снапшот, затем закрытие окон документов.** `=`

Подтверждено, включая гарантию порядка, трассировкой реальной последовательности закрытия проекта в
`ProjectManagerImpl.kt`: `projectClosingBeforeSave` → `saveSettingsUnderModalProgress(project)`
(**реальное** сохранение состояния на диск — здесь выполняется `PersistentStateComponent.getState()`,
**пока каждое окно документов всё ещё полностью открыто**) → `fireProjectClosing` →
`closePublisher.projectClosing`. `DockManagerImpl` сам — `PersistentStateComponent`
(`@State(name="DockManager",...)`); его `getState()` обходит каждый контейнер и для тех, у кого
`window.supportReopen`, сериализует `window.id` плюс собственное состояние контейнера (для редакторов
— `splitters.writeExternal(...)`) — захватывая живой список вкладок. Только *после* этого срабатывает
`projectClosing`, и `FrameWrapper` (общий предок `DockWindow`) закрывается в ответ — **общий**
механизм, не специфичный для окон документов: `init`-блок `FrameWrapper` безусловно подписывается на
`ProjectCloseListener.TOPIC` и зовёт `close()` на `projectClosing` (`FrameWrapper.kt:111-117`),
разделяемый каждым `FrameWrapper`-производным окном платформы. При следующем запуске
`DockManagerImpl.readStateFor` реконструирует контейнеры/окна из сохранённого XML и переоткрывает их.
Итог точно соответствует DA-7.6, достигнут через стандартный save-before-teardown контракт
платформы, а не через пару bespoke-слушателей, как использует TW-7.5 — arguably более чистый
прецедент для Berth, чем сама TW-7.5.

## Раздел 8. Вкладки панелей и док-зона

**DA-8.1 — таблица допуска `canHost`.** `≈` (для ячейки «панель P → хосты док-зоны») / `—` (для
остальных)

IDEA не имеет единого реестра `canHost`, проверяющего владельца вкладки по id внутри одного дерева
(уже установлено в TW-9.7, idea-tracing.md, `—`). Однако прямая проверка исходников
terminal-плагина показала, что для направления «панель → док-зона» у IDEA всё же есть **настоящий,
двунаправленный, DnD-интегрированный мост** — не общий реестр, а opt-in SPI на конкретном тул-окне.

Платформенный интерфейс: `ToolWindowInEditorSupport`
(`openapi/wm/impl/content/ToolWindowInEditorSupport.kt:10-13`):

```kotlin
interface ToolWindowInEditorSupport {
  fun canOpenInEditor(project: Project, content: Content): Boolean
  fun openInEditor(content: Content, targetWindow: EditorWindow)
}
```

Единственная известная реализация — `TerminalInEditorSupport`
(`plugins/terminal/frontend/.../toolwindow/impl/TerminalInEditorSupport.kt:15-24`), регистрируется на
конкретном `ToolWindow` через `ToolWindowContentUi.setToolWindowInEditorSupport`
(`openapi/wm/impl/content/ToolWindowContentUi.java:897-907`). Мост подключён прямо в общий
drag-helper тул-окон, не bespoke-хак: `ToolWindowInnerDragHelper.kt` проверяет `canOpenInEditor`
дважды за жест — при определении, можно ли начать перетаскивание одиночной вкладки (`:67-86`), и
непрерывно во время движения мыши (`updateCurDropLocation`, `:395-421`, с комментарием «Drop into the
editor - allowed only if the tool window provides necessary support», `:415-417`). На дропе —
`dropIntoEditor` (`:264-270`) зовёт `support.openInEditor(content, editorWindow)`.

Направление «панель → док-зона» подтверждено действием `MoveTerminalSessionToEditorAction`
(`plugins/terminal/frontend/.../action/MoveTerminalSessionToEditorAction.kt:26-76`): живая
терминальная сессия **не пересоздаётся**, а извлекается (`manager.detachTab(terminalTab)`, `:44-45`)
и заворачивается в `VirtualFile`-фасад (`TerminalViewVirtualFile`, `:47`), который открывается
штатным `FileEditorManager.openFile`. Флаг `FileEditorManagerKeys.CLOSING_TO_REOPEN` явно маркирует
операцию как «не настоящее закрытие», а перенос. Обратное направление тоже существует —
`TerminalDockContainer` (`.../TerminalDockContainer.kt`) с doc-комментарием прямо в коде: «Terminal
tab can be moved to the editor... This dock container is responsible for an ability to drop the
editor tab with the terminal back to the terminal tool window» (`:24-28`). `getContentResponse`
(`:37-42`) принимает дроп только для терминальных `DockableContent`; `add` (`:48-70`) зовёт
`TerminalToolWindowTabsManager.attachTab` — обратную операцию к `detachTab`, тоже без пересоздания
сессии. Независимость контента от «закрытия»/«переноса» источника закодирована явно:
`TerminalToolWindowTabsManagerImpl.kt:186-193` — сессия завершается только если таб был реально
закрыт, но не если он был отсоединён (`TAB_DETACHED_KEY`).

**Вывод.** Ячейка DA-8.1 «Хосты док-зоны ← вкладки панели P: да» — не выдумка без прецедента; у
IDEA есть реальный, живой (session-preserving), двунаправленный, DnD-подключённый мост для этого
конкретного направления. Отличие от нашей модели: это opt-in SPI на тул-окне, типизированный по
конкретному классу `VirtualFile`-обёртки, а не универсальная проверка владельца по `Id` в одном
дереве — платформа не знает общего понятия «вкладка панели» вне `ContentManager`; для каждого
будущего моста нужен отдельный `VirtualFile`+`FileEditor`+`DockContainer`. Отсюда `≈`, не `=`.
Ячейки «нет» остаются `—`: обратный мост типизирован строго под terminal-контент и в принципе не
примет произвольный документ; обобщённого механизма «любой тул-window принимает контент
другого/редактора» нет. Уточняет (не опровергает) исходную `—`-оценку TW-9.7: полного отсутствия
моста нет, но и обобщённого моста тоже нет — есть только один частный двунаправленный канал.

**DA-8.2 — перемещения панель ↔ док-зона теми же командами.** `—`/`≈`, наследует TW-9.8/9.9/9.10

У IDEA нет общих команд `MoveTab`/`SplitTab`/`MoveTabToNewWindow`, работающих одинаково для тул-окна
и дока — уже установлено в TW-9.8/9.9/9.10. Terminal-мост подтверждает это конкретно: перенос
реализован не общей командой ядра, а специализированной парой `detachTab`/`openFile` (туда) и
`getContentResponse`/`attachTab` (обратно) — два разных API с разными сигнатурами, не одна
параметризуемая операция. Ограничение «уехать может только в панель-владельца» тоже подтверждается:
`TerminalDockContainer` всегда резолвит синглтон-менеджер терминала на проект и принимает дроп
только терминального контента, так что editor-хостед terminal-вкладка структурно не может
приаттачиться ни к какому тул-окну, кроме терминального — по духу совпадает с ограничением DA-8.2,
хотя у IDEA это следствие типизации канала, а не явной проверки владельца.

**DA-8.3 — вкладки панели в док-зоне независимы от панели-владельца.** `≈`, усиливает TW-9.10
конкретным примером

Помимо общей независимости персиста (TW-9.10), terminal-мост даёт живой рантайм-пример именно
сценария DA-8.3: `TerminalToolWindowTabsManagerImpl.detachTab` (`:104-128`) — после детача, если
content-manager тул-окна опустел, тул-окно **прячется** (`toolWindow.hide()`, `:120-123`), но это
никак не затрагивает уже открытую в редакторе вкладку — она получила независимую жизнь через
`TerminalViewVirtualFile`, не связанную с `Content`/`ContentManager` тул-окна. Можно довести
тул-окно-владелец до полностью пустого/скрытого состояния, пока его бывшая вкладка продолжает жить и
работать в док-зоне — ровно то, что утверждает DA-8.3 и DA-E11. Обратное направление тоже
закодировано явно: `closeOnProcessTermination`-параметр протаскивается через весь путь как
самостоятельный флаг жизненного цикла, отдельный от «где сейчас показывается вкладка».

**DA-8.4 — вырожденное дерево панели прячет полосу табов [UI].** `≈`, наследует TW-9.5

Пункт помечен `[UI]` и прямо ссылается на TW-9.5, уже полностью протрассированную в idea-tracing.md.
Новых Core-претензий нет: «пустое дерево панели легально» — прямое следствие DA-2.3, видимость
полосы табов — рендер-слой вне контракта, который трассирует этот документ.

## Раздел 9. Персистентность

**DA-9.1 — область `Full`; `Arrangement` не затрагивает док-зону.** `≈`, наследует TW-10.6/TW-10.7

`FileEditorManagerImpl` — отдельный, самостоятельный `PersistentStateComponent`
(`@State(name="FileEditorManager", storages=[Storage(StoragePathMacros.PRODUCT_WORKSPACE_FILE)]...)`,
`:201-202`) с `getState()`/`loadState()` (`:1947-1963`), сериализующими/десериализующими **всё**
дерево сплиттеров одним вызовом — «всё или ничего», без какого-либо понятия «частичного»,
«arrangement-only» состояния. У IDEA попросту нет второго, облегчённого API для дерева редакторов,
аналогичного `setLayout`-семантике тул-окон (TW-10.7). Подтверждает наш дизайн от противного: раз на
стороне дока нет вообще никакого механизма, который «Arrangement» мог бы случайно задеть, требование
«Arrangement не затрагивает док-зону вовсе» не встречает сопротивления модели IDEA — но и не находит
там подтверждения как отдельная спроектированная возможность. `CurrentTabId`-аналог —
`currentInTab`-атрибут на `FileEntry` (`platform/.../fileEditor/FileEntry.kt:33,40,83`), часть того
же единого XML-снапшота, ровно как в нашем `Full`.

**DA-9.2 — нормализация при `Apply` (дедуп первого вхождения, доли к сумме 1).** `—`/`≈` — см.
DA-E9/DA-E10 в разделе 10 ниже

**DA-9.3 — восстановление контента лениво; фабрика вправе не восстановить вкладку.** `≈`, с важной
оговоркой: IDEA неоднородна между «файл исчез» и «провайдер исчез»

Случай «файл исчез» (URL не резолвится) действительно приводит к молчаливому пропуску вкладки —
прямое совпадение с DA-9.3. `UiBuilder.processFiles` (`EditorsSplitters.kt:1259-1353`) вызывает
`computeFileEntry` параллельно, затем `.mapNotNull { it.getCompleted() }` — записи, вернувшие `null`,
тихо выпадают до создания composite/tab. `computeFileEntry` отдаёт `null`, если
`resolveFileOrLogError` не смог найти файл. Никакой вкладки для такого файла не появляется вовсе.

Случай «провайдер исчез» (файл на месте, но `FileEditorProvider` разрегистрирован — плагин выгружен)
устроен **иначе** — см. DA-9.4 ниже: вкладка не пропадает, а создаётся composite с пустым списком
редакторов, остающийся видимым, но пустым. «Фабрика вправе не восстановить вкладку» верно только для
под-случая «сам объект контента не резолвится»; для под-случая «конкретно провайдер не резолвится»
IDEA предпочитает не закрывать, а показать пустоту.

**DA-9.4 — ⚠ вкладки незарегистрированного владельца (открытый вопрос владельцу).** `—` для самой
ситуации, но найден точный и информативный ближайший прецедент

Прямого аналога нет: у IDEA контент тул-окна и контент редактора — разные вселенные, «вкладка панели
без регистрации владельца в дереве дока» структурно невозможна как факт (см. DA-8.1). Но ближайшая
реальная ситуация — «сохранённая вкладка редактора ссылается на `FileEditorProvider`, которого
больше нет» — это и есть аналог «спящего владельца». У IDEA два разных ответа в двух разных
подсистемах, оба мягче нашего текущего «удаляется с нормализацией и отчётом»:

**(1) Живой layout — деградация, не удаление.** При восстановлении с кэшированным списком
провайдеров IDEA сперва пытается резолвить именно сохранённые id провайдеров, но если хоть один не
резолвится — **не выбрасывает вкладку**, а целиком отбрасывает кэш и пересчитывает список
провайдеров с нуля («что вообще умеет открывать этот файл сейчас»):
`EditorCompositeModelManager.kt:85-114`:

```kotlin
private fun CoroutineScope.computeFileEditorProviders(...): Deferred<List<FileEditorProvider>> {
  if (fileEntry == null || fileEntry.ideFingerprint != ideFingerprint()) {
    return async { ... getProvidersAsync(project, file) }
  }
  return async {
    val list = fileEntry.providers.keys.mapNotNullTo(...) { fileEditorProviderManager.getProvider(it) }
    // if some provider is not found, compute without taking cache in an account
    if (fileEntry.providers.size == list.size && list.isNotEmpty()) list
    else { LOG.warn(...); fileEditorProviderManager.getProvidersAsync(project, file) }
  }
}
```

Только если и полный пересчёт возвращает пустой список, вкладка становится видимо пустой, но **всё
ещё существует** в дереве (`EditorComposite.kt:281-288`, до сих пор помечено `// TODO comment this
and log a warning` в самих исходниках IDEA — разработчики платформы сами считают этот угол
недоведённым до конца, а не осознанным дизайном). Заголовок/иконка вкладки при этом никуда не
деваются — читаются независимо из кэша `FileEntryTab`.

**(2) MRU-история — точечная прополка, не удаление записи.** Слушатель на удаление extension'а
провайдера **не трогает всю запись** — вычищает только её (`EditorHistoryManager.kt:62-68`,
`HistoryEntry.kt:119-124`, `onProviderRemoval`): `if (selectedProvider === provider) selectedProvider
= null; providerToState = providerToState.removing(provider)`. Запись (файл) остаётся в MRU-истории
со всеми остальными провайдерами живой.

**Итог для владельца.** Ни один из двух реальных механизмов IDEA не выбирает наш путь «стереть с
отчётом»: (1) сначала пытается заменить исчезнувший провайдер актуальным списком (то есть скорее
«переоткрыть, чем закрыть»), и только когда заменить нечем — оставляет пустую, но живую вкладку
(осознанно недоделанный угол, `TODO` в исходниках); (2) для истории — хирургически вычищает только
мёртвую часть записи. Ослабленный вариант, ближе к обоим прецедентам сразу: не удалять вкладку панели
из дерева дока на `Apply`, а материализовывать её как «пустую/неизвестную» (аналог заголовка без
содержимого), и требовать явного `CloseTab`, если пользователь захочет от неё избавиться. Прецедента
однозначного «удалить сразу» — ни в живом layout, ни в истории — не нашлось нигде.

**DA-9.5 — формат и версия, общие с tool-windows.** `≈`, наследует TW-10.1/TW-10.5

`EditorHistoryManager` и `FileEditorManagerImpl` используют тот же XML/JDOM-в-workspace-файле
механизм, что и тул-окна (`Storage(StoragePathMacros.PRODUCT_WORKSPACE_FILE)`, буквально совпадает с
TW-10.1). Явного `SchemaVersion`-атрибута верхнего уровня, как проектирует DA-9.5, нет ни у
`FileEditorManagerImpl`, ни у `EditorHistoryManager` — версионирование полагается на
`IdeFingerprint` (используется для решения «доверять ли кэшу заголовка/иконки/провайдеров», не для
отказа в загрузке) и общую толерантность JDOM-парсинга. Прямого аналога «явная ошибка загрузки при
неподдерживаемой версии» (TW-10.5) для редакторного/MRU состояния не нашлось — см. DA-E13
(кандидат) ниже про частичную порчу.

### Детали, заслуживающие развёрнутого изложения

**DA-E1.** У IDEA два несогласованных живых поведения на один и тот же ввод — не единое правило.
`SplitAction.isEnabled` требует `window.getTabCount() >= 2` при `closeSource=true`
(`SplitAction.java:65-74`) — команды-кандидаты на роль `=`-аналога `SplitTab` (`MoveTabRightAction`/
`MoveTabDownAction`) **дизейблены полностью** на группе с одной вкладкой; IDEA предотвращает
ситуацию на уровне UI-enablement, а не нормализует её. Интерактивный split-chooser такого гейта не
имеет: `SplitterService.split(move=true)` зовёт `activeWindow.split(...)` безусловно, `split()`'s
собственная защита отклоняет только `tabCount<1` — при перемещении единственной вкладки того же окна
в направление, где «соседняя группа» — сама эта же группа, `split()` создаёт нового соседа и тут же
`initialWindow.closeFile(file)` осушает исходное окно до нуля вкладок, немедленно триггеря
`removeFromSplitter()` — только что созданный 2-детный сплиттер тут же схлопывается обратно в один
голый оконный узел. Наш единообразный «всегда разрешить, пусть нормализация уладит вырождение» дизайн
покрывает **оба** несогласованных реальных поведения IDEA одной логикой.

**DA-E8.** `removeFromSplitter()` на G в `Row[A, Column[G, Row[B,C]]]`: `splitter=G.parent`
(средний `Column`), `otherComponent=Row(B,C)`; поскольку средний `Column` — `secondComponent`
внешнего `Row`, продвижение ставит `outerRow.secondComponent = innerRow(B,C)` — **одним** вызовом,
заменяя весь 2-детный `Column` (с G) выжившим соседом. Этот единственный атомарный replace делает
работу и N1 (G, пустая, исчезает), и N2 (теперь однодетный `Column` схлопывается к оставшемуся
ребёнку) одновременно — у IDEA никогда не бывает промежуточного состояния «G ушла, но `Column` с
одним ребёнком ещё существует». Результат, `outerRow(A, innerRow(B,C))`, — `Row`, напрямую вкладывающий
`Row`, именно та одноориентационная вложенность, которую запрещают наши N3/INV-D1, и IDEA оставляет её
так навсегда. Мысленное применение нашего N3 (слить одноориентационного ребёнка, распределить его долю
его же детям умножением) к `outerRow(A, innerRow(B,C))` даёт точно `Row[A,B,C]` с долями B/C,
масштабированными на прежнюю долю внутреннего ряда внутри внешнего — совпадает с DA-E8 «доли B, C
масштабированы» дословно.

**DA-E14.** Явный вывод долей, не только топологии. У IDEA нет прямого 3-детного узла для сплита —
проверены оба возможных бинарных представления «A(0.3) B(0.4) C(0.3) в ряд»: **право-лежащее**
(`Outer(A, Inner(B,C))`, `Outer.proportion=0.3`, `Inner.proportion=0.4/0.7≈0.5714`) — сплит B
устанавливает `Inner.firstComponent = newSplitter(B,New)`, `proportion=0.5`; развёртка через N3 даёт
`[A,B,New,C]` с `A=0.3` нетронуто, `C=(1-0.5714)×0.7=0.3` нетронуто, `B,New` делят
`0.5714×0.7=0.4` пополам → `0.2` каждой. **Лево-лежащее** представление (`Outer2(Inner2(A,B), C)`)
даёт ту же топологию и те же доли по симметричной арифметике. Оба единственно возможных
представления одного и того же видимого ряда, после правильной развёртки через N3, приземляются
точно на предсказанные DA-E14 `A=0.3, B=0.2, New=0.2, C=0.3` с нетронутыми соседями — подлинный `=`,
и обнадёживающее доказательство, что операции нашей спеки не зависят от конкретного представления
вложенности, как и задумано ADR-0005/INV-D1.

| DA-E-C10 | Восстановление дерева стартует раньше, чем все внешние регистрации (плагинные фабрики/провайдеры) точно готовы | `Apply`/`Load` на старте приложения | IDEA сознательно терпит гонку, не гарантирует порядок (`createEditorCompositeModelOnStartup`, комментарий «we don't check canOpenFile (we don't have providers yet)», `FileEditorManagerImpl.kt:1506-1522`) — материализация дерева опережает готовность реестра провайдеров, пустота компенсируется позже |
| DA-E-C11 | Вкладка панели физически перенесена в док-зону через реальный мост (не гипотетика) | `MoveTab` панель→док-зона | Перенос не должен восприниматься нижестоящими подписчиками/MRU как закрытие+открытие — IDEA явно маркирует специальным флагом (`FileEditorManagerKeys.CLOSING_TO_REOPEN`) именно для этого; стоит решить заранее, эмитить ли Core будущее событие «TabMoved» отдельно от close+open |
| DA-E-C12 | `Row[A(0.1+ε), B]`, где A уже почти у своего минимума | `SplitTab(A, Right)` несколько раз подряд по той же группе | У нас легально по INV-D3 сколь угодно близко к 0; у IDEA каждый отдельный `Splitter` индивидуально ограничен `[0.1,0.9]`, хотя составной эффект нескольких вложенных 10%-лимитированных сплитов может дать такой же маленький итоговый видимый регион — конкретизация INV-D3-расхождения |
| DA-E-C13 | `Row[A(0.3), B(0.7)]` | Получить `Column[A(0.3), B(0.7)]` — те же вкладки, без потери долей, без лишних групп | У IDEA прямой одношаговый аналог есть — действие `ChangeSplitOrientation` (класс `SplitterAction.ChangeOrientation` → `EditorWindow.changeOrientation()`, регистрация `PlatformActions.xml:935`, группа `EditorTabsGroup`, та же группа, что `SplitVertically`/`Unsplit`/`ReopenClosedTab`): переключает ориентацию родительского `Splitter` на месте с теми же двумя детьми, без потери пропорции. Не New-UI-специфика и не легаси — обычное, не отгороженное по UI-теме действие, доступное через контекстное меню вкладки/Find Action; дефолтного шортката в стандартном кеймапе IDEA нет (только у VSCode-кеймап-плагина). *(Уточнение: альтернативный путь — Ctrl/Cmd-клик по разделителю через `Splitter.setAllowSwitchOrientationByMouseClick(true)`/`OnePixelDivider.java:277-286` — существует только как выключенная по умолчанию возможность виджета; вызовов этого метода, включающих её, нигде в `intellij-community-master` не нашлось — мёртвый путь, не часть реально работающего механизма.)* В DA-5.x нет соответствующей команды вовсе — не «сознательное расхождение», а неадресованный пробел; стоит решить явно: завести `RotateSplit`-команду или подтвердить, что 3-шаговая lossy-композиция (сплит поперёк + перенос + закрытие) — приемлемый ответ |
| DA-E-C14 | Вкладка D — единственная в окне A; окно B (другое) — активное | Интерактивный split-chooser на D с фокусом на B, выбрать «Split» | У IDEA один жест одновременно создаёт группу **внутри B** и открывает туда D, **и** закрывает D в A — `SplitterService` поддерживает `activeWindow != initialEditorWindow` (навигация `nextWindow`/`previousWindow`), затрагивая два структурно несвязанных места дерева одним жестом. `SplitTab` в нашей спеке всегда сплитит **собственную** группу перемещаемой вкладки — этот составной эффект неадресован; стоит явная пометка объёма |

## Итог: что перенять и на что смотреть

**Совпадает с IDEA (можно опираться уверенно):** позиция вставки новой вкладки и активация вместо
переноса при повторном открытии (DA-5.1); выбор соседки после закрытия, дефолт-путь (DA-5.2);
взаимное вытеснение активности документа/панели на уровне наблюдаемого эффекта (DA-5.3/6.2, TW-6.5);
клемп индекса при переносе (DA-5.4); механика `split()` — куда встаёт новая группа, доля 0.5
(DA-5.5); закрытие окна системной кнопкой = закрытие всех вкладок без «режима» (DA-7.3); порядок
снапшот-затем-закрытие при закрытии главного окна (DA-7.6); валидация bounds при восстановлении, по
эффекту (DA-7.4/5.8, DA-E15); нормализация «пустая группа»/«однодетный сплит» (N1/N2); фолбэк
активной вкладки на первую при восстановлении (N5); арифметика сплитов вдоль/поперёк оси после
корректной развёртки через N3 (DA-E13/E14).

**Расхождения — держать в поле зрения при ревью спеки:**

1. **`CurrentTabId` структурно двухуровневая у IDEA** (per-host + живой резолв активного хоста из
   фокуса), наша спека делает поле плоским (DA-6.1) — требует явного решения: зафиксировать
   требование к проводке UI («любой focus-gained в хост = activation») или сделать модель per-host.
2. **INV-D3 у IDEA строже на уровне состояния** — жёсткий `[0.1,0.9]`-клемп в самом `Splitter`, не
   только на рендере. Наша спека разрешает состоянию весь `(0,1)`.
3. **INV-D6/DA-7.1/7.2 — окно документов IDEA переживает пустое дерево вкладок**, если хостит хотя
   бы одно тул-окно (независимый `ToolWindowPane`). Не проблема сегодня (Berth не даёт окнам
   документов такой способности), но повод пересмотреть формулировку, если способность появится.
4. **DA-8.1 стоит переформулировать с `—` на `≈`** для направления «панель → док-зона» — есть
   работающий прецедент (`ToolWindowInEditorSupport`/Terminal), не только теоретическая возможность.
5. **DA-9.4 стоит пересмотреть в сторону мягче** — оба реальных прецедента IDEA (деградация вместо
   удаления в живом layout; точечная прополка в MRU) избегают полного удаления записи.
6. **DA-E9/INV-D2 — верный вердикт `—`, не `≈`**: у IDEA нет дедупликации при восстановлении, потому
   что нет самого понятия дубля-как-дефекта.
7. **DA-2.4 сформулирована шире факта** — расходится не дефолт (совпадает дословно), а сама
   возможность дубликата (у IDEA — редкий явный жест, у нас — структурно невозможна).
8. **N3 не имеет аналога у IDEA вовсе** (бинарность делает вложенность перманентной) — уже
   акцентировано ADR-0005, трассировка добавляет точное арифметическое подтверждение эквивалентности.
9. **DA-5.4/`MoveTab` не имеет единого атомарного аналога** — у IDEA три места заново реализуют
   close-then-open с разной механикой; наш единый примитив — architecturally чище, не искажение.
10. **DA-1.3 — уточнение, не блокер**: окна документов (не узлы дерева) получают у IDEA мелкий
    персистентный counter-id для восстановления bounds — техническая деталь, не меняющая расчёт по
    самому вопросу.

**Специфичные для владельца открытые вопросы:**

- **DA-1.3** (адресация без id): подход подтверждён IDEA на уровне узлов дерева дословно
  (`≈ IDEA`, как спека уже и утверждает) — новых оснований пересматривать решение нет. Единственная
  находка — окна документов у IDEA всё же получают лёгкий counter-id для восстановления bounds;
  можно рассмотреть аналогичный минимальный id **только для окон**, если/когда потребуется, но это
  не отменяет и не усложняет решение «нет id для узлов дерева».
- **DA-9.4** (вкладки незарегистрированного владельца): оба найденных прецедента IDEA («переоткрыть
  актуальным набором фабрик, а не удалить» и «прополоть только мёртвую часть записи») говорят в
  пользу более мягкого ответа, чем текущее «удалить с отчётом». Рекомендация: рассмотреть вариант
  «материализовать как пустую/неизвестную вкладку, требовать явного `CloseTab`» — ближе к обоим
  прецедентам сразу, хотя ни один не воспроизводится буквально.

**Новые вопросы, которые трассировка подняла и которых не было в исходном списке ⚠:**

- Нужно ли явно зафиксировать в спеке требование к UI-проводке для DA-6.1 (per-host focus-gained →
  activation), или переработать `CurrentTabId` в per-host модель?
- Стоит ли ослабить/переформулировать INV-D3, разрешив UI задавать свой клемп (аналог IDEA
  `[0.1,0.9]`), или сознательно оставить наш `(0,1)` как есть с формулировкой «мы строже
  разрешаем, чем эталон»?
- Нужна ли `RotateSplit`-команда (аналог IDEA `changeOrientation`, DA-E-C13), или 3-шаговая
  композиция — приемлемый ответ на v1? Уточнение к причине асимметрии (не источник в IDEA, а
  анализ нашей же модели): дело не в арности как таковой, а в том, что у IDEA нет инварианта
  канонической формы вообще (см. N3 выше) — перевернуть тег ориентации там всегда no-op, потому что
  совпадение ориентации с родителем/ребёнком там не запрещено. У нас INV-D1 запрещает ребёнку
  повторять ориентацию родителя — значит, «просто перевернуть тег» гарантированно триггерит N3
  **в обе стороны**, как только у узла есть родитель (сплит сливается вверх) или ребёнок-сплит
  (тот сливается вниз). Единственный по-настоящему бесплатный случай — узел является корнем
  дерева и все его дети листовые (ровно сценарий `Row[A,B]` из DA-E-C13, и он одинаково дёшев для
  любого N, не только для двух детей). Как только есть вложенность (у нас она реже, чем у IDEA,
  но не исчезает — например, настоящая сетка 2×2 всё равно требует вложенного сплита), поворот
  каскадно схлопывает соседние уровни через N3 — детерминированно (нормализация уже это умеет,
  DA-3.2), но не всегда визуально совпадает с «физическим поворотом всей композиции на 90°», как
  мог бы интуитивно ожидать пользователь. Вопрос не в реализуемости, а в UX: схлопывать молча (как
  уже делает `SplitTab` на последней вкладке группы, DA-E1) или предупреждать/ограничивать поворот
  только листовыми узлами.
- Стоит ли явно зафиксировать вне-периметра статус undo-стека закрытых вкладок (DA-E-C7, аналог
  `ReopenClosedTabAction`), раз он уже реален у эталона и такого пункта нет в разделе 11 «Вне рамок
  v1»?
