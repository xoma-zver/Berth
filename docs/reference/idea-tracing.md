# Эталон: как устроен докинг тул-панелей в IntelliJ IDEA

Настольный справочник по образцу (открытые исходники `intellij-community`, Apache 2.0):
устройство подсистемы тул-окон IDEA, ключевые файлы и сознательные расхождения Berth с
эталоном. Не нормативен — поведение Berth определяет `docs/spec/tool-windows.md`; здесь —
«как это сделано у них» и «почему у нас так». Построчная трассировка каждого пункта спеки
(вердикты `=`/`≈`/`≠` с цитатами `файл:строка`), по которой принимались решения, — в
git-истории этого файла (версии до 2026-07).

## Модель соответствия терминов

| Термин Berth | В IDEA | Замечание |
|---|---|---|
| Слот (Side, Group) | `anchor ∈ {LEFT,RIGHT,BOTTOM,TOP}` + `isSplit: Boolean` | `Primary` = `isSplit=false`; у IDEA есть ещё TOP — мы не берём |
| Прописка (`Side`,`Group`,`Order`) | `anchor` + `isSplit` + `order: Int` | `order = -1` — «не задан, в конец» |
| Режим (5 шт.) | `type: {DOCKED,SLIDING,FLOATING,WINDOWED}` × `isAutoHide` | `DockPinned`=DOCKED; `DockUnpinned`=DOCKED+autoHide; `Undock`=SLIDING; `Float`=FLOATING; `Window`=WINDOWED |
| `LastInternalMode` | `internalType` | та же память возврата из Float/Window |
| Слой слота (INV-2) | равенство `type` при вытеснении в `doShowWindow` | DOCKED и SLIDING — разные `type`, потому сосуществуют |
| `Weight` стороны | `UnifiedToolWindowWeights[anchor]`, дефолт 0.33 | одно число на сторону |
| `PairRatio` + выводимая доля R1 | одно поле `sideWeight` на панель + транзиентный кэш пропорции | у нас доля пары **выводится** нормализацией `P/(P+S)` — одно состояние вместо двух синхронизируемых |
| Толщина Undock-оверлея | `weight` окна, при дефолте new UI затираемый весом стороны | у нас сразу `Weight` стороны, отдельного поля нет |
| `IsOpen` / `IsIconVisible` | `isVisible` / `isShowStripeButton` | |
| `ActiveToolWindowId` | **вычисляется** из фокуса, не хранится | у нас хранимое поле; «активность» для шортката TW-5.5 всё равно определяется по фокусу |
| `QuickAccessSide` | `ToolWindowManagerState.moreButton` | |
| `ContentTree` | `ContentManager` + режимы `InternalDecoratorImpl` | у IDEA split бинарный и модель контента панели ≠ модели редакторов — наше единое дерево это дизайн-решение |

## Устройство по подсистемам

### Слоты, стрипы, вытеснение

Слот — пара `anchor` + `isSplit`; всё вытеснение ключуется по `(anchor, isSplit, type)`:
`doShowWindow` скрывает только окна **того же** `type`, поэтому докированная панель и
sliding-оверлей сосуществуют (наш INV-2 цитирует это точно). `order` у IDEA нормализуется
по `anchor` — Primary и Secondary одной стороны делят одну нумерацию, сегменты разводит
компаратор стрипа (наш INV-3 строже: плотность на слот). Каждый тулбар держит
`topStripe` + `bottomStripe` + кнопку «⋯» между ними; разделитель сегментов — виртуальный
элемент списка, виден только при обоих непустых сегментах; нижний сегмент new UI
инвертирован — читается снизу вверх. Переполнение стрипа иконками IDEA не обрабатывает
вовсе — вырожденный случай, на который эталон просто не рассчитывает (наш раздел 12).

### Геометрия

Два постоянных `ThreeComponentsSplitter` (вертикальный держит BOTTOM, горизонтальный —
LEFT/RIGHT + редактор); widescreen — только смена их вложенности (~20 строк). Вес — одно
число на сторону (`UnifiedToolWindowWeights`); open/close/switch его не трогают, панель
показывается в унаследованной ширине, а при активации даже **берёт** вес стороны себе:
при дефолте new UI (`rememberSizeForEachToolWindowNewUI = false`) персональный `weight`
окна затирается весом стороны при каждой активации — отсюда наше «толщина Undock-оверлея
= `Weight` стороны». Пара группы: одно поле `sideWeight` на панель; живое образование
пары (`addAndSplitDockedComponentCmd`) асимметрично — прибывший Primary диктует чистое
предпочтение, прибывший Secondary нормализуется формулой `old/(old+new)`; наш R1 взял
нормализацию симметрично (совпадает с эталоном при согласованной паре). Драг разделителя
пары пишет `sideWeight` **обоих** декораторов (у каждого свой `ComponentListener`) — наш
R2 «учим обе» = IDEA. Минимальные размеры — на уровне UI-сплиттеров, доли не портятся.

### Автоскрытие и фокус

Механизм целиком фокусный (`ToolWindowManagerLifecycle`, глобальный `AWTEventListener`):
FOCUS_LOST → `hideIfAutoHideToolWindowLostFocus` для `isAutoHide || type==SLIDING`;
исключения — попапы/балуны (`getParentBalloonFor`), диалоги, «фокус ушёл в никуда»
(покрывает деактивацию приложения), «панель сама запросила фокус». `activeToolWindowId` —
вычисляемое свойство от фокус-владельца; получение фокуса редактором чистит
`activeStack`. Шорткат активации трёхпозиционный: закрыта → открыть, не активна →
активировать, активна → закрыть; «активна» у эталона — по фокусу. Драг разделителя фокус
не двигает и action'ом не является — потому ресайз не закрывает unpinned-панель.

### Float и Window

FLOATING → `FloatingDecorator` — owned-окно поверх главного, вне таскбара; WINDOWED →
`WindowedDecorator` — независимый фрейм с заголовком «Title - Project»; общий предок с
окнами документов — `FrameWrapper`. Системное закрытие окна = `hideToolWindow`, `type`
жив — следующий Open снова откроет окно. Валидация сохранённых bounds: `isValidBounds`
(видимость по экранам) → при провале центрирование у главного окна. `internalType` —
память возврата в док (наш `LastInternalMode`, INV-7). В вебе (RemDev) WINDOWED
недоступен и деградирует только эффективный режим — хранимый `type` не меняется (наш
TW-7.6). Платформенный хром: на Windows float-окну системная рамка не рисуется, на Linux
урезанный хром owned-окна даёт сам WM — различие достигается типом окна, не хромом.

### Персистентность

XML/JDOM в workspace-файле; версия закодирована именем тега (`layout` vs `layoutV2`),
числового SchemaVersion нет. Философия противоположна нашей: IDEA никогда не падает на
workspace — нераспознанное игнорируется, невалидное гасится молча (`normalizeAfterRead`),
отчёта об исправлениях нет (только `LOG.debug`). Спящие состояния незарегистрированных id
переживают round-trip (`readExternal`/`writeExternal` пишут всё). Примирение при
регистрации: сохранённый `existingInfo` побеждает дескриптор, новичок получает
`getNextOrder` (в конец слота). `setLayout` — атомарная замена в три прохода и по природе
«arrangement»: контент не пересоздаётся, материализация ленивая.

### Контент панели

Контент тул-окна живёт в `ContentManager`; `InternalDecoratorImpl` умеет режимы
`SINGLE/VERTICAL_SPLIT/HORIZONTAL_SPLIT/CELL` — **бинарный** split с отдельным
`ContentManager` на ячейку. Модель контента панелей и модель редакторов
(`EditorsSplitters`) — разные вселенные; наше единое дерево панель+док-зона — дизайн
Berth, не буквальная IDEA. Terminal Reworked — лучший живой пример: собственного движка
у него нет, только `setTabsSplittingAllowed(true)` + `splitContentProvider` поверх общего
`InternalDecoratorImpl`. Важные факты оттуда: меню-сплит эталона **копирует** контент
(`createContentCopy`; для терминала «копия» = новая сессия в той же директории), переносит
только DnD — поэтому наши переносящие пункты названы «Split and Move …», а короткий ярлык
«Split …» зарезервирован за возможным «Split and Duplicate» v2; титул панели и кнопки
окна врисованы в полосы угловых ячеек (наша полоса корневой группы в ряду заголовка
декоратора — то же визуально); сплиты панелей IDEA **не персистит** (Berth сильнее);
одиночная вкладка контента показывается (Run/Terminal), панель одного тела (Project) —
нет (наша DA-8.4); второе представление полосы — combo (`content_ui`), кандидат v2.

### DnD

Базовый класс всех источников — `MouseDragHelper` на glass-pane: порог старта 7 px;
клик от drag отличается флагом на отпускании. Один хелпер (`ToolWindowDragHelper`) ведёт
и кнопки стрипов, и заголовки панелей: дроп в стрип = та же команда
`setSideToolAndAnchor` (наш ADR-0004 подтверждён эталоном), отпускание вне целей →
FLOATING у точки + активация — **единственная** активация во всём хелпере (`startDrag`
не активирует и не фокусирует; наша отложенная на отпускание активация заголовка).
Зоны дропа вкладки редактора: приоритет полосы над клиньями (`processDropOver`), клинья
по диагоналям с долей `dragToSplitRatio` = 0.2. Ghost — недекорированное always-on-top
окно ОС (`DialogDragImageView`, opacity 0.85, миниатюра до 220 px) с glass-pane-фолбэком
(Wayland). Спринг-лоадинга нет и у эталона. Отмена чиста у кнопок/заголовков (состояние
не мутировалось до коммита), но **грязна** у вкладок редактора (выбор переключается на
нажатии и не возвращается; reorder мутирует реальный порядок) — Berth всюду держит чистую
модель (DA-E22). Исключение автоскрытия для drag-релиза у эталона эмерджентно (кнопки
нефокусируемы) — у нас явное (TW-6.2).

## Сознательные расхождения Berth ↔ IDEA

1. **`Order` плотен по слоту**, у IDEA — по стороне (INV-3): строже при том же
   наблюдаемом порядке, делает TW-5.11 точным.
2. **SideStack не берём** (восстановление вытесненной панели при закрытии заменившей):
   в new UI IDEA сам выключен реестром.
3. **`ActiveToolWindowId` хранится**, у IDEA вычисляется из фокуса (INV-5, TW-6.5);
   «активность» для шортката всё равно определяется по фокусу (TW-5.5).
4. **Единое дерево таб-групп** для панелей и док-зоны против двух моделей IDEA
   (раздел 9 спеки) — главное структурное решение Berth (ADR-0005).
5. **Отчёт нормализации `Apply` и явная ошибка версии** (TW-10.4/10.5) против молчаливой
   деградации IDEA.
6. **Частичный `Arrangement`-merge** (TW-10.7) — наша семантика; `setLayout` IDEA полон.
7. **Доля пары выводится** (R1-нормализация) вместо пары «sideWeight + транзиентный
   кэш»; расхождение наблюдаемо только у прибывающего Primary с несогласованным весом.
8. **E22-отображение вставки при скрытых иконках** (после видимого предшественника) —
   наша конвенция, предсказуемее эталонной.
9. **Явные исключения автоскрытия и чистая отмена DnD** там, где у эталона они
   эмерджентны или грязны (TW-6.2, TW-5.17).
