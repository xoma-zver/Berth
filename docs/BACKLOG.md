# Бэклог

Правила:
- Задача = один ревьюируемый дифф (ветка → ревью → мердж в main).
- Сессия разработки берёт верхнюю незакрытую задачу, если владелец не указал иную.
- Завершение задачи включает: тесты на затронутые пункты спеки, обновление этого файла, ноль предупреждений сборки.

## Статус v1

Фазы 0–7 завершены (каркас; ядро tool windows; статический UI и ходячий скелет; режимы и
фокус; документная зона UI; DnD внутри окна; плавающие окна; интеграция и выпуск 0.1.0),
как и разовые хозяйственные задачи. Летопись выполненных задач — с решениями владельца,
находками реализации и итогами ревью по каждой — хранится в git-истории этого файла
(версии до 2026-07).

## Тематизация и оформление (решение владельца, 2026-07-21: приоритет до 1.0 — единственное направление, двигающее публичный API)

Обсуждение с владельцем: канонический lookless-путь (`TemplatedControl` + `ControlTheme` на
всё) отвергнут для персистентных узлов — заменяемость шаблона означает право платформы
пересоздавать визуальное поддерево, что несовместимо с гарантиями TW-9.13/DA-9.6
(однократное создание хостов, сохранение фокуса и view-state); шаблонизируется только
лист-хром, которому спека явно разрешает пересоздание. Порядок направлений: тематизация →
дотяжка DnD-визуализации панелей (расступание иконок стрипа — контракт уже в разделе 12
tool-windows) → «show tool window names» (потребует спекового решения и нового поля
состояния).

## Хозяйственное (вне фаз, по решению владельца)

- [x] Нереентерабельность прохода проекции — TW-9.14 (2026-07, гипотеза владельца об «окне-призраке», подтверждена регрессионным тестом): команда, порождённая внутри прохода `Sync` (реакции на перенос фокуса при переприсоединении хостов DA-6.4/TW-6.1, события платформенных окон TW-7.x), влетала вложенным проходом в середину подпроходов; согласование окон документов по пересечению вкладок (DA-1.3) заставало кэш `_documents` очищенным и создавало окно-дубликат — полосу вкладок без контента, гибнущую на следующем проходе. Страж в `BerthWorkspace.Sync`: вложенный запрос помечает проекцию устаревшей, перепроход по актуальному состоянию — по завершении текущего. По ревью владельца: сходимость ограничена пределом перепроходов (осциллирующая пара команд — громкое исключение вместо молчаливого зависания UI-потока; дедупликация равных состояний не исключает двухтактный цикл), а полный сброс кэшей при смене `Registry`/`Lifecycle` отведён в ту же воронку — немедленный `Reset()` из середины прохода ронял внешний проход на ближайшем обращении к кэшу. Спека: TW-9.14, кросс-ссылка в DA-9.6; тесты `ProjectionReentrancyTests` (оба слоя плавающих окон, детерминированный шов — действие из `OnAttachedToVisualTree` контента, с фиксацией срабатывания именно внутри прохода): призрак, осцилляция, сброс из середины прохода. Вторым коммитом (вопрос владельца) — предел глубины вложенности команд: фокусные реакции рекурсивны по стеку мимо проекции (каждый уровень запускает свой ограниченный проход), конечность держалась только на дисциплине проводки, а регрессия умерла бы неперехватываемым переполнением стека; превышение предела — громкое исключение, тест — фокусный пинг-понг тел двух панелей (`Open(a)` ↔ `Open(b)` из `OnGotFocus`). Третьим коммитом — регрессионный тест на боевую трассу владельца (пункт меню «Move to …» → перенос фокуса → реакция на фокус → падение в цикле `SplitView.Update` на сплите, теряющем ребёнка) с проверкой на коде до стража: воспроизводит исходное `ArgumentOutOfRangeException` кадр в кадр; попутно починены два теста `CrossWindowDragTests`, красневшие на Windows с задачи 6.2 — они ведут заголовок декоратора как источник слот-жеста, а это ветка декорированного Float (TW-7.1), тогда как на Windows заголовок безрамочного окна — ручка перемещения; шов презентации сделан трёхпозиционным (`FramelessFloatOverride`), обе ветки теперь достижимы явно
